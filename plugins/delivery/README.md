# delivery

Плагин для Claude Code, который проводит тебя по всем шагам — от идеи или бага до **готового MR, одобренного фронтендером**, — даже если ты никогда не работал с кодом напрямую.

> 🔧 **ADAPT.** Это форк обобщённого delivery-скелета. Механика флоу (git-хост, дев-VPN, чат-каналы, механика токенов) сохранена, но host-специфичные значения вынесены в метки. Замени помеченные `⟪ADAPT: …⟫` места: URL/имя твоего marketplace-репо, имя плагина, фронт-репо, команды сборки, версию Node, продукт. **i18n-платформа адаптируется отдельно** — впиши свою i18n-платформу/TMS или убери i18n-шаги, если мультиязычности нет. Полный гайд форка — в `ADAPT.md` в корне репозитория marketplace.

> 🖥️ **Где это запускать — прочитай до установки.** Флоу доставки рассчитан на **десктоп-приложение Claude Code** (вкладка **Code**): там удобный UI, превью и скриншоты. **Не** веб-версия (`claude.ai`) и **не** «навсегда в терминале».
>
> - **Десктоп-приложение** — для самой работы по флоу. Нет приложения? Поставь: **https://claude.com/download** → войди в аккаунт → вкладка **Code** → окружение **Local** → выбери папку проекта.
> - **Терминальный `claude`** (в Terminal / iTerm) — нужен **только один раз**, для установки и обновления плагина (§3). Дальше он не нужен.
> - **Веб (`claude.ai/code`)** — для этого флоу **не подходит** (нет доступа к локальным файлам и репо).

> 🤖 **Самый простой старт — пусть Claude проведёт тебя по этому README.** Не хочешь ставить вручную: открой **Claude Code** (десктоп → вкладка **Code**, либо терминал `claude`) в любой папке, **прикрепи или перетащи этот файл `README.md`** в чат и напиши: «проведи меня по этому файлу — помоги поставить плагин и настроить доступы». Claude пойдёт по шагам: что может — проверит и сделает сам (окружение), а где нужны твои действия (ввести `/plugin`-команды; позже — создать токен для рабочего git-хоста) — даст точную команду и дождётся тебя. Предпочитаешь вручную — иди по разделам ниже.
>
> **Где взять сам файл `README.md`:** репо публичный — открой на GitHub `https://github.com/echangli/delivery-flow/blob/main/plugins/delivery/README.md` и скопируй текст (или перетащи файл в чат). Ссылку на этот README обычно дают в чате команды.

## 1. Что это и зачем

`delivery` — это пошаговый флоу доставки фичи или бага на ⟪ADAPT: вашей платформе⟫. Ты описываешь задачу (карточку, идею, баг), а плагин ведёт тебя по стадиям:

- **Трек A** — новая фича или идея: полный цикл с дизайном и несколькими вариантами на выбор дизайнеру, реализацией и QA-смоуком.
- **Трек B** — фикс бага или правка существующего поведения: облегчённый цикл без дизайна, с локализацией причины и проверкой на стейдже.
- **Трек C** — вход в середину флоу: проверить или дофиксить **существующий MR**, разобрать проблему **на стейдже** или поправить **UI-тексты/переводы**. Подхватывает с твоей точки входа, не заставляя начинать «с идеи».

Финал всех треков — **draft-MR/PR с заполненным описанием на английском**, готовый к ревью фронтендером на вашем git-хосте. **Мержить и деплоить может только QA или фронтендер — и только после апрува.**

Помимо флоу доставки, в плагине есть **QA-скиллы**, которые зовутся и сами по себе:

- **`/qa_mr`** — полный QA-разбор фронтового MR: что изменилось, куда смотреть, корнер- и смоук-кейсы, живой прогон на слоте стейджа (desktop + mobile, инкогнито), UX/UI-советы и отчёт.
- **`/autotest`**, **`/selectors_sync`**, **`/autotest_run`** — durable-автотесты фронта (Playwright-сюита ⟪ADAPT: путь e2e-сюиты, напр. `e2e/web`⟫): написать тест, синхронизировать `data-test`-селекторы с живой поверхностью, прогнать на слоте и довести до зелёного.

Подходит для всех: продакта, аналитика, QA, дизайнера — даже если нет опыта работы с кодом или терминалом.

## 2. Предтребования

**Чтобы поставить плагин (нужно сразу — см. §3):**

- **Десктоп-приложение Claude Code** — основной интерфейс флоу (установка — заметка в начале). Терминальный `claude` понадобится только на время установки.
- Больше ничего. Плагин лежит в **публичном GitHub-репо** `github.com/echangli/delivery-flow` — скачивается сразу, аккаунт GitHub, приглашение, вход и VPN **не нужны**.

**Понадобится позже по ходу флоу — шаг 0 плагина проверит сам и подскажет, что делать:** ⟪ADAPT: Node ≥ версия⟫, доступ к рабочему фронт-репо (роль ≥ Developer) + токен для его git-хоста, дев-VPN (если рабочий репо/стейдж за VPN), MCP (Figma / Notion). Заранее настраивать не нужно.

## 3. Установка плагина

Делается один раз. **Команды `/plugin …` ниже выполняются в терминальном `claude`** (Terminal / iTerm → команда `claude`), а не в десктоп-приложении. После установки терминал больше не нужен — работа по флоу идёт в десктопе (см. заметку в начале).

> **Два разных доступа — не путай.** (а) **Скачать плагин** — репо публичный, ничего не нужно (ни аккаунта GitHub, ни приглашения, ни VPN). (б) **Пушить код** по флоу = токен git-хоста **твоей команды** (`⟪ADAPT: ваш git-хост⟫`), это отдельный шаг (§3.5) и нужен только когда дойдёшь до реализации. Здесь, в §3, ставим сам плагин — доступа для этого не требуется.

**Шаг 1. Запусти `claude` в терминале** (Terminal / iTerm → команда `claude`) и **внутри** этой сессии добавь marketplace (один раз):

```
/plugin marketplace add https://github.com/echangli/delivery-flow.git
```

**Шаг 2. Установи плагин:**

```
/plugin install delivery@delivery-toolkit
```

Claude Code спросит **scope** — куда поставить плагин. Выбирай **`User`**: флоу не привязан к одному репозиторию, ты запускаешь его из разных папок, и при `User` он доступен везде.

- **`User`** — для тебя, во всех проектах (`~/.claude`). ← **то, что нужно.**
- **`Project`** — для всех в этом репо (пишется в `.claude/` репозитория и попадает в git). Не подходит.
- **`Local`** — только ты и только в этом репо. Не подходит.

Ставится **только этот плагин** — остальное содержимое marketplace не скачивается и не активируется.

### Обязательная зависимость: superpowers

Флоу `delivery` опирается на скиллы из плагина **`superpowers`** (`brainstorming`, `writing-plans`, `using-git-worktrees`, `systematic-debugging`, `verification-before-completion` и др.). Без него оркестратор будет ссылаться на несуществующие скиллы — поэтому установи его **обязательно**:

```
/plugin install superpowers@claude-plugins-official
```

Маркетплейс `claude-plugins-official` — официальный, **встроен** в Claude Code (отдельно `marketplace add` не нужно). Scope при установке выбирай тоже **`User`** (во всех проектах).

**Включать автообновление не нужно:** у официального маркетплейса оно по умолчанию включено, поэтому superpowers обновляется сам, независимо от твоего marketplace.

### 3.5. Токен для рабочего git-хоста (доступ (б) — push кода)

Отдельно от доступа к плагину. Нужен, **когда дойдёшь до реализации** (шаг 0 плагина / `delivery_setup §2.0` напомнит) — тем же токеном агент пушит твою ветку в рабочий фронт-репо и (если git-хост это поддерживает) сам поднимает временный стейдж через API. Делается **один раз**, в обычном терминале:

1. Создай **Personal Access Token** на git-хосте твоей команды (`⟪ADAPT: ваш git-хост⟫` → профиль → Access Tokens). Scope: **право push в репозиторий** + **доступ к API** (нужен агенту для авто-деплоя стейджа; на git-хостах вроде GitLab это `api` + `write_repository`). Скопируй (показывается один раз).
2. Выполни в терминале (подставив свой токен; **в чат его не вставляй**):

   ```bash
   PAT="сюда-вставь-токен-рабочего-git-хоста"
   { echo "export ⟪ADAPT: env-переменная API-токена, напр. GIT_API_TOKEN⟫=\"$PAT\"" ; } >> ~/.zshenv
   printf 'protocol=https\nhost=⟪ADAPT: ваш git-хост⟫\nusername=oauth2\npassword=%s\n' "$PAT" | git credential approve
   echo "✅ готово — открой новый терминал"
   ```

   Токен кладётся и в **Keychain** (git будет пушить ветку), и в **`~/.zshenv`** (агент: push ветки + авто-деплой стейджа для non-interactive Bash). **Проверка:** в новом терминале `git ls-remote https://⟪ADAPT: ваш git-хост⟫/⟪ADAPT: группа/фронт-репо⟫.git` — вывелись refs → доступ есть.

   > 🔧 **ADAPT — токен i18n-платформы (опционально).** Если у вас есть i18n-платформа/TMS со своим CLI-токеном — добавь его в ту же команду: `echo "export ⟪ADAPT: env-переменная токена вашей i18n-платформы⟫=\"...\"" >> ~/.zshenv` (+ хост, если нужен: `echo 'export ⟪ADAPT: env-переменная хоста i18n-платформы⟫="..."' >> ~/.zshenv`). Токен создаётся в UI твоей i18n-платформы (роль `⟪ADAPT: роль/scope токена⟫`). **Нет мультиязычности — пропусти этот шаг.**

### Нюансы

- **Аккаунт GitHub не нужен.** Репо публичный — плагин качается без логина. Нет доступа к самому репо (недоступен URL) — напиши автору (`github.com/echangli`) или в **⟪ADAPT: ваш чат-канал⟫**.
- **Токен рабочего git-хоста** — это отдельная вещь (§3.5), нужна не для плагина, а для push кода по флоу. `delivery_setup §2.0` лишь **проверяет** его наличие.
- **Вешай marketplace на `main`.** Шаг 1 без указания ветки берёт дефолтную ветку репо (`main`) — так и нужно. Не привязывай к feature-ветке: её удаляют после мержа, и тогда обновление падает с `Remote branch ... not found`.

### Автообновление (рекомендуется)

Для своего marketplace автообновление по умолчанию выключено — включи его **одним кликом** через UI:

**`/plugin` → вкладка `Marketplaces` → твой marketplace → `Enable auto-update`.**

После этого Claude Code сам подтягивает свежий `main` при старте; по нотификации — `/reload-plugins`. Состояние Claude Code хранит сам (`~/.claude/plugins/known_marketplaces.json`, `"autoUpdate": true`) — руками ничего править не нужно. Это один раз — дальше обновления приходят без команд. Репо публичный, так что апдейты тянутся сами — токен или логин не нужны.

**Обновить вручную** (если без autoUpdate): `/plugin marketplace update delivery-toolkit` → `/reload-plugins`. Если marketplace был привязан к удалённой ветке и `update` падает с `Remote branch ... not found` — пересоздай: `/plugin marketplace remove delivery-toolkit` → `/plugin marketplace add https://github.com/echangli/delivery-flow.git` (возьмёт `main`) → `/plugin install delivery@delivery-toolkit` → `/reload-plugins`.

> 🔧 **ADAPT — каналы и git-хост в таблицах ниже.** В стадиях указаны ссылки на ваш чат для запросов доступа/передачи (`⟪ADAPT: ваш чат-канал⟫`) и git-хост команды. Механику (кто что делает на каждой стадии) не трогай.

## 5. Стадии флоу и что делает человек

### Трек A — новая фича / идея

| Стадия | Что делает агент | Что делает человек |
|---|---|---|
| **0. Преднастройка** | Проверяет Node, git, доступ к фронт-репо, дев-VPN, MCP | Поднимает VPN, входит в git-хост (PAT), даёт нужные доступы (по подсказке) |
| **1. Захват идеи** | Задаёт уточняющие вопросы простым языком | **Описывает задачу подробно** — прикладывает скриншоты, макеты, ссылки на карточки |
| **2. Дизайн + прототип** | Создаёт кликабельный HTML-прототип в нескольких вариантах (разные UX / размещение / механика) — целиком, не по частям; сверяет с прод-скринами | **Дизайнер смотрит варианты и выбирает один** — передача в чат (`⟪ADAPT: ваш чат-канал⟫`) |
| **3. План** | Разбивает задачу на шаги реализации | Согласует или корректирует план |
| **4. Реализация** | Пишет код по правилам фронт-репо в отдельном worktree | Ждёт; при необходимости — входит в систему, поднимает VPN |
| **5. Локальная проверка (до push)** | Поднимает dev-сервер (выбор API stage/prod), сверяет визуал со скринами, green-sequence. **Может прокликать сам** (Playwright) — нужен лишь актив/сценарий | Выбирает API; логинится — или даёт агенту минимум данных для самопроверки |
| **6. Push + Draft-MR** | Пушит ветку, формирует ссылку на создание MR (EN-описание + чек-лист) и поясняет, что будет дальше | **Открывает ссылку → создаёт MR** → передаёт фронтендеру на git-хост |
| **7. QA-смоук на стейдже** | **После push:** слот или **временный деплой через API git-хоста** (агент сам поднимает и даёт ссылку); desktop + мобайл, инкогнито | QA получает чек-лист в чате; логинится при необходимости |
| **8. Ревью** | — | Фронтендер ревьюит MR; правки — в ту же ветку; **мержит только QA/фронтендер после апрува** |

### Трек B — баг / правка существующего поведения

| Стадия | Что делает агент | Что делает человек |
|---|---|---|
| **0. Преднастройка** | То же, что в Треке A | То же |
| **1. Захват бага** | Уточняет детали | **Описывает баг:** где, что не так, шаги воспроизведения, ожидаемое vs фактическое, скрины/видео, ссылка на карточку |
| **2. Локализация причины** | Читает код фронта, воспроизводит баг, находит корень проблемы | При необходимости — подтвердить правильность версии с агентом |
| **3. Дизайн (опционально)** | Если изменение визуальное — делает before/after в чат | Дизайнер смотрит; если изменений не видно — стадия пропускается |
| **4. Фикс** | Правит код в worktree, по возможности добавляет регресс-тест | — |
| **5. Локальная проверка (до push)** | Воспроизводит баг на dev-сервере, before/after, green-sequence. **Может прокликать сам** | Логинится — или даёт данные для самопроверки |
| **6. Push + Draft-MR** | Пушит ветку фикса, формирует ссылку на MR (before/after + чек-лист), поясняет, что дальше | **Открывает ссылку → создаёт MR** → передаёт фронтендеру |
| **7. QA-смоук на стейдже** | **После push:** слот или временный деплой через API git-хоста; баг устранён, регресса рядом нет (desktop + mobile, инкогнито) | QA получает чек-лист; **мержит только QA/фронтендер** |

### Трек C — вход в середину (готовый MR / стейдж / тексты)

| Стадия | Что делает агент | Что делает человек |
|---|---|---|
| **0. Преднастройка** | То же (+ токен i18n-платформы при работе с переводами) | То же |
| **1. Захват + ветка** | **Надёжно** определяет ветку MR (спрашивает, не угадывает) | **Даёт ссылку на MR и называет ветку**; описывает, что не так |
| **2. Понять проблему** | Разбирает diff / стейдж / i18n; при баге — локализует причину | Подтверждает при неоднозначности |
| **3. Фикс в ветке MR** | Правит в ветке; для текстов — варианты RU+EN, синк исходного сообщения, i18n-сборка | Заводит/обновляет ключи в i18n-платформе; выбирает формулировки |
| **4. Локальная проверка (до push)** | Поднимает dev-сервер, проверяет правку (**может сам**); если состояние трудно поднять локально — на стейдж после push | Логинится — или даёт данные для самопроверки |
| **5. Push + QA-смоук на стейдже** | Push в ветку MR (с согласия автора, **не force-push**); затем смоук на слоте/деплое: проблема ушла, регресса нет | `commit`/`push` сам или разрешает агенту; QA получает чек-лист |
| **6. Согласование** | Готовит diff/изменения; **не force-push чужую ветку** | Возвращает MR на ревью; **мержит только QA/фронтендер** |

## 6. Доступы и куда писать

| Что нужно | Как получить / куда писать |
|---|---|
| **Сам плагин** | Публичный GitHub-репо `github.com/echangli/delivery-flow` — доступ не нужен, ставится тремя командами (§3). URL недоступен → автор (`github.com/echangli`) или **⟪ADAPT: ваш чат-канал⟫** |
| **Дев-VPN** (если рабочий репо/стейдж за VPN) | ⟪ADAPT: ваш VPN-клиент⟫; если нет доступа → **⟪ADAPT: ваш чат-канал⟫** |
| **Доступ к фронт-репо** | Нужны права **≥ Developer** (push ветки + MR) на git-хосте команды; запрашивать в **⟪ADAPT: ваш чат-канал⟫** |
| **Аккаунт на git-хосте команды** | Возможно нужно завести; попроси в **⟪ADAPT: ваш чат-канал⟫** |
| **MCP Figma** | `/mcp` в Claude Code → Authenticate; при проблемах → **⟪ADAPT: ваш чат-канал⟫** |
| **MCP Notion** | Аналогично Figma |
| **i18n-платформа (переводы)** | Только для новых i18n-ключей; если нет доступа — агент покажет список ключей и попросит добавить их через чат |

Общий принцип: если что-то не работает или нет прав — не застревай молча. Напиши в **⟪ADAPT: ваш чат-канал⟫**, агент подскажет, что именно попросить.

## 7. Guardrails — что нельзя нарушать

- **Без поясняющих комментов в коде.** Если комментарий не специфичен и не критичен — не ставить.
- **Коммиты/ветки фронт-репо — по git-конвенции репо (обычно проверяется хуками + CI):** ⟪ADAPT: краткая суть вашей git-конвенции — язык коммитов, формат субджекта, префиксы веток; детали — в `repo-conventions.md`.⟫ **Claude не добавляет `Co-Authored-By`/`Generated with Claude Code` в коммиты фронта** — иначе хук/CI отклонит пуш (это ловушка Claude Code в любом репо).
- **Стадия 5 (dev-server) + выбор API — обязательны перед push; стоит закрепить PreToolUse-хуком.** Опциональный локальный harness-гейт (`<workspace>/.claude/hooks/dev-check-gate.py` + `settings.json`): блокирует запуск dev-server без явного выбора API (`# api-stage`/`# api-prod`) и `git push` рабочей ветки без маркера `# devcheck-ok` (dev-проверка + custom-audit дизайн-системы сделаны). Делает шаги harness-enforced, а не «на внимательность».
- **Никогда не мержить и не деплоить самостоятельно.** Флоу заканчивается на draft-MR и апруве фронтендера. Мерж и деплой — это работа QA или фронтендера, и только после апрува.

## 8. Лог заметок

По ходу флоу агент ведёт файл заметок — записывает затыки, тупики, нестандартные ситуации. Файл создаётся в **рабочей папке пользователя** (не в плагине), с именем вида `delivery-notes-YYYY-MM-DD.md`.

**Как посмотреть:** спроси агента «покажи лог заметок» или открой файл напрямую.

**Как отправить автору:** в конце флоу агент сам предложит создать **GitHub Issue** — сформирует предзаполненную ссылку (заголовок + тело на русском, выжимка из заметок/ретро) и отдаст её в чат. Один клик — issue появляется в Issues репо плагина у автора. Либо вручную: **Issues → New issue → форма «Обратная связь по флоу»**. Это помогает дорабатывать флоу по реальным кейсам.

**Гигиена `CLAUDE.md`:** детали и журнал задач живут в папке задачи, не в `CLAUDE.md` (он грузится каждую сессию). Если ведёшь в `CLAUDE.md` список задач — только активные; завершённые сворачивай в одну строку (или в `tasks/DONE.md`).
