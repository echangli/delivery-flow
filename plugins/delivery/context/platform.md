# Платформенный контекст ⟪ADAPT: название продукта⟫

> 🔧 **ADAPT — это болванка-шаблон.** В исходном проекте здесь жил курированный справочник о платформе, который
> оркестратор грузит один раз в начале сессии и использует при уточнении задачи, прототипировании и
> написании MR. Замени содержимое своим продуктом, сохранив разделы-подсказки. Держи файл **фиксированным**
> (живой контекст — статусы карточек, свежие прод-скрины — подтягивается из чата/Notion/Figma, не отсюда).

Этот файл — что нужно знать о продукте любому, кто участвует во флоу доставки. Пиши факты, не маркетинг.

> **Углубление по требованию** (грузить, когда задача касается — не в каждой сессии): доменная модель
> центрального объекта → **`context/dca.md`**; продуктовая аналитика → **`context/analytics.md`**.
> ⟪ADAPT: переименуй dca.md под свой центральный объект, если нужно.⟫

## 4. Дизайн-система и конвенции кода
⟪ADAPT: где живёт дизайн-система (токены `var(--*)`, компоненты) — путь во фронт-репо. В прототипах и

## 5. Ревью и хэндоффы
Флоу проходит через ревью: **дизайнер** (прототип, Трек A) → **QA** (смоук на стейдже, desktop+mobile,
инкогнито) → **фронтендер** (ревью MR). **Мерж и деплой — только QA/фронтендер, только после апрува.**
Флоу и не-кодер сами не мержат и не деплоят никогда.
⟪ADAPT: кто именно ревьюит/мержит у вас; Slack-каналы для хэндоффов и запросов доступа
(ваш чат-канал для запросов доступа — замени/дополни своими продуктовыми).⟫

## 6. Словарь
Универсальные термины флоу (оставлены — они не про продукт): **Worktree** — параллельная рабочая
директория `git worktree` для ветки. **Green-sequence** — набор CI-проверок перед коммитом (состав —
`repo-conventions.md`). **Draft MR** — MR в статусе Draft, стандартный выход флоу. **Стейдж/Прод** —
тестовое vs публичное окружение.
⟪ADAPT: добавь 5–15 терминов своего домена, которые встречаются в задачах и коде.⟫
