# Аналитика ⟪ADAPT: аналитическая платформа⟫ — как устроена + seed событий

> 🔧 **ADAPT — это болванка-шаблон.** В исходном проекте здесь был справочник по продуктовой аналитике для задач, где
> нужно завести/проверить/расширить событие. Грузи **по требованию** (аналитические задачи), не в каждой
> сессии. Замени содержимое своей аналитикой. Общий обзор — `context/platform.md`.

---

## 1. Как шлётся аналитика
⟪ADAPT: как в коде логируется событие — какая функция/пакет, сигнатура вызова, куда уходят события
(трекеры). Отметь правило «не терять аналитику при рефакторинге», если оно у вас есть в конвенциях.⟫

## 2. Централизована ли таксономия
⟪ADAPT: есть ли единый реестр событий (enum/справочник) или имена захардкожены по месту. Если реестра
нет — как получить «список всех событий» (grep по чему). И **с кем согласовывать** имена новых событий/
свойств (аналитик), чтобы не плодить дубли.⟫

## 3. Паттерны свойств
⟪ADAPT: если есть сквозные свойства с общим форматом (в исходном проекте — `place` в формате `{feature}-{id}` для
шеринга) — опиши формат и где формируется. Иначе удали раздел.⟫

## 4. Seed событий (подмножество, пополняется по задачам)
> 🔧 **ADAPT — реестр событий вашего продукта.** Веди таблицы «Событие → свойства → где в коде» по группам
> (в исходном проекте было: центральный объект, шеринг, основной рабочий флоу, депозит, auth, навигация, рефералка). Это не полный
> реестр, а рабочий seed — наполняй по мере задач. Пример строки формата:

| Событие | Свойства | Где |
|---|---|---|
| ⟪ADAPT: `event_name`⟫ | ⟪ADAPT: `prop1, prop2`⟫ | ⟪ADAPT: путь к файлу⟫ |

## 5. Рабочее правило
1. Новая фича с трекингом → **согласуй имя события и свойства с аналитиком**.
2. Расширяешь существующее событие → **сохрани старые свойства** (не ломай воронки), добавляй новые.
3. Сквозные свойства — по паттерну (§3), не произвольная строка.
4. Проверка на стейдже — ⟪ADAPT: как проверяете, что событие ушло с верными свойствами (MCP/live-прогон).⟫
