# delivery_prototype — кликабельный HTML-прототип (пространство выборов)

Ты строишь кликабельный прототип фичи на реальной дизайн-системе из фронт-репо.
Цель — дать дизайнеру несколько конкретных вариантов реализации на ревью, а не абстрактный макет.

## 1. Когда использовать

Стадия дизайна **Трека A** — после или в процессе `brainstorming` (когда идея/механика оформлена,
но до написания плана реализации). Запускается оркестратором на шаге 2 Трека A: «дизайн + прототип».

Прототип нужен **всегда**, даже если элемент кажется «подтверждённым» после брейншторма —
согласование идеи словами ≠ визуальное подтверждение реализации. Правило: «подтверждение ≠ не нужен
прототип».

## 2. Источник дизайн-системы

Используй **только** дизайн-систему из фронт-репо (`⟪ADAPT: имя/путь фронт-репо⟫`):
- `⟪ADAPT: путь к tokens.css, напр. design-system/ui-kit/tokens.css⟫` — CSS-переменные (source of truth для цветов, типографики,
  отступов, скруглений);
- `⟪ADAPT: путь к исходным токенам, напр. design-system/ui-kit/src/tokens/*.json⟫` — исходные токены (для сверки значений);
- `⟪ADAPT: путь к компонентам ДС, напр. design-system/ui-kit/src/<Component>/⟫` — компоненты (разметка, классы, состояния).

**Запрещено:**
- Использовать любую design-system вне фронт-репо (отдельный пакет/форк ДС, сжатые справочники и т.п.).
- Хардкодить hex-цвета (`#1a1a2e`, `rgba(...)` и т.п.) — только `var(--*)` токены.

Перед сборкой прототипа прочитай актуальный `tokens.css` из фронт-репо, чтобы токены совпадали
с продом.

## 3. Самодостаточный HTML

Каждый вариант прототипа — **полный самодостаточный HTML-файл** (не фрагмент, не компонент,
не сниппет). Структура: `<!DOCTYPE html>` + `<head>` со встроенным CSS + `<body>` + встроенный JS.

**Почему full page:** brainstorm-компаньон (Claude Preview) отдаёт только новейший HTML целиком.
Фрагмент конфликтует с CSS фрейма и ломает отображение — поэтому прототип всегда полностраничный.

**Картинки и статика:**
- Иконки, логотипы, аватары, иллюстрации — вшивать как **data-URI** (`src="data:image/svg+xml;base64,..."`)
  прямо в HTML.
- Если статики много (мемы, фотографии) — поднять локальный сервер:
  `python3 -m http.server 8080 --directory ./prototypes` и ссылаться на `http://localhost:8080/...`.
  Запустить **до** открытия прототипа в браузере/preview.
- Не полагаться на brainstorm-компаньон для статических ассетов — они теряются между итерациями.

## 4. Рендер и самопроверка

После сборки каждого варианта — **обязательно отрендерить и проверить**:

1. **Рендер — рабочий рецепт (Claude Preview + статика):**
   - создать `.claude/launch.json` со статик-сервером:
     `{"version":"0.0.1","configurations":[{"name":"prototypes","runtimeExecutable":"python3","runtimeArgs":["-m","http.server","8765","--directory","<абс. путь к prototypes>"],"port":8765}]}`
   - `preview_start("prototypes")` → получить `serverId`;
   - открыть нужный файл: `preview_eval(serverId, "location.href='http://localhost:8765/<файл>.html'")`;
   - `preview_resize` под desktop (1440×900) и mobile (402×844) → `preview_screenshot` для каждого;
   - состояния за тумблерами/анимацией снимать **детерминированно** через `preview_eval` (выставить
     состояние напрямую и сделать скрин), а не ловить тайминг.
   Альтернатива — `python3 -m http.server` + скриншот браузером.
2. **Скриншот** каждого состояния/варианта (desktop 1440×900 + мобильный 402×844, если есть мобильная раскладка).
3. **Сверка с эталонными прод-скринами:** попросить пользователя предоставить скриншоты
   продакшн-интерфейса (или взять из ранее загруженных); сравнить типографику, отступы, цвета,
   выравнивание. Точность — типичная правка при ревью дизайнера, не упускай.

Если обнаружено расхождение с эталоном — исправить до хэндоффа.

## 5. Отдать дизайнеру всё пространство выборов (не один вариант)

**Правило: никогда не отдавать один зафиксированный вариант.** Прототип — это **пространство выборов**,
из которого дизайнер собирает финал за один заход, без хождения к нему итерациями. Все открытые
развилки должны быть в прототипе сразу.

**Дефолт — ОДИН самодостаточный файл с демо-панелью развилок.** Фиксированная панель (подписать
«(прототип)», на mobile `position: static`, чтобы не перекрывала контент) с тумблерами на **каждую**
открытую развилку, переключающими состояние **живьём**:
- **механика / состояния** — loading → ready → error, edge/пустые состояния, «нет средств» и т.п.;
- **плейсмент** элемента;
- **подача** отдельных под-блоков (списком / ячейками / чипами / hero …);
- **лейбл / копи**, если ещё не решены.

Пример из практики: один файл, тумблеры — плейсмент кнопки · лейбл · подход при
нехватке средств · лейаут окна · подача метрик · состояния. Дизайнер прожал всё в одном окне.

**Отдельные full-page файлы — только для сравнения «бок-о-бок»**, когда развилку удобно видеть рядом
одновременно (обычно крупный плейсмент/лейаут). Тогда: по файлу на этот вариант, а **остальные
развилки — всё равно тумблерами внутри каждого**. Не плодить файлы без нужды.

**Micro-options page для спорного под-элемента.** Если открыт вопрос «как показать конкретный блок»
(напр. ⟪ADAPT: пример спорного под-блока вашего продукта⟫ — ячейки vs список vs чипы vs hero vs формула) — собрать отдельную **side-by-side
страницу** с 5–6 раскладками одного элемента, дать выбрать, затем выбранное завести тумблером в
основной прототип.

**Деривация субагентами.** Если всё же нужны отдельные файлы под плейсменты — эффективно поручить
субагентам (Sonnet) сделать варианты из базового файла (перенос только различающейся части). Если файл
один с тумблерами — строить напрямую, субагенты не нужны.

**Подтверждённые элементы тоже прототипировать** — «договорились словами» ≠ «дизайнер одобрил».

**При хэндоффе** — отдать файл(ы) + явно перечислить, **что дизайнер выбирает** (по каждой развилке:
плейсмент / лейбл / состояние-подход / подача …), и попросить пройтись тумблерами и выбрать. Не «какой
из N файлов», а «какие значения развилок берём».

### Копи — на-voice сразу (shift-left)

Тексты прототипа (кнопки, лейблы, заголовки, empty states) писать **на бренд-voice с самого начала** — по вашему
гайду voice/глоссария/microcopy-паттернов, затем прогнать черновик копи через ваш линтер исходной копи
⟪ADAPT: если есть⟫ до показа дизайнеру. Так копи уже чистая, и когда станет i18n-ключами — меньше правок на
i18n-гейте (стадия 5). Фолбэк без линтера: sentence-case, глагол-действие на кнопках, без em-dash, без идиом
(мини-чеклист — `repo-conventions.md §i18n`).

## 6. Где хранить файлы

Прототипы — в **папке задачи**, в рабочей папке пользователя (там, где проект/задачи), **не в папке
плагина** (единый канон раскладки — `delivery_orchestrator/references/workspace-layout.md`):

```
tasks/<тикет>-<слаг>/          # без тикета — просто <слаг>
  prototype.html               # дефолт: ОДИН файл с демо-панелью всех развилок
  prototype-<вариант>.html     # только для сравнения бок-о-бок; иначе развилки — тумблерами в одном файле
  prototype-<элемент>-options.html   # micro-options page для спорного под-блока
  screenshots/
    <вариант>-desktop.png
    <вариант>-mobile.png
```

Слаг — короткое имя задачи ⟪ADAPT: примеры слагов вашего продукта⟫; с тикетом — `<ТИКЕТ>-<слаг>`.

Папки задачи ещё нет → создать первым действием (`mkdir -p tasks/<слаг>`). Плоские
`prototypes/*.html` в рабочей папке не заводить — анти-паттерн, признак старой свалки
(`delivery_orchestrator/references/workspace-layout.md` § 4).

Прототипы и скрины **не коммитить в репо проекта** — рабочие артефакты, живут локально до ревью.

## 7. Хэндофф дизайнеру

После рендера и самопроверки — хэндофф дизайнеру:

1. **Сообщение в Slack** (публичный Slack-чат команды, тот же, что для QA-хэндоффа):
   - Кратко что за фича и зачем прототип.
   - Перечень **развилок**, которые выбирает дизайнер (плейсмент / лейбл / состояние-подход / подача …),
     и как их прожать (тумблеры демо-панели).
   - Прикрепить скриншоты ключевых состояний (desktop + mobile).
   - Явный вопрос: «Что берём по каждой развилке?» (или «что доработать?»).

2. **Отдать прототип — ссылкой (предпочтительно) или файлом.**

   **Проактивно предлагай ссылку.** Как только прототип готов к ревью — сразу предложи задеплоить его
   на Vercel и отдать дизайнеру **ссылку** (открыть ссылку удобнее, чем скачивать/открывать HTML; при
   правках ссылка та же — переотдавать файл не нужно). HTML-файл или локальный сервер
   (`python3 -m http.server`) — фолбэк, если деплой недоступен.

   **Разовая настройка (один раз на человека):** бесплатный аккаунт Vercel + токен
   (`vercel.com` → Account → Tokens) в `.env` папки задачи (`VERCEL_TOKEN=…`, не коммитить, `.env` в `.gitignore`).
   Токен в чат не слать; прислали — использовать и предложить перевыпустить. Дальше деплой — одной командой / `deploy.sh`.

   **Деплой:**
   ```bash
   # Из папки задачи (tasks/<тикет>-<слаг>/):
   mkdir -p dist && cp prototype.html dist/index.html
   npx vercel --prod --yes --scope <team> --token=$VERCEL_TOKEN
   ```
   - `--scope` нужен, если токен принадлежит команде (без него деплой падает с ошибкой).
   - Если в проекте Vercel включён **Deployment Protection (SSO)** — публичный доступ заблокирован
     (401). Снять через API:
     ```bash
     curl -s -X PATCH "https://api.vercel.com/v9/projects/<project-id>" \
       -H "Authorization: Bearer $VERCEL_TOKEN" \
       -H "Content-Type: application/json" \
       -d '{"ssoProtection": null}'
     ```
   - Алиас `*.vercel.app` иногда таймаутит с части сетей (edge-IP) → фолбэк: прямой URL деплоя
     (вида `https://<hash>-<team>.vercel.app`) или локальный файл `prototype.html`.
   - `deploy.sh` в папке задачи — для повторных обновлений (cp + `vercel --prod` + realias одной командой);
     ссылка стабильная, дизайнер видит правки по тому же URL.

3. **После ответа дизайнера** — зафиксировать выбор и передать его в оркестратор как вход для
   шага 3 (план реализации). Не начинать план до получения выбора.

## Правила

- **Один зафиксированный вариант — не хэндофф.** Отдавать всё пространство выборов: один файл с тумблерами развилок (дефолт); отдельные файлы — только для сравнения бок-о-бок.
- **Только `var(--*)` из фронт-репо** — никаких захардкоженных цветов.
- **Full page, не фрагмент** — иначе конфликт с фреймом preview.
- **Сверяй с прод-скринами** — точность важна, её часто правят на ревью.
- **Рендер + скриншот перед хэндоффом** — не отдавать «слепой» HTML.
- **На ревью прототип — предпочтительно ссылкой** (Vercel deploy), не HTML-файлом: дизайнеру удобнее открыть ссылку, правки видны по тому же URL. Предлагать проактивно; файл/локальный сервер — фолбэк (§7).
- **Подтверждённые элементы тоже прототипировать** — «договорились словами» ≠ «дизайнер одобрил».
- Язык комментариев и хэндоффа — русский; названия компонентов/токенов — как в коде.
