# Политика выбора моделей

Справочник для оркестратора: какой тир модели использовать на каждой стадии флоу доставки и почему.
Тиры важнее конкретных версий — версии меняются, логика остаётся. Актуальные названия на момент написания:
**Opus 4.8** (сильная), **Sonnet 4.6** (стандартная), **Haiku 4.5** (дешёвая/быстрая).

---

## Таблица: стадия / роль → тир → почему

| Стадия / роль | Тир модели | Конкретная модель (сейчас) | Почему |
|---|---|---|---|
| Дизайн, спека, ревью (spec + code-quality) | **Сильная** | Opus 4.8 | Требует глубокого рассуждения: выявить пропущенные кейсы, оценить качество архитектуры, сформулировать спеку без пробелов. Экономить здесь — значит получать плохие спеки и поверхностное ревью. |
| Имплементер-субагент — изолированная механика (1–2 файла, чёткий спек) | **Дешёвая / быстрая** | Haiku 4.5 | Задача хорошо ограничена: конкретный файл, конкретное изменение, понятный результат. Дорогая модель здесь избыточна; Haiku даёт нужную скорость и экономию при параллельном запуске нескольких субагентов. |
| Имплементер-субагент — мультифайловая интеграция / паттерн-матчинг | **Стандартная** | Sonnet 4.6 | Нужно понять несколько модулей, найти правильный паттерн в кодовой базе, корректно вписать изменение. Haiku тут ошибается; Opus — избыточен. |
| Прототип / HTML / рутинные правки | **Стандартная** | Sonnet 4.6 | Достаточно для генерации качественного HTML/CSS по дизайн-токенам и правок по шаблону. Не требует тяжёлого рассуждения. |
| Ведущий оркестратор (диалог с человеком) | **Сильная, Fast-режим** | Opus 4.8 (Fast) | Ведёт весь флоу, задаёт уточняющие вопросы, принимает решения о треке, управляет субагентами. Fast-режим — чтобы ответы в диалоге не тормозили. |
| «Разведчик» — поиск файлов, прогон проверок | **Дешёвая / быстрая** | Haiku 4.5 (субагент) | Механические операции: найти файл, запустить команду, вернуть результат. Параллельный запуск нескольких разведчиков — дёшево и быстро. |
| Синтез результатов разведки, принятие решений | **Сильная** | Opus 4.8 | Собрать результаты от разведчиков, сделать вывод, решить что делать дальше — требует понимания контекста целиком. |

## Заметки по применению

**Параллельность.** Независимые задачи — запускать субагентами параллельно. Например: несколько разведчиков ищут нужные файлы одновременно; несколько имплементеров правят разные модули одновременно. Дожидаться результата одного перед запуском другого — только если есть явная зависимость.

**Кэш — беречь.** Большой контекст (primer платформы, `CLAUDE.md` фронт-репо, таксономия) грузить один раз в начале сессии. Не перечитывать те же файлы повторно внутри одного флоу — это сбрасывает кэш и увеличивает стоимость.

**Версии меняются, тиры — нет.** При обновлении Anthropic SDK или появлении новых моделей — обновить конкретные названия в таблице выше, логику тиров не трогать.

**Тривиальная транскрипция — иногда дешевле инлайн, чем субагент.** Если план уже содержит готовый код
для правки (1 файл, чистая транскрипция без решений) — round-trip до субагента-имплементера (даже
Haiku/Sonnet) может стоить дороже по времени, чем сделать инлайн: на практике имплементеры на такой
задаче шли ~4–5 минут, один подвис. Субагентов стоит резервировать для крупных/мультифайловых задач,
параллельной работы, или где реально помогает свежий контекст (не загрязнённый историей текущей сессии)
— не для «код уже расписан в плане, просто перенеси».
