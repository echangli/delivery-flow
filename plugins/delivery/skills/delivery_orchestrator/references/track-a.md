# Трек A — Новая фича/идея (полный флоу)

Применяется, когда человек описывает **новую** функцию, идею, карточку из бэклога или стрима.
Если неоднозначно — уточни у человека, прежде чем идти по этому треку.

Сквозь весь трек ведётся **лог заметок** — все затыки, тупики, зависания, нестандартные решения
записываем по ходу. Папку задачи `tasks/<тикет>-<слаг>/` создать на старте, лог — `notes.md` в ней
(раскладка — `workspace-layout.md`). В конце флоу предложить
отправить лог/ретро автору плагина через **GitHub Issue** (предзаполненная ссылка) для улучшения
процесса — см. SKILL.md § «Лог заметок».

## Стадия 0 — Преднастройка

### Что делает человек
- Подтверждает, что Claude Code установлен.
- При необходимости — поднимает дев-VPN (⟦VPN_CLIENT⟧), входит в git-хост (PAT со scope `api` + `write_repository`) и i18n-платформу (если есть).
- Выполняет шаги, которые агент не может выполнить сам (sudo/keychain, `/etc/hosts`, mkcert — интерактив).

### Что берёт агент
- Вызывает скилл **`delivery_setup`** — один раз на машину; если окружение уже проверено в текущей сессии, пропустить.
- **Первое действие внутри delivery_setup:** прочитать `CLAUDE.md` фронт-репо (⟦FRONTEND_REPO⟧) и основной под-гайд ⟪ADAPT: путь к под-гайду главного пакета, напр. `apps/web-client/CLAUDE.md`; если под-гайдов нет — убрать⟫ — это первоисточник всех правил кода и green-sequence.
- Затем проверяет: Node ⟪ADAPT: требуемая версия Node, напр. ≥24⟫ (`node -v`; ниже неё или битые нативные бинди → `brew install node@⟦NODE_MAJOR⟧` keg-only + **чистый ⟪ADAPT: bootstrap-команда для dev, напр. `bootstrap:dev`⟫ на нужной версии Node**), git, доступ к фронт-репо (≥ Developer, дев-VPN), MCP figma/notion (`/mcp`).
- **Аналитический MCP — по умолчанию не предлагать** (нужен редко, при необходимости подключат отдельно).
- При любом access-гейте: **пауза + понятная плашка** «что сделать / куда писать», не блокироваться молча.
  Куда писать: ⟦CHAT_CHANNEL⟧. Детали гейтов — `access-gates.md`.
- Критерий готовности: Node нужной версии ок, репо клонирован и CLAUDE.md прочитан, дев-VPN поднят, нужные MCP отвечают.

## Стадия 1 — Захват фичи/идеи

### Что делает человек
- Описывает фичу/идею **подробно**: что хочет, зачем нужно, для кого.
- **Прикладывает:**
  - скриншоты или макеты (экраны, к которым относится фича);
  - ссылки на Notion-фокусы/карточки (если есть — напр. «Focus» или «PM Inbox»);
  - ссылки на аналоги/референсы, если есть.
- Отвечает на уточняющие вопросы.

### Что берёт агент
- Подтягивает контекст: загружает `context/platform.md` (primer), при необходимости читает связанные карточки через **Notion MCP**.
- Делает **эталонные прод-скрины** затронутых поверхностей (через preview/screenshot) — референс для дизайна и сверки.
- Задаёт **простые уточняющие вопросы** (не технические термины): сценарий использования, что считается успехом, есть ли граничные случаи, что точно не входит в скоуп.
- Фиксирует итоговое описание в лог заметок.
- **Guardrail:** не двигаться дальше, пока описание не достаточно конкретно для дизайна.

## Стадия 2 — Дизайн + прототип

### Что делает человек
- Участвует в брейнсторме: отвечает на вопросы агента, уточняет требования.
- Проверяет прототипы, выбирает вариант (или просит доработать).
- **Делает ревью как дизайнер** (или передаёт дизайнеру): выбирает один из вариантов в Slack.

### Что берёт агент

**Шаг 2.1 — Брейнсторм и спека:**
- Вызывает **`superpowers:brainstorming`** для проработки идеи: сценарии использования, варианты UX-подхода, ограничения, edge-кейсы, acceptance criteria.
- Формирует краткую спеку (что, зачем, для кого, что не входит, AC).
- **Data-availability check перед метрикой/числом:** до фиксации показателя проверить, есть ли данные и какова их точная семантика по модели фронта (какое поле; сбрасывается ли; включает ли расчётные/нереализ. величины). Нет данных → не строить вслепую, эскалировать на бэк. Экономит раунды переделок смысла/лейбла.
- **«Конкретные сущности vs весь класс»:** если ТЗ называет конкретные id (сущностей, конфигураций и т.п.) — уточнить: «это именно эти или весь класс по типу/признаку?». Переобобщение («эти 2» → «все такого типа») придётся откатывать; allowlist id лучше держать объединённым prod+stage без env-детекта — в каждом окружении матчатся свои.
- **«Если возможно, прокинуть X в Y»:** такие интеграции проверять **по контракту целевого API до плана** — какие параметры принимает? Если нужного нет → фича бэк-зависима; явно помечать как риск в спеке и не включать в план без подтверждения.
> 🔧 **ADAPT — грабли ваших задач (data/scope/integration).** Сюда команда складывает продуктовые уроки из реальных задач (в исходном проекте тут были кейсы: метрика без данных на фронте → эскалация на бэк; переобобщение allowlist id; интеграция, упавшая на контракте API без нужного параметра). Три пункта выше — обобщённый формат-пример; заменяйте на свои по мере накопления.

**Шаг 2.2 — Прототип (вызов `delivery_prototype`):**
- Строит **кликабельный самодостаточный HTML-прототип** на дизайн-системе из фронт-репо:
  - Источник ДС — **только** ⟪ADAPT: путь к вашей дизайн-системе/ui-kit в фронт-репо (токены + компоненты), напр. `design-system/ui-kit`: tokens.css, src/tokens, src/\<Component>⟫; сторонние/отдельные ДС не использовать.
  - Только `var(--*)` токены, без хардкода hex/цветов.
  - Полный self-contained HTML (не фрагмент); картинки — data-URI или статик-сервер (`python3 -m http.server`).
  - brainstorm-компаньон отдаёт только новейший HTML — не полагаться на него для хранения статики.
- Рендерит прототип через preview/screenshot, сравнивает с **эталонными прод-скринами** (точность — частая правка ревью).

**Ключевое требование — отдать всё пространство выборов, а не один вариант** (даже если человек явно **не** просил «несколько вариантов» — это дефолт):
- **Дефолт — ОДИН самодостаточный HTML с демо-панелью**, тумблеры которой переключают **каждую** открытую развилку живьём: механика/состояния, плейсмент, подача под-блоков, лейбл/копи. Дизайнер собирает финал за один заход.
- **Отдельные full-page файлы — только для сравнения «бок-о-бок»** (обычно крупный плейсмент/лейаут); остальные развилки — всё равно тумблерами внутри. Не плодить файлы без нужды.
- Всё — **полные self-contained страницы** (не фрагменты). Цель: **дизайнер выбирает значения развилок** на ревью, не придумывает с нуля.
- Пример: один файл, тумблеры — кнопка (иконка/лейбл/размер) · плейсмент (рядом с блоком / в шапке / в меню) · состояния.
- Подтверждённые элементы тоже прототипируются («подтверждение ≠ не нужен прототип»). **Никогда — один зафиксированный вариант.** Детали и рецепты — `delivery_prototype` §5.

**Шаг 2.3 — Хэндофф дизайнеру:**
- Отдаёт прототипы через preview/screenshots.
- **Проактивно предлагает отдать прототип ссылкой** (деплой на Vercel) — дизайнеру удобнее открыть ссылку, чем HTML-файл; при правках ссылка та же. Рецепт и разовая настройка (аккаунт + токен) — `delivery_prototype` §7. Файл/локальный сервер — фолбэк.
- Пишет сообщение в публичный Slack (хэндофф-чат) с прикреплёнными скринами вариантов.
- **Явно просит дизайнера выбрать по каждой развилке** (плейсмент / лейбл / состояние-подход / подача — прожать тумблерами демо-панели), а не «какой из файлов»; либо сказать, что доработать.
- Записывает прототипы в папку задачи (`tasks/<тикет>-<слаг>/prototype.html`; скрины — `screenshots/`; раскладка — `workspace-layout.md`).
- Ждёт выбора дизайнера; при необходимости — дорабатывает выбранный вариант.
- **Restate-гейт (осознанность):** после выбора — человек одной строкой формулирует *какую развилку выбрал и почему* (в `notes.md`; зерно «why» для MR). Не экзамен — осознанность решения перед ревью фронтендера. Канон — `SKILL.md §8`.

## Стадия 4 — Реализация

### Что делает человек
- При необходимости — поднимает дев-VPN, помогает с git-кредами: **osxkeychain** — для интерактивного `git push` человека; агент пушит oauth2-URL и вызывает API git-хоста токеном `GIT_API_TOKEN` из `~/.zshenv` (не через keychain). PAT и токены настраиваются через **`delivery_setup §2.0`** (одна команда: env + keychain).
- Принимает решения по неоднозначным вопросам архитектуры или продуктовой логики.
- Если есть новые i18n-ключи и доступа к i18n-платформе нет — заводит ключи по списку, который подготовит агент.

### Что берёт агент
- Создаёт **git-worktree** фронт-репо (не работать в основной рабочей копии):
  использует **`superpowers:using-git-worktrees`** для создания изолированного worktree.
  **Перед созданием — `git fetch origin`, ветку делать от свежего `origin/main`**
  (`git worktree add -b feature-<slug> <path> origin/main`).
  Ветка — **`feature-<kebab-slug>`** ⟪ADAPT: если ваша конвенция веток иная — подставьте её (тип-префикс/формат из `repo-conventions.md`)⟫ (не `feat/`; без слеша).
  Если работа продолжается на уже существующей ветке — сверить с `origin/main` и **влить main, если
  отстала** (merge, не rebase; конфликты → пауза). Детали — `repo-conventions.md` § «Свежесть ветки».
- Реализует фичу с **`superpowers:subagent-driven-development`** и политикой тиров моделей (см. `model-policy.md`):
  - Дизайн/спека/ревью → Opus.
  - Субагенты-имплементеры: изолированная механика (1–2 файла) → Haiku; мультифайловая интеграция → Sonnet.
  - Параллелить независимые задачи; беречь кэш.
- Строго следует правилам из `repo-conventions.md` (источник — `CLAUDE.md`/под-гайд вашего фронт-репо):
  > 🔧 **ADAPT — стек-конвенции кода.** Здесь перечисляются жёсткие правила вашего фронт-стека, которые ревьюер бьёт на ревью. В исходном проекте это были: `FC`+`memo`+**named export** (без `export default`); деньги через `Decimal.fromNumber()` (ноль — `Decimal.zero`); `import type` отдельной группой; строки — `FormattedMessage` (бренд-литералы можно хардкодить); кросс-фича — только через barrel/`integration`; тесты импортируют `@jest/globals`. Впишите свои (детали держите в `repo-conventions.md`).
  - **Правка ОБЩЕГО (shared) компонента — явно развести скоуп.** Если ТЗ гейтит фичу по id/флагу, а
    изменение задевает shared-компонент, используемый другими консьюмерами (не только целевым) — разделить
    «что гейтится (новый контрол, за id/флагом)» от «что общий рестайл (задевает всех консьюмеров)».
    Общую часть — согласовать с продуктом **до** коммита, а не постфактум, и явно **отметить в описании
    MR** («shared change: X affects all consumers, confirmed with product»), иначе ревьюер прочитает это
    как незапрошенный scope creep. (Реальный кейс из исходного проекта: рестайл общего hero-блока применился ко **всем**
    консьюмерам, хотя гейтился только новый контрол.)
  - **Без поясняющих комментов в коде** (только очень специфичные — по запросу).
  - Коммиты — простые английские императивы (`Add …`, `Update …`), без conventional-prefix; сквоша нет, сообщения должны быть нормальными.
- **i18n (если есть i18n-платформа/TMS):** если есть новые `FormattedMessage`-ключи — показать список (id + defaultMessage + локали) и попросить завести через Slack, если у человека нет доступа к i18n-платформе; после заведения — сборка i18n (⟦I18N_CMD⟧) в ту же ветку. Нет мультиязычности — пропусти. Перед заведением — **копи-линт EN-исходника** (`repo-conventions.md §i18n → Копи-линт исходных EN-ключей`).
- **Аналитика:** имена событий/свойств — согласовать с аналитиком до коммита; формат place: `имя_фичи-{id}`.

## Стадия 5 — Локальная проверка

### Что делает человек
- Поднимает дев-VPN (если не поднят), логинится на стейдж или прод.
- Для mkcert/`/etc/hosts` — делает сам (sudo/keychain; агент не может).
- Отвечает QA в Slack по итогам смоука.

### Что берёт агент
- Запускает **green-sequence на нужной версии Node** (⟦NODE_MAJOR⟧; источник команд — ⟪ADAPT: где описана green-sequence, напр. `apps/web-client/docs/commands.md`⟫):
  > 🔧 **ADAPT — green-sequence.** Впишите точные команды проверки вашего репо (в исходном проекте: из пакета — `ci:type`, `ci:eslint`, `ci:stylelint`, `ci:test`, `i18n:extract`; из корня — `ci:unused` (knip), `oxfmt:check`). Полную последовательность гонять только на требуемой версии Node — на другой линтеры/форматтеры могут виснуть/падать.
- **Запускает dev-сервер — обязательный гейт перед MR, не опция:**
  ⟪ADAPT: команда запуска dev-сервера, напр. `npm start`⟫ → ⟪ADAPT: локальный URL dev-сервера, напр. `https://local.<...>:8443`⟫; визуальная сверка реализованной фичи с прототипом desktop+mobile **до push**. Пропускать нельзя — мелочи вёрстки, цвета, спейсинг ловятся только на живом сервере; без этого шага будут раунды визуальных правок уже после создания MR.
  - **Выбор API** (предлагает человеку):
    - **Stage** (⟪ADAPT: команда старта против stage-API, напр. `npm start`⟫, дефолт): ⟦STAGE_API_HOST⟧; нужен дев-VPN + stage-аккаунт.
    - **Prod** (⟪ADAPT: команда старта против prod-API, напр. `start:api:prod`⟫): ⟪ADAPT: prod-API-хост⟫, публичный; логин своим прод-аккаунтом, VPN к бэку не нужен.
  - В обоих вариантах: локальный HTTPS-хост (нужны `/etc/hosts` + mkcert — делает человек); дев-VPN нужен для GitLab/стейджа в любом случае.
- Делает **скриншоты** реализованной фичи и сравнивает с эталонными прод-скринами.
- **Агент может прокликать сам — предлагай это проактивно.** Через Playwright (как в `delivery_qa_smoke`: ⟪ADAPT: способ логина в вашем приложении, напр. вход тестовым аккаунтом/кошельком, при нужде — сид данных⟫) + `ignoreHTTPSErrors` (обходит mkcert cert-интерстишл) агент поднимает dev-сервер и проверяет фичу **сам**, в т.ч. за логином. Нужен лишь **минимум данных** от человека (какой экран/состояние/сценарий) — формулируй так: «подниму и проверю сам, скажи только X».
- **Где агент реально ограничен:** состояние, которое не поднять тестовым аккаунтом (специфичный прод-аккаунт/роль), и **пиксельная точность снапшот-зоны шер-картинки** (если есть) → доводим **итерациями по скринам человека**, не «на глаз».
> 🔧 **ADAPT — чек-лист шер-картинки / снапшот-зоны (если в продукте есть шеринг-картинки).** Если фича генерит картинку для шеринга — держите здесь чек-лист частых расхождений до показа. В исходном проекте это было: использовать компонент `Logo` (не текстовый бренд); только токены `var(--*)`; градиент — на контент-бэнде, не на всей карточке; боковые отступы как у соседних шер-механик; контролы (табы/период) **вне** snapshot-зоны `[data-share]`; одинаковая высота табов; единицы значений сверены с существующими компонентами. Нет шеринг-картинок — секцию удалить.
- **ui-kit custom-audit перед push — обязательный под-шаг, с отчётом.** После визуальной проверки сверить изменённые UI-элементы с вашей ДС/ui-kit и живыми платформенными паттернами на КАСТОМ (нестандартные шрифты/иконки/стили/разметка вместо компонентов и токенов ДС). Эталоны брать **из кода** (найти существующий канонический компонент под ту же роль — info-подсказка, бейдж и т.п. — и повторить его, а не собирать raw-вариант, дающий не-canonical шрифт/стиль). **Выдать пользователю отчёт:** есть ли кастом; если да — заменить на ui-kit, иначе согласовать с фронтендером (он бьёт за кастом на ревью). См. `repo-conventions.md` § Конвенции кода (DS-first).
- **i18n перед push (если у вас есть i18n-платформа/TMS и появились новые ключи) — ПОРЯДОК:** dev-server-проверка OK → завести ключи в i18n-платформе (файлами ⟪ADAPT: имена файлов локалей, напр. `locales/en_US.json`+`ru_RU.json`⟫) → ⟦I18N_CMD⟧ (на ветке, синхронизированной с `origin/main`; в дифе ⟪ADAPT: пути генерённых i18n-артефактов, напр. `generated`/`backups`⟫ — только свои ключи) → commit этих артефактов → **затем** push. НЕ откладывать i18n на «перед снятием draft» — иначе MR/стейдж покажут сырые ключи (`set of symbols`). Токен i18n-платформы — ⟦I18N_TOKEN_ENV⟧ в `~/.zshenv` (`delivery_setup`). Нет мультиязычности — пропусти этот шаг.
- **Автотест-импакт-чек (до push) — `autotest_impact`:** когда дифф трогает фронт-приложение, сверить, не задел ли он спеки e2e-сьюты (переименован/убран `data-test`, изменён лейбл/флоу/роут). 🔴 — показать человеку до push и решить: product-side аккомодация (сохранить id/лейбл → правка теста не нужна) или хэндофф QA. **Тесты не правим.** При доступном слоте — прогон-подтверждение (стадия 6/7).
- **Готовит материалы** для стейдж-смоука (чек-лист, что изменилось, куда смотреть, корнер-кейсы), но **сам `delivery_qa_smoke` на слоте/деплое запускается ПОСЛЕ push** (стадия 6 — слот/деплой требует запушенной ветки). На этой стадии (до push) — только локальная проверка и подготовка.
- После стейдж-смоука — **хэндофф QA в Slack** (описание изменений, скриншоты, чек-лист `- [ ]`); правки идут в ту же ветку feature-*.

## Стадия 6 — MR (draft)

### Что делает человек
- Открывает ссылку, которую агент подаёт текстом в чат.
- Проверяет предзаполненное описание MR, при необходимости дополняет.
- Создаёт MR (он уже в статусе Draft).
- Пересылает ссылку на MR фронтендеру для ревью (в GitLab или Slack).

### Что берёт агент
- **Сначала прочитать `repo-conventions.md` § MR** — требования к описанию (draft, EN, чек-лист `- [ ]`,
  предзаполненная markdown-ссылка) учесть ДО написания описания, а не подгонять после вопроса ревьюера.
- **Перед пушем — синхронизация с main:** `git fetch origin`; если ветка отстала от `origin/main` —
  влить (`git merge origin/main`; конфликты → пауза) и пере-прогнать green-sequence. Так MR будет чистым против main.
- Пушит ветку: `git push -u origin feature-<slug>` — **или отдаёт команды человеку, если он предпочитает пушить сам** (если из контекста не ясно, кто пушит — коротко спросить).
  `git push` даёт «To create a merge request…»-ссылку в выводе — сохранить.
- Формирует **ссылку создания MR** с предзаполненным описанием:
  ```
  https://⟦GIT_HOST⟧/⟦REPO_GROUP⟧/-/merge_requests/new
    ?merge_request[source_branch]=feature-<slug>
    &merge_request[target_branch]=main
    &merge_request[title]=Draft%3A%20<Title%20in%20English>
    &merge_request[description]=<url-encoded-description>
  ```
  - **Отдавать markdown-ссылкой с коротким лейблом, не голым URL** (`%`-кодированный URL в чате нечитаем):
    `[Создать draft-MR (title + description предзаполнены)](https://…)`.
  - Заголовок: `Draft: <краткое название на английском в стиле репо>`.
  - Описание (английский): что сделано, зачем, скриншоты (описание через `%0A`, не через push-опцию — push-опция не принимает переносы).
  - **Чек-лист `- [ ]` — только конкретные ручные проверки** (что нажать и что убедиться): новый функционал + что могло быть задето рядом (смежные экраны/состояния, общие компоненты). В GitLab рендерится интерактивными чекбоксами — они должны быть кликабельными «иди → нажми → убедись», не процессными шагами.
    **Не класть в чекбоксы:** `tested locally`, `green-sequence`, `i18n build`, `designer review`, `frontender review` — это шум для ревью. Процессные шаги пишутся отдельной прозой в секции «Before removing Draft», не чекбоксами. Принцип и пример — `track-b.md` § Стадия 6.
- **Готовый скелет описания — копировать и заполнить, не сочинять формат с нуля** (всё по-английски):
  ```markdown
  ## What & why
  <что сделано и зачем; ссылка на карточку/Notion>

  ## Screenshots
  <desktop + mobile>

  ## Check (functional impact)
  - [ ] <что должно работать после изменения — конкретное действие → результат>
  - [ ] <что рядом могло быть задето — смежный экран/состояние>
  ```
- **СТОП-самопроверка перед отдачей ссылки** (`repo-conventions.md` § MR): `[ ]` push сделан; `[ ]` `Draft:` + заголовок на EN; `[ ]` описание на EN; `[ ]` чек-лист `- [ ]` из конкретных проверок; `[ ]` markdown-ссылка, не голый URL. Не всё — сначала исправить.
- **Отдаёт ссылку текстом в чат** (не кликает сам, буфер затирается) и **подсказывает, где найти MR после создания:** в GitLab слева **Code → Merge requests** → вкладка **Open** (черновик с меткой `Draft`), твой MR сверху списка; прямая ссылка — `https://⟦GIT_HOST⟧/⟦REPO_GROUP⟧/-/merge_requests`.
- **Объясни человеку, что дальше** (простым языком): «создашь MR по ссылке → в нём пойдёт пайплайн (тесты + сборка) → я подниму временный стейдж и пришлю ссылку для проверки → после смоука передаёшь фронтендеру на ревью в GitLab → мерж и деплой делают QA/фронтендер после апрува, **не ты**». **И предложи:** «фичу на стейдже могу прокликать сам — скажи только, что проверить (актив/состояние)».
- Напоминает человеку передать ссылку на MR фронтендеру в GitLab для ревью.
- **Цикл ревью фронтендера:**
  - Фронтендер оставляет комментарии в GitLab.
  - Агент читает комментарии и вносит правки в ту же ветку.
  - Повторить до апрува.

## Стадия 7 — Апрув → передача

### Что делает человек
- Получает апрув от фронтендера в GitLab.
- Согласовывает с QA или фронтендером время мержа и деплоя.

### Что берёт агент
- Фиксирует факт апрува в лог заметок.
- **Мерж и деплой НЕ выполняет.** Флоу на этом завершается.
- Выводит итоговое сообщение: MR апрувнут, ссылка на MR, напоминание — «мержат QA или фронтендер после апрува».
- Предлагает отправить лог заметок/ретро (затыки/тупики флоу) автору через **GitHub Issue**
  (предзаполненная ссылка, markdown-ссылкой в чат) для улучшения процесса — см. SKILL.md § «Лог заметок».

## Сквозные правила (guardrails)

- **Никогда не мержить и не деплоить** — только QA или фронтендер после апрува фронтендера.
- Работа ведётся в **git-worktree** фронт-репо; в основную рабочую копию не лезть.
- **Green-sequence только на требуемой версии Node** (⟦NODE_MAJOR⟧) — на другой версии чеки виснут/падают.
- **Без комментов в коде** (фидбэк фронтендера; только очень специфичные по запросу).
- **DS-first: проверять ui-kit перед версткой, не плодить кастом** (фидбэк фронтендера) — переиспользовать/расширять DS-компоненты вместо локальных кнопок/чипсов/слайдеров. Детали — `repo-conventions.md` § Конвенции кода.
- Коммиты и MR — на **английском** в стиле репо.
- Общаться с человеком на **его языке** (по умолчанию русский).
- **Скриншоты + сравнение с продом** на стадиях прототипирования и локальной проверки.
- При любом затыке/зависании → в лог заметок.
- При access-гейте → **пауза + плашка**, не молча блокироваться; куда писать — `access-gates.md`.
