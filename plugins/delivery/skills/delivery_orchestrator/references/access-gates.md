# Access-гейты: когда нужна рука человека

## Паттерн поведения

Агент не умеет получать доступ за вас. Когда что-то заблокировано:

1. **Остановиться и показать плашку** — чётко: что нужно, что сделать, куда написать.
2. Дождаться подтверждения («готово» / «сделано») и **продолжить флоу**.
3. **Никогда не зависать молча** — если непонятно, что делать, написать об этом явно.

Где агент может помочь сам — помогает через терминал (с пояснениями). Где нужны sudo, keychain, GUI или логин — отдаёт точные инструкции человеку.

Каналы связи: ваш командный чат для запросов доступа — ⟪ADAPT: ваш чат-канал для запросов доступа⟫ (напр. общий Slack-канал команды; ссылки не хардкодятся — найти по названию).

> 🔧 **ADAPT — чат-каналы.** Ниже по файлу вместо `⟪ADAPT: ваш чат-канал для запросов доступа⟫` подставь свой(и) командный(е) канал(ы), куда просить VPN/доступ к репо/аккаунт/i18n.

## Гейты

### 1. dev-VPN (VPN-клиент)

> 🔧 **ADAPT — VPN опционален.** Нужен, только если рабочий git-хост/стейдж команды за корпоративным VPN.
> Если репо и стейдж публично доступны — удали этот гейт.

| | |
|---|---|
| **Что нужно** | Подключённый корпоративный VPN через ⟪ADAPT: ваш VPN-клиент⟫ |
| **Зачем** | Доступ к рабочему git-хосту (⟪ADAPT: ваш git-хост⟫) и стейдж-окружению |
| **Признак отсутствия** | `git push` / `git clone` зависает или отдаёт `Connection refused`; стейдж-URL недоступен |
| **Что делает человек** | Открыть ⟪ADAPT: ваш VPN-клиент⟫ → подключиться к корпоративному VPN |
| **Куда писать, если нет VPN-клиента** | ⟪ADAPT: ваш чат-канал для запросов доступа⟫ — попросить доступ к VPN |

> **Плашка (показывать при отсутствии VPN):**
> ```
> ⛔ Нужен dev-VPN
> Подключитесь через VPN-клиент и напишите «готово».
> Если VPN-клиент не установлен — напишите в ⟪ADAPT: ваш чат-канал для запросов доступа⟫.
> ```

### 2. Доступ к фронт-репо (⟪ADAPT: группа/путь вашего фронт-репо, напр. `<group>/<repo>`⟫)

| | |
|---|---|
| **Что нужно** | Роль **Developer** — минимально достаточный и **безопасный** уровень: push feature/bugfix-ветки + создание MR. Developer **не может** мержить в protected `main` и деплоить — это совпадает с guardrail «не мержим/не деплоим» |
| **Признак отсутствия** | `git push` отдаёт `403 Forbidden` или `You are not allowed to push code to protected branches` |
| **Кто выдаёт** | ⟪ADAPT: кто в команде раздаёт Developer-роли⟫ — попросить у него в личке или в общем командном чате. Запасной канал — ⟪ADAPT: ваш чат-канал для запросов доступа⟫ |
| **Если Developer не дают** | Согласовать с разработчиком альтернативный процесс: он пушит ветку / создаёт MR от вашего имени |

> **Безопасность держится не на роли, а на защите `main`:** `main` должен быть protected
> (merge — только Maintainer + required approval фронтендера), prod-секреты — в protected
> variables/environments. Тогда Developer физически не сломает main и не выкатит прод.

> **Роль Developer есть, а `git push` всё равно `403`** — частая причина: HTTPS-токен (PAT) создан
> с read-only-скоупом (хватает лишь на clone). Нужен scope, покрывающий и push, и API/авто-деплой
> (⟪ADAPT: конкретные scope-имена вашего git-хоста, напр. у GitLab `api`; как минимум `write_repository`⟫;
> см. гейт §6 — разовый вход в git-хост).

> **Авто-деплой временного стейджа** (`delivery_qa_smoke`) — **основной путь проверки**: деплой и забор
> ссылки делает агент токеном с API-скоупом (⟪ADAPT: если у вашего git-хоста push-scope не даёт прав на
> API/деплой — нужен именно API-scope; у GitLab на `write_repository` API отвечает `403`⟫). Фолбэк —
> человек запускает деплой временного стейджа и открывает приложение в UI git-хоста сам, присылает ссылку.

> **Плашка:**
> ```
> ⛔ Нет доступа к фронт-репо
> Нужна роль Developer на ⟪ADAPT: группа/путь фронт-репо⟫ (push ветки + MR; без мержа в main).
> Developer-роли раздаёт ⟪ADAPT: кто в команде⟫ — напишите ему в личку или в общий командный чат
> (запасной канал — ⟪ADAPT: ваш чат-канал для запросов доступа⟫).
> Если Developer не дают — договоритесь с разработчиком об альтернативном пуше/MR.
> ```

> **Протух git-кред (не путать с ролью):** push падает `HTTP Basic: Access denied` (а **не** `403 protected`)
> = истёк/недоскоуплен токен в osxkeychain (бывает read-only: `fetch` идёт, `push` — нет).
> **osxkeychain** — для интерактивного `git push` человека; агент креды в helper не вводит.
> Агент пушит oauth2-URL с токеном из `GIT_API_TOKEN` (`~/.zshenv`):
> `git push "https://oauth2:$GIT_API_TOKEN@⟪ADAPT: ваш git-хост⟫/<group>/<repo>.git" <branch>`.
> Токен из чата — отозвать после. Детали — `repo-conventions.md` §Окружение.

### 4. MCP (Figma / Notion)

| | |
|---|---|
| **Что нужно** | Авторизованный MCP `figma` и/или `notion` — **только если задаче реально нужен** (notion при карточке Notion, figma при макете) |
| **Как проверяет агент** | Сам, из Bash: `claude mcp list` (статус), `claude mcp get <name>` (Type/Command). Интерактивную панель `/mcp` агент открыть не может |
| **Частый затык — битый конфиг** | `claude mcp get` показывает `Type: stdio` + `Command: https://…` → клиент пытается запустить URL как программу (`ENOENT`/`Failed to connect`). **Агент чинит сам:** `claude mcp remove <name> -s user` → `claude mcp add --transport http --scope user <name> <url>`. Для hosted-MCP всегда `--transport http` |
| **Что делает человек (OAuth)** | Если сервер сконфигурён, но не авторизован — пройти `/mcp → <коннектор> → Authenticate`. ⚠️ **Из десктоп-приложения OAuth часто не доходит — делать через терминальный `claude`** (или раздел Connectors). Возможен **рестарт** сессии, чтобы тулзы подхватились |
| **Фолбэк (равноправный)** | Не блокировать флоу: попросить человека вставить содержимое карточки/макета прямо в чат и идти дальше |
| **Если не получается** | Написать в ⟪ADAPT: ваш чат-канал для запросов доступа⟫ |

> **Аналитический MCP — по умолчанию не предлагать.** Подключается только если человек явно запрашивает аналитику.

> **Не бойся рестарта.** Артефакты (лог, спека, прототипы, worktree) на диске → рестарт теряет только чат, не работу. Возобновление — заново `/deliver`, агент читает лог. Поэтому MCP-чек делай рано.

> **Плашка (MCP нужен, но не готов):**
> ```
> ⚠️ MCP [figma/notion] не готов
> Агент проверил: claude mcp list → [нет в списке / Failed to connect].
> • Если конфиг битый — агент починит сам (add --transport http).
> • Нужен вход: /mcp → [коннектор] → Authenticate — ЧЕРЕЗ ТЕРМИНАЛЬНЫЙ claude
>   (из десктопа OAuth часто не проходит), затем рестарт сессии.
> • Не хочется возиться — вставьте содержимое карточки/макета в чат, продолжим.
> Не помогает — напишите в ⟪ADAPT: ваш чат-канал для запросов доступа⟫.
> ```

### 5. Прод-i18n-платформа (переводы / i18n)

> 🔧 **ADAPT — весь гейт опционален.** Нужен, **только** если у вас есть i18n-платформа/TMS для переводов.
> Детали ниже (имя env-токена, хост, роль, команды) — иллюстрация одного из возможных стеков; замени на
> свою платформу. **Нет мультиязычности — удали эту секцию.**

| | |
|---|---|
| **Что нужно** | Доступ к вашей продуктовой i18n-платформе для добавления новых i18n-ключей |
| **Когда нужно** | **Только** если в коде появились новые `FormattedMessage`-ключи (не переиспользуемые) |
| **Признак отсутствия** | Нет логина к i18n-платформе; доступ есть не у всех |
| **Токен для сборки i18n** | ⟪ADAPT: env-переменная токена вашей i18n-платформы⟫ (+ хост, если нужен) — настраиваются один раз командой `delivery_setup §2.0`. **Токен — не слать в чат**; прислали — использовать и сразу предложить перевыпустить |
| **⚠️ Токен короткоживущий — частый блокер** | ⟪ADAPT: команда вашей i18n-платформы⟫ → **401** / `Api token not provided` = токен протух. Это самая частая причина упавшей сборки i18n. **Перед каждой сборкой i18n убедиться, что токен свежий**: ⟪ADAPT: UI/хост вашей i18n-платформы⟫ → Auth Tokens → создать с ролью ⟪ADAPT: роль/scope токена⟫ → заново запустить `delivery_setup §2.0` или обновить ⟪ADAPT: env-переменная токена вашей i18n-платформы⟫ в `~/.zshenv` вручную. |
| **Что делает человек** | Заводит/обновляет ключи в i18n-платформе |
| **Если доступа нет** | **Сперва** проверить ключи на i18n-readiness/voice (см. `repo-conventions.md §i18n`) и применить Critical-правки. Затем агент отдаёт файлы ключей (`locales/*.json`) / список новых ключей в формате `id + defaultMessage + список локалей` и просит завести их через ⟪ADAPT: ваш чат-канал для запросов доступа⟫. После заведения — запустить сборку i18n (попадёт в ту же ветку MR). Детали формата i18n — в `repo-conventions.md`. |

> **Плашка (нет доступа к i18n-платформе):**
> ```
> ⚠️ Нужны новые i18n-ключи, доступа к i18n-платформе нет
>
> Ключи для заведения (файлы locales/*.json / список):
> - id: <key_id>
>   defaultMessage: "<текст>"
>   locales: [en, ru, ...]
>
> Попросите завести их в ⟪ADAPT: ваш чат-канал для запросов доступа⟫.
> После подтверждения скажите «готово» — запущу сборку i18n.
> ```

> **В git-worktree `.env` нет** (gitignored) → сборка i18n из worktree упадёт `Api token not provided`.
> Решение: ⟪ADAPT: env-переменная токена вашей i18n-платформы⟫ в `~/.zshenv` (настраивается в `delivery_setup §2.0`) — агент подхватывает во всех worktree автоматически. Детали — `repo-conventions.md` §i18n.

**Как добавить/поправить перевод — по шагам (простым языком, для неопытных):**

1. **Токен i18n-платформы** (один раз на машину): настраивается через `delivery_setup §2.0` (одна команда).
   Если токен протух — пересоздать на **⟪ADAPT: UI/хост вашей i18n-платформы⟫**
   (роль ⟪ADAPT: роль/scope токена⟫) и обновить ⟪ADAPT: env-переменная токена вашей i18n-платформы⟫ в `~/.zshenv`.
2. **Завести ключи в самой i18n-платформе:** для каждого нового ключа задать текст на EN + RU (+ другие языки).
3. **В коде:** `<FormattedMessage id="my_key" defaultMessage="English text" />`. `defaultMessage` должен
   совпадать с EN из i18n-платформы (иначе i18n-проверка ругнётся `Default message is outdated`).
4. **Подтянуть переводы:** из ⟪ADAPT: пакет/директория для сборки i18n⟫ — ⟪ADAPT: команда вашей i18n-платформы⟫. Он скачает переводы из i18n-платформы в
   ⟪ADAPT: путь к генерённым словарям, напр. `src/i18n/generated/<locale>.json`⟫.
5. **Закоммитить генерённые i18n-артефакты (⟪ADAPT: пути, напр. `src/i18n/generated/**` и `backups/i18n/**`⟫) в ветку MR — это ключевой шаг.** Стейдж берёт
   переводы из **закоммиченных генерённых словарей**, а не из i18n-платформы напрямую. Без коммита на стейдже будет
   «набор символов» (минифицированный id вместо текста).
6. Проверить: i18n-проверка (⟪ADAPT: команда вашей i18n-платформы⟫) чиста по затронутым ключам; на стейдже строки локализованы.

### 6. Установка плагина из приватного репо плагина (marketplace)

> **Marketplace плагина** — приватный GitHub-репо; для *скачивания плагина* VPN не нужен, нужен лишь
> доступ к этому GitHub-репо. Это **не** рабочий git-хост команды (для него — гейт §2).

| | |
|---|---|
| **Что нужно** | Доступ к приватному GitHub-репо плагина (delivery-toolkit) для установки и автообновления плагина |
| **Признак отсутствия** | `/plugin marketplace add <url>` возвращает ошибку авторизации |
| **Что делает человек** | GitHub-доступ к репо marketplace (или PAT, если репо приватное и клиент просит логин); отдельно — PAT рабочего git-хоста для *реализации* (гейт §2). Токены заводятся через **`delivery_setup §2.0` одной командой** — она кладёт токен и в `~/.zshenv`, и в osxkeychain. |
| **Кто выдаёт** | **Доступ к GitHub-репо плагина выдаёт ⟪ADAPT: кто в команде даёт доступ к репо плагина⟫** — попросить у него в личке или общем командном чате (запасной канал — ⟪ADAPT: ваш чат-канал для запросов доступа⟫) |
| **Если нет доступа к репо** | Технический минимум для *использования* плагина — доступ на чтение (clone/pull) к GitHub-репо marketplace; write-доступ обязателен лишь чтобы *править* сам плагин |

> **Плашка:**
> ```
> ⛔ Нет доступа к репо плагина (delivery-toolkit)
> Доступ на чтение выдаёт ⟪ADAPT: кто в команде даёт доступ к репо плагина⟫ — напишите ему в личку или
> общий командный чат (запасной канал — ⟪ADAPT: ваш чат-канал для запросов доступа⟫).
> Токены (PAT рабочего git-хоста (+ опц. токен i18n-платформы)) настраиваются одной командой в delivery_setup §2.0.
> ```
