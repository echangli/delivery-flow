# Раскладка артефактов задачи и порядок в рабочей папке

Единый канон, как раскладывать артефакты флоу доставки и как навести порядок, если их насыпали
плоско. На этот файл ссылаются `delivery_orchestrator` (§ Раскладка артефактов задачи),
`delivery_prototype`, `delivery_setup`, треки A/B и команда `/delivery_tidy`.

---

## 1. Дефолт — одна папка на задачу

Все артефакты одной задачи — в одной папке, в **рабочей папке пользователя** (там, где живут
проект/задачи), **не в папке плагина** и **не в репо проекта**:

```
tasks/<тикет>-<слаг>/          # без тикета — просто <слаг> (напр. share-card)
  design.md                    # спека/PRD: что, зачем, для кого, AC, тест-план, edge-кейсы
  plan.md                      # план реализации (декомпозиция, порядок green-sequence)
  prototype.html               # кликабельный прототип на токенах ДС (демо-панель всех развилок)
  prototype-<вариант>.html     # ТОЛЬКО для сравнения бок-о-бок; иначе развилки — тумблерами в одном файле
  screenshots/                 # эталонные прод-скрины + скрины прототипа/реализации
  notes.md                     # лог заметок флоу: затыки, тупики, нестандартные решения
  retro.md                     # ретро-синтез (actionable-правки скиллов, 🔴/🟡/🟢)
  MR.md                        # черновик описания MR/PR (опц.)
  locales/                     # i18n: файлы локалей (опц. TMS) — если есть новые ключи
  .env                         # локальные секреты задачи (deploy-токен и т.п.); в .gitignore
```

- **Слаг** — короткое kebab-имя задачи ⟪ADAPT: примеры слагов вашего продукта⟫. С тикетом —
  `<ТИКЕТ>-<слаг>`; без трекера — просто `<слаг>`.
- **Не коммитить в репо проекта:** прототипы, скрины, `.env`, `locales/`. (В git проекта идут только
  сгенерированные артефакты i18n, если применимо; сами `locales/*.json` — вход для TMS + артефакт задачи.)
- Все имена — **короткие и предсказуемые**: одна задача → одна папка → одни и те же файлы. Не плодить
  `design-v2-final.md` и датированные копии внутри папки задачи (папка уже привязана к задаче).

---

## 2. Фолбэк — плоские пути (когда папки задачи ещё нет)

Быстрый фикс без папки / несколько мелких задач за день — допустимы плоские пути в рабочей папке:

```
prototypes/YYYY-MM-DD-<задача>.html
prototypes/screenshots/…
delivery-notes-YYYY-MM-DD.md
delivery-retro-YYYY-MM-DD-<задача>.md
```

Как только задача обрастает несколькими артефактами (спека + план + прототип + скрины) — **свернуть в
`tasks/<слаг>/`** по раскладке из §1.

---

## 3. Workspace-инструкции переопределяют дефолт

Если в `CLAUDE.md`/`AGENTS.md` воркспейса задана своя раскладка — **следовать ей** (она приоритетнее
дефолта плагина). Частый случай: workspace фиксирует `tasks/<тикет>-<слаг>/` с именами из §1 как
жёсткий стандарт — тогда плоский фолбэк §2 не применять.

**Гигиена `CLAUDE.md`/`AGENTS.md`** (файл инструкций грузится в контекст **каждую** сессию — это
инструкции, а не журнал задач):

- Детали задачи (спека/план/решения/скрины) — в **папке задачи**, не в `CLAUDE.md`.
- Список задач в `CLAUDE.md` — держать **только активные** (кратко); после мержа **свернуть до одной
  строки** (✅ тайтл · MR · указатель на папку) или вынести в `tasks/DONE.md`.
- Переиспользуемые уроки → **references плагина / память**, не в запись задачи.
- **Почему:** копящиеся завершённые задачи раздувают контекст каждой сессии и размывают внимание на
  главном (пайплайн/конвенции).

## 4. Навести порядок (миграция существующей свалки)

Процедура для **авто-предложения в `delivery_setup`** и для команды **`/delivery_tidy`**. Один
источник — правь здесь, оба места ссылаются сюда.

### Когда предлагать (авто, в delivery_setup)

Рабочая папка выглядит как свалка — в корне вперемешку лежат `*.html`-прототипы,
`delivery-notes-*.md`, папка `prototypes/`, скрины, спеки/планы **без группировки по задачам**; или
задачи ведутся в произвольном месте (Desktop, Downloads, корень home). Тогда — **предложить** навести
порядок. Не делать молча и не навязывать: коротко показать, что нашли, и спросить.

### Шаги уборки

1. **Определить рабочую папку флоу.** Если задачи ведутся где попало — предложить завести **отдельную
   папку под флоу** (напр. `~/<команда>-delivery/`) и дальше вести задачи там.
2. **Сгруппировать по задачам.** По слагу/дате в именах (`…-share-card.html`,
   `delivery-notes-2026-07-01.md`) разложить артефакты по `tasks/<слаг>/`.
3. **Показать план переноса** (что → куда) и получить **подтверждение до перемещения**.
4. **Перенести:** под git — `git mv`; иначе `mkdir -p tasks/<слаг> && mv …`. Соответствие имён:
   прототип → `prototype.html`, скрины → `screenshots/`, заметки → `notes.md`, ретро → `retro.md`,
   спека → `design.md`, план → `plan.md`, черновик MR/PR → `MR.md`, i18n → `locales/`.
5. **Неоднозначное не угадывать** — если непонятно, к какой задаче относится файл, оставить на месте и
   спросить человека.
6. **Итог:** короткий отчёт «что куда собрано»; напомнить, что новые задачи начинать сразу в
   `tasks/<слаг>/`.

### Guardrails уборки

- **Ничего не удалять** — только перемещать. При сомнении — спросить.
- **Не трогать git-репозитории проектов** (код-репо, worktree и пр.) — уборка касается только
  артефактов флоу в рабочей папке.
- **Секреты** (`.env`) — не коммитить и не светить в чат; убедиться, что `.env` в `.gitignore`.
- Если workspace-инструкции (§3) задают иную раскладку — приводить к **ней**, а не к дефолту §1.
