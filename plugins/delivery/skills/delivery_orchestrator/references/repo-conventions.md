> - ⟪ADAPT: главный гайд приложения (напр. `apps/<app>/CLAUDE.md`): TL;DR жёстких правил,

> первоисточник во фронт-репо.

> 🔧 **ADAPT — несколько репо/стеков.** Если команда доставляет в несколько репозиториев с разными

## Node

⟪ADAPT: версия Node (напр. `.nvmrc` `24.5.0`, `engines: { node: ">=24" }`). Подставь свою.⟫

На **неправильной версии Node** инструменты линтеров/форматтеров/анализаторов (`eslint`, `stylelint`,
`knip`, `oxfmt` и т.п.) **виснут или падают** — не из-за кода, а потому что npm не устанавливает нативные
бинари под платформу (`darwin-arm64`) при несовпадении версии (баг optional-deps npm). Ошибка выглядит как
«binary not found» или просто бесконечное зависание.

**Как чинить:**
1. `brew install node@⟪ADAPT: версия⟫` (keg-only — нормально, работает).
2. Переключиться на нужный Node (`nvm use ⟪ADAPT: версия⟫` / `export PATH=...`).
3. Сделать **чистый** bootstrap (⟪ADAPT: команда, напр. `bootstrap:dev`⟫) **на правильном Node**
   (удалить `node_modules`, переустановить). Нельзя просто переключить Node и запустить проверки —
   нативные бинари уже битые.

Проверки типов и тестов менее требовательны к версии Node и могут пройти на неидеальной, но **полную
green-sequence — запускать только на целевой версии Node**.

## Green-sequence

«Зелёная последовательность» перед коммитом:

> 🔧 **ADAPT — команды CI-проверок.** Подставь команды своего репо. Типичный набор (пример):
> из приложения (быстрее) — `ci:type` (TypeScript), `ci:eslint`, `ci:stylelint`, `ci:test`,
> `i18n:extract` (если менялись i18n-сообщения); из корня репо (по всем воркспейсам) — `ci:unused` (knip:
> мёртвый код/неиспользуемые импорты), `oxfmt:check` (форматирование). Полная карта команд — в доках репо.

```
# из директории приложения:
npm run ⟪ADAPT: типы⟫
npm run ⟪ADAPT: eslint⟫
npm run ⟪ADAPT: stylelint⟫
npm run ⟪ADAPT: тесты⟫
npm run ⟪ADAPT: i18n-extract, если есть⟫
# из корня репо:
npm run ⟪ADAPT: анализ мёртвого кода⟫
npm run ⟪ADAPT: проверка форматирования⟫
```

⟪ADAPT: отметь, какие команды запускаются из корня, а какие из приложения — часто knip и форматтер
работают корректно только из корня.⟫

**Для не-TS правок** (yaml/CI-конфиг, markdown, json) из всей последовательности применима обычно только
**проверка форматирования** (если форматтер трогает `.yaml`/`.md`) — остальное нерелевантно. Нюанс: в
свежем worktree без `node_modules` гонять форматтер бинарём из основного клона репо.

Агент может запускать всё это сам через терминал. Полная последовательность — **только на целевой версии Node**.

**Не маскировать exit code пайпами.** `npm run <check> | tail -5` скрывает реальный `exit 1` проверки — пайп возвращает код `tail`, не команды. Авторитетный прогон = `&&`-цепочкой **без `| tail`/`| head`**. Если нужен хвост вывода — `set -o pipefail` или проверять `${PIPESTATUS[0]}` вручную.

**Полный субсет перед коммитом каждой задачи.** Не ограничиваться типами/тестами — гонять типы+eslint+stylelint+формат на каждой задаче. Lint/format-долг, запущенный только в конце, накапливается и даёт ложный green на промежуточных этапах.

> 🔧 **ADAPT — ESM-нюанс тестов.** Частая ловушка: тесты гоняются только через обёртку раннера
> (напр. `npm run ci:test -- <path>`), а не голым `npx jest` — обёртка ставит
> `NODE_OPTIONS=--experimental-vm-modules`, обязательный для ESM-диста shared-пакетов. Если у тебя монорепо
> с ESM-пакетами — вероятно, та же ловушка; вписывай под свой раннер.

## Dev-сервер

⟪ADAPT: команда запуска и URL дев-сервера (напр. `npm start` → `https://local.example.dev:8443`).⟫

**Обязательные предусловия** (если dev-сервер на локальном HTTPS-хосте):
- `/etc/hosts`: строка `127.0.0.1 ⟪ADAPT: локальный хост⟫` (добавляется с `sudo` — делает человек).
- SSL-сертификаты: ⟪ADAPT: команда генерации, напр. `bootstrap:certs` (mkcert; добавляет в keychain —
  делает человек, требует sudo/интерактива)⟫.

**Выбор API (предложить человеку):**

| Вариант | Команда | API-хост | Нужен дев-VPN к бэку? | Логин |
|---|---|---|---|---|
| **stage** (по умолчанию) | ⟪ADAPT: команда⟫ | ⟪ADAPT: stage-API-хост⟫ | Да | stage-аккаунт |
| **prod** | ⟪ADAPT: команда prod-API⟫ | ⟪ADAPT: prod-API-хост⟫ | ⟪ADAPT: публичный?⟫ | свой прод-аккаунт |

В обоих случаях нужен локальный HTTPS-хост. Дев-VPN **в любом случае нужен** (если инфра за VPN) для
доступа к рабочему git-хосту и стейдж-окружению.

**Без дев-VPN (вариант stage) dev-сервер бесполезен — белый экран и не залогиниться:** `/api` и `/socket`
проксируются на стейдж-хосты, соц-логин тоже идёт через VPN. **Перед выдачей URL человеку — health-check
бэка** (не таймаут = VPN поднят):
```
curl -sk -m8 -o /dev/null -w '%{http_code}\n' https://⟪ADAPT: stage-API-хост⟫
```
Таймаут / `000` → поднять VPN-клиент (или переключиться на prod-API, если бэк публичный).

**mkcert/hosts — sudo/keychain → делает человек**, агент не может.

**Запускать detached** (фоновый процесс, переживающий ход/сессию): иначе дев-сервер умирает между ходами,
и в браузере — «ошибки сокета» / страница не открывается. Симптом → проверить процесс на порту
(`lsof -nP -iTCP:⟪ADAPT: порт⟫ -sTCP:LISTEN`) и перезапустить detached.

**Локалку через preview-инструмент агент часто прокликать не может** (preview виснет на долгом dev-сервере).
**Но через Playwright — может, и это надо предлагать:** поднять dev-сервер detached, зайти Playwright-ом с
`ignoreHTTPSErrors: true` (обходит mkcert cert-интерстишл), залогиниться (как `delivery_qa_smoke`), при
нужде подготовить состояние и выполнить действия. Так агент проверяет фичу **локально сам** — нужен лишь
минимум данных от человека (какой сценарий/состояние). Где Playwright не вытягивает (специфичный
аккаунт/роль, пиксельная точность снапшот-зоны) — **точный кликовый тест-кейс человеку**, и **не заявлять
«проверено»** по не-прокликанному рантайму. Статик-гейты (типы/eslint/тесты/анализ) обязательны в любом случае.

## Конвенции кода

> **Источник правды — актуальные фронт-доки, не эта выжимка.** Правила кода меняются (фронтендер льёт);
> `delivery_setup` стадия 0 читает их и флагует свежие правки. Здесь — **навигация** к источникам +
> **грабли из задач**, которых в доках нет. На стадии реализации/перед MR сверяйся с источником, не с памятью.

> 🔧 **ADAPT — карта правил кода.** Заполни таблицей «Тема → где во фронт-репо». Пример типичных тем:
> слои (UI→Models→Repos→Transport) и boundaries; шаблон компонента (`FC`+`memo`+named export, inline через
> `useMemo`/`useCallback`); DS-first (переиспользовать дизайн-систему, не плодить кастом); деньги через
> `Decimal`; `import type`; запреты (`@ts-ignore`, `export default`, `console`, комменты-пересказы, `TODO`
> без тикета, потеря аналитики, ручная правка сгенерированного i18n); строки/i18n; стили (CSS Modules,
> `var(--*)`, адаптив); тесты; чек-лист перед коммитом. Впиши свои источники и правила.

**Грабли из реальных задач** (переносимые уроки — держим здесь):

- **Новая точка входа, переиспующая чужой guard, ловит состояния, которых старый флоу не знал.** Реальный
  кейс: скопировали guard `if (account)` из одного оверлея в другой, но новый экран открывался и для
  «не добавленного» объекта (`account === null`) → кнопка молча ничего не делала. Сверяй **все достижимые**
  состояния новой точки (added/not-added, есть/нет данных/сессии, zero-data); решение бери из платформы
  **целиком**, а не копируй только хвост happy-path. **То же про тексты на shared-компоненте:** статичный
  лейбл/заглушка, переиспользуемый в разных состояниях, должен быть истинным во **всех** — иначе разведи
  текст по состояниям (детали — `track-c.md` § правка UI-текстов).
- **Лимиты числового ввода — нативной валидацией на уровне инпута, не пост-клампом.** Контролируемое
  значение маскированного инпута при активном вводе **не всегда** сбрасывает текст обратно → пост-кламп
  через `onChange`/blur «не ограничивает». Отвергай ввод/вставку сверх лимита на уровне инпута
  (напр. проп `isAllowed` у `NumericFormat` из `react-number-format`).
- **Расширяешь shared-компонент опциональным пропом — давай стабильный дефолт.** При строгих опциональных
  типах (`exactOptionalPropertyTypes`) проброс `prop?: T` в обязательный (в типе библиотеки) проп падает на
  `undefined`. Модульная константа-дефолт снимает это и не меняет поведение для прочих консьюмеров.

> 🔧 **ADAPT — грабли ваших задач.** Дописывай сюда уроки из своих реальных задач (то, чего нет во
> фронт-доках) — по мере накопления.

## Нейминг (git-конвенция — обычно enforced хуками + CI)

> ⚠️ **Единая конвенция для людей и Claude-агентов, часто проверяется автоматикой.** Типично локальные
> хуки `commit-msg`/`pre-push` ставятся на `npm install` (`core.hooksPath`), та же проверка — в CI на
> каждый MR; нарушение → пуш/MR **отклонён**.
> 🔧 **ADAPT — впиши git-конвенцию своего репо** (источник правды — гайд репо + его линтер веток/коммитов).
> Ниже — рабочая выжимка формата (пример); замени под себя. Эти правила надо знать ДО коммита.

**Ветки:** ⟪ADAPT: схема веток. Пример: `<type>-[<TICKET>-]<kebab>`, type ∈ `feature`·`bugfix`·
`hotfix`·`tech`; TICKET — опц. ключ трекера ЗАГЛАВНЫМИ (`DEV-123`); регекс
`^(feature|bugfix|hotfix|tech)-([A-Z][A-Z0-9]*-[0-9]+-)?[a-z0-9]+(-[a-z0-9]+)*$`. Префикс `claude/` и
суффиксы версий не использовать; `main` защищена.⟫

**Коммиты:** ⟪ADAPT: правила субджекта. Пример: заглавная + английский императив, ≤72 символов, без
точки в конце, опц. тикет-префикс `DEV-123 Add …`; тело — только если «почему» неочевидно; Conventional
Commits (`feat:`/`fix:`/`chore:`) НЕ используются; регекс `^([A-Z][A-Z0-9]*-[0-9]+ )?[A-Z].*[^.]$`. Учти,
делает ли репо squash при мерже.⟫

**🔴 Claude-агентам — иначе хук/CI ОТКЛОНИТ пуш во фронт-репо (это ловушка Claude Code в любом репо):**
- **НЕ** добавлять трейлер `Co-Authored-By: Claude …` и строку `Generated with Claude Code` в коммиты
  фронт-репо. Это **переопределяет дефолт Claude Code** (он добавляет такой трейлер сам).
- **НЕ** создавать ветки с префиксом `claude/` — только схема репо.
- Касается коммитов во **фронт-репо**; репозиторий самого marketplace/плагина — отдельно, там таких хуков нет.

**⚠️ В git-worktree локальный хук `commit-msg` может быть НЕ исполняемым** — тогда локально он **не**
отклонит нарушение (проверено на практике: коммит с трейлером прошёл без ошибки в worktree). CI всё равно
поймает на пуше/MR, но это уже поздно и требует правки истории. Раз локальный хук не страхует —
самоконтроль обязателен: держать правила в голове при каждом коммите и **явно проговаривать их в диспатче
субагентам-имплементерам** («no trailer, plain imperative, no conventional prefix») — они не читают этот
файл сами.

## Свежесть ветки (синхронизация с main)

Ветка всегда должна стоять на актуальном `origin/main` — иначе MR конфликтует, а QA-смоук на слоте
гоняет устаревший код.

**При старте (создание ветки/worktree) — от свежего main:**
```
git -C "⟪ADAPT: путь к фронт-репо⟫" fetch origin
git -C "⟪ADAPT: путь к фронт-репо⟫" worktree add -b ⟪ADAPT: префикс⟫-<slug> <path> origin/main
```

**Существующая ветка — проверить, не отстала ли от main:**
```
git fetch origin
git log --oneline <branch>..origin/main   # пусто = актуальна; есть коммиты = отстала
```
Отстала — **влить main в ветку** (`git merge origin/main`). Конфликты → **пауза**: показать список
конфликтных файлов человеку, разрулить продуктовые/логические конфликты с ним (тривиальные агент может
сам), затем завершить merge и прогнать green-sequence заново.

**Перед MR (и после правок по ревью):** повторить `fetch` + проверку; если `main` ушёл вперёд — влить
снова, чтобы MR был чистым против `main`.

**merge, а не rebase.** Ветки в этом флоу **пушатся** (MR + QA-смоук на слоте) → rebase переписал бы
запушенную историю. Всегда merge.

## MR

> **Прочитай этот раздел ПЕРЕД сборкой описания MR** (а не после того, как описание уже написано):
> draft-статус, заголовок и описание **на английском**, чек-лист `- [ ]`, предзаполненная markdown-ссылка.
>
> 🔧 **ADAPT — синтаксис prefill-URL зависит от git-хоста.** Пример ниже — форма `/-/merge_requests/new?merge_request[...]`;
> под свой git-хост/веб-платформу подставь его формат предзаполнения MR/PR (у разных платформ параметры и
> путь отличаются). Механика та же: собрать URL с URL-энкодингом переносов и отдать markdown-ссылкой.

**Грабля: push-опция `merge_request.description` не принимает переносы строк.**

`git push -o merge_request.description="Line1\nLine2"` — переносы не сработают, описание будет
одной строкой или сломается.

**Рабочий путь — URL создания MR** с предзаполненным описанием:
```
https://⟪ADAPT: ваш git-хост⟫/⟪ADAPT: группа/репо фронта⟫/-/merge_requests/new
  ?merge_request[source_branch]=<branch>
  &merge_request[target_branch]=main
  &merge_request[title]=Add+feature
  &merge_request[description]=Line1%0ALine2%0A%0A-%20[%20]%20Checklist+item
```
Переносы строк кодируются как `%0A`. Весь URL собирать заранее и **отдавать текстом в чат**
(буфер обмена перезатирается последующими командами — человек потеряет ссылку).

**Отдавать markdown-ссылкой с коротким понятным лейблом — не голым URL.** Предзаполненный URL длинный и
`%`-кодированный → в чате это нечитаемый «набор символов». Обернуть в markdown-ссылку — кликабельно и
читаемо: `[Создать draft-MR (title + description предзаполнены)](https://…)`. Голый URL в чат не вставлять.

**Порядок: ссылку создания MR отдавать только ПОСЛЕ успешного `push`** — не заранее и не «впрок».
До пуша ветки нет в origin, ссылка не открывает форму и **сбивает порядок шагов** (несколько ссылок
до пушей запутывают). Собрать URL можно заранее, но показать человеку — только после подтверждённого
пуша; если пушит человек — дождаться его «запушил», затем дать ссылку.

**Plain `git push`** без опций выдаёт в stdout строку вида «To create a merge request for
this branch, visit: https://...» — это работает и не требует опций. Автоматического MR не
создаёт, но даёт готовую ссылку.

**Требования к MR:**
- Статус: **draft** (префикс `Draft:` в заголовке).
- Заголовок и описание — **по-английски**, в стиле репо.
- Чек-лист — `- [ ]` (git-хост рендерит как интерактивные чекбоксы).
- Коммиты — английские (см. «Нейминг»).

**Где найти созданный MR** (частый вопрос человека — давать это пояснение **вместе со ссылкой создания MR**):
в UI git-хоста → список Merge Requests / Pull Requests → вкладка **Open** (черновик помечен `Draft`),
новый MR сверху списка. Прямая ссылка на список — `https://⟪ADAPT: ваш git-хост⟫/⟪ADAPT: группа/репо⟫/-/merge_requests`.

## i18n

> 🔧 **ADAPT — i18n целиком под вашу платформу.** Ниже — детальный референс **на примере** одного из
> возможных стеков (i18n-платформа/TMS с CLI-сборкой). **Перепиши весь раздел под свою i18n-платформу/TMS**
> (пути генерации, locale-коды, команды сборки, имя env-токена), а если **мультиязычности нет — удали
> раздел целиком**. Упоминания «i18n-платформа» / `i18n:build` / `<I18N_TOKEN>` / хостов ниже — иллюстрация
> конкретного стека, не требование. Переносим только общий принцип «новые ключи завести на платформе →
> собрать → закоммитить сгенерированное → push»; детали — нет.

Только **новые** ключи `FormattedMessage` требуют действий; переиспользуемые (уже есть в проде) — нет.

**Флоу для новых ключей:**
1. Проверить, нет ли уже в `⟪ADAPT: путь к сырым переводам, напр. backups/i18n/*.json⟫` (поиск по `id`).
2. Если ключа нет → **завести в i18n-платформе/TMS**.
3. После создания ключей в i18n-платформе → ⟪ADAPT: команда сборки i18n, напр. `i18n:build`⟫
   (генерирует файлы переводов в ту же ветку MR).
4. **Закоммитить сгенерированное** (`⟪ADAPT: путь generated⟫` + `⟪ADAPT: путь backups⟫`) в ветку MR и запушить.

**Проактивно, не реактивно:** как только в коде появился новый `id` — сразу пройди этот флоу (табличка
`id`/en/ru → завести в i18n-платформе → сборка → commit сгенерированного), **не дожидаясь** `MISSING_TRANSLATION`
в консоли или «набора символов» на стейдже.

**Сборка i18n требует токен i18n-платформы (⟪ADAPT: env-переменная токена вашей i18n-платформы⟫) — настраивается в `delivery_setup §2.0`** (одна команда). **Токен — не в чат**; если прислали — использовать и предложить перевыпустить.

**В git-worktree `.env` нет** (gitignored → не переносится из основного клона) → сборка i18n из worktree
падает `TransportError: Api token not provided`. Решение: `~/.zshenv` с ⟪ADAPT: env-переменная(ые) токена/хоста вашей i18n-платформы⟫ (настраивается в `delivery_setup §2.0`) — агент подхватывает во всех worktree автоматически.

### Как переводы попадают на стейдж (и диагностика «набора символов»)

- Стейдж/CI **не ходят в i18n-платформу в рантайме** — сборщик бандлит **закоммиченные** файлы. Источник
  истины для того, «что видит юзер» — **`⟪ADAPT: путь generated/<locale>.json⟫`** (минифицированный AST),
  **а не** сырой pull (`⟪ADAPT: backups⟫`, удобен для грепа по `id`).
- **«Набор символов» / минифицированный id на стейдже** = добавили новый `<FormattedMessage>`, но
  `generated` не перегенерили/не закоммитили → рантайм показывает minified-ключ. Фикс: пересобрать i18n +
  commit generated+backups + push (на ветку, ребейзнутую/смерженную со свежим main).
- **Английский на русском интерфейсе** = ключ есть в `generated/en`, но нет/устарел в `ru` →
  перегенерировать; либо стейдж задеплоен из коммита до добавления переводов → пересобрать слот.
- **`defaultMessage` в коде ≠ перевод из i18n-платформы.** `generated` берёт перевод из i18n-платформы (даже для EN);
  `defaultMessage` — лишь fallback. Их расхождение → i18n-warning `Default message is outdated`
  (не блокер). После правки текста в i18n-платформе — синхронизировать и `defaultMessage` в коде.
- **Текст на скрине стейджа — это перевод, а не `defaultMessage`.** Искать ключ по `id`, не по тексту;
  grep текста ловит артефакты (`generated`/`backups`) разных веток и обманывает.

### После сборки i18n — провалидировать pull (до коммита `generated`)

**Сначала — diff на уровне JSON-ключей, а не `git diff`.** `generated/*.json` однострочный минифицированный
(числовые хеш-id), поэтому текстовый `git diff` и grep по имени ключа бесполезны. Сверь **множества ключей**
HEAD vs рабочее дерево: для своей правки ожидаешь ровно **+N добавлено / 0 удалено / 0 изменено** в каждой
локали. Из корня фронт-репо (⟪ADAPT: путь в glob подставь свой⟫):
```bash
python3 - <<'PY'
import json, glob, subprocess
for f in sorted(glob.glob('⟪ADAPT: путь generated⟫/*.json')):
    new = set(json.load(open(f)))
    try: old = set(json.loads(subprocess.check_output(['git','show',f'HEAD:{f}'])))
    except Exception: old = set()
    print(f.split('/')[-1], ' +', sorted(new-old), ' -', sorted(old-new))
PY
```
Любое непустое `-` (удаление) или чужие ключи в `+` = что-то не так (ребейз / чужой pending-ключ из общего
pull i18n-платформы). **`generated/` содержит только активные локали** — orphan-бэкапы локалей, которых нет в
списке активных, сюда не входят, сверять их не нужно.

Авто-перевод в i18n-платформе может молча сломать строку:
- **ICU-плейсхолдеры = источнику.** `{days}` должен остаться `{days}` во всех локалях; авто-перевод иногда
  переводит и имя (`{days}` → вьетн. `{số ngày}`) → `MALFORMED_ARGUMENT`, и вся сборка (generate) падает.
  Чинится в i18n-платформе (плейсхолдер — буквально, переводится только текст вокруг).
- **Гомоглифы / смешанный скрипт** в кириллице: латинские `c/o/p/a/e/x` вместо кириллических (визуально
  неотличимо, ломает поиск/единообразие). Быстрый скан сырых переводов (node):
  ```js
  const o = require('⟪ADAPT: путь к backups/ru_RU.json⟫');
  for (const [k, v] of Object.entries(o)) {
    const s = (typeof v === 'string' ? v : JSON.stringify(v)).replace(/\{[^}]*\}/g, '');
    const stray = [...s].filter(c => /[A-Za-z]/.test(c) && !'ROIAPR'.includes(c));
    if (stray.length) console.log(k, JSON.stringify(s), stray);
  }
  ```

### Скоуп диффа: только свои ключи (сборка i18n тянет всю Спектру)

Сборка = extract → pull → generate, а pull тянет **весь** проект — в дифф лезут чужие ключи (других
команд/веток), даже если ты трогал один экран. **Скоупить до своих ключей:**
1. Сбросить `generated`+`backups` к `origin/main` (`git checkout origin/main -- ⟪ADAPT: пути generated и backups⟫`).
2. Добавить в `backups` **только свои** ключи (взять их из полного pull, не весь pull целиком).
3. Прогнать extract + generate **без** pull (иначе снова затянет всё).
4. Проверить, что дифф `generated`/`backups` = только свои ключи (см. python-сверку выше).

**Merge-конфликт `generated`** (хэшированный однострочный JSON) — отдельная грабля:
- **НЕ** мержить хэши руками — нечитаемо и хрупко.
- **НЕ** регенерить вслепую из своих `backups` — это может **потерять чужие ключи main** (проверено на
  практике: −4 ключа, ушедших вместе с чужими pending-записями).
- **Правильно:** взять `generated` из `origin/main` целиком, затем наложить **только свои ключи** (их
  хэш-записи — из своего регенерированного файла, не переписывая остальное), и сверить key-count + extract после.

**Формат i18n-платформы (импорт/экспорт) — пример:**
- Импорт/экспорт = **пер-локальный плоский JSON** `{ "string_id": "message" }` — **НЕ CSV**. Экспорт = zip `locales/<xx_XX>.json`.
- `generated/*` — скомпилированный ICU-AST с числовыми хешами; **руками не трогать**.
- Достаточно завести **en+ru**; остальные локали — авто-AI-перевод в i18n-платформе.
- FormatJS извлекает только **строковые литералы** `id`/`defaultMessage`; динамические (map/переменная) молча **не** извлекаются. Нельзя переиспользовать чужой ключ с другим `defaultMessage` (duplicate-id).

**Отдача новых ключей человеку — ФАЙЛАМИ JSON по локалям, не текстом/таблицей.** Сложить
`tasks/<слаг>/locales/en_US.json` + `ru_RU.json` (locale-code вашей платформы, ключи **отсортированы** по id, плоский JSON) —
прямой импорт в i18n-платформу + артефакт задачи. EN — source; **RU давать сразу** из согласованных на дизайне
формулировок (родная локаль команды), не гонять через переводчиков; остальные — авто-AI в i18n-платформе. НЕ
markdown-таблицей и НЕ CSV.

**Нет доступа к i18n-платформе** (есть не у всех) → не блокироваться:
- Отдать те же файлы `locales/en_US.json`+`ru_RU.json` и попросить завести через ⟪ADAPT: ваш чат-канал для запросов доступа⟫.
- После подтверждения — запустить сборку i18n.

**Бренд-литералы** (название продукта, тикеры валют и т.п.) — хардкодить, ключ не нужен.

**Однозначный английский для лейблов.** В текстах кнопок/шер-карточек избегать слов-омонимов глагол/сущ.:
`Share performance` авто-переводчик читает «Share» как существительное → «результат акций». Форсировать
глагол (`Share My …` / явный объект) или задать однозначный исходник; финально — зафиксировать RU-перевод
в i18n-платформе (машинный перевод коротких строк ненадёжен).

## Прототип

**Грабля: brainstorm-компаньон (preview) держит только последний HTML.**

Браузер-превью / brainstorm-режим отдаёт только **последнее сгенерированное состояние** HTML,
не статические ассеты. Если прототип ссылается на внешние изображения (`<img src="...">`) —
они не будут доступны.

**Решения:**
- Вшивать картинки как **data-URI** (`<img src="data:image/png;base64,...">`) прямо в HTML.
- Или поднять статик-сервер: `python3 -m http.server 8080` в папке прототипа.

**Требования к прототипу:**
- **Самодостаточный full HTML** (не фрагмент, не JSX) — иначе CSS фрейма ломает стили.
- Дизайн-система **только из фронт-репо**: ⟪ADAPT: путь к токенам ДС, напр.
  `design-system/ui-kit/src/tokens/`⟫ (CSS-переменные `var(--*)`). Форк/зеркало ДС вне фронт-репо — не использовать.
- После генерации — проверять через **скриншот** (preview), не на глаз.
- Сверять с **эталонными прод-скринами** (частая правка на ревью — не совпадает с реальным UI).

**Пространство выборов обязательно** (даже если «несколько вариантов» явно не просили — это дефолт):
**дефолт — ОДИН self-contained HTML с демо-панелью**, тумблеры которой переключают каждую открытую развилку
(состояния / плейсмент / подача / лейбл) живьём. **Отдельные full-page файлы — только для сравнения
«бок-о-бок»** (остальные развилки — тумблерами внутри). Никогда не отдавать один зафиксированный вариант;
подтверждённые идеи тоже прототипировать («подтверждение ≠ не нужен прототип»). Детали — `delivery_prototype` §5.

## Окружение

**MCP:**
MCP-серверы (figma, notion) могут отвалиться или потребовать переподключения в ходе сессии.
Симптом — ошибка «tool not found» или таймаут. Лечение: `/mcp` → найти сервер → Reconnect.

**Что делает агент сам, что — человек:**

| Задача | Кто |
|---|---|
| `npm ci`, `npm run build`, green-sequence | Агент |
| Дев-сервер, запуск тестов | Агент |
| VPN (VPN-клиент) | Человек |
| Логин на стейдже / в проде | Человек |
| Git-хост-креды (PAT, scope на push + API), интерактивный push (osxkeychain) | Человек / osxkeychain |
| Агентский push + авто-деплой стейджа через API | Агент: oauth2-URL + `GIT_API_TOKEN` из `~/.zshenv` |
| mkcert, `/etc/hosts` | Человек (sudo) |
| i18n-платформа (i18n-ключи) | Человек |

> 🔧 **ADAPT — имена scope зависят от git-хоста.** Ниже «scope на push» / «scope на API» — обобщение;
> подставь конкретные scope-имена своего git-хоста (напр. у GitLab это `write_repository` для push и `api`
> для API/авто-деплоя; у других платформ — свои repo/write-скоупы).

**osxkeychain** хранит HTTPS-кредентиалы для **интерактивного** `git push` человека — если логинился (PAT со scope на push + API), push проходит без запроса. Read-only scope → `git push` `403`; только push-scope → push ок, но API/авто-деплой `403` (поэтому нужен и API-scope). PAT заводится через **`delivery_setup §2.0`** — она кладёт его и в `~/.zshenv`, и в keychain одной командой.
**Агент** креды в helper **не вводит**; пушит через oauth2-URL (`https://oauth2:$GIT_API_TOKEN@…`), API git-хоста вызывает токеном `GIT_API_TOKEN` из `~/.zshenv`.

**Протух токен → push падает `HTTP Basic: Access denied`** (даже если раньше в той же сессии пушилось;
бывает read-only токен: `fetch` идёт, `push` — нет). **Человек:** osxkeychain для интерактивного push; если протух — одноразовый push с токеном в URL (в своём терминале):
```
git push "https://oauth2:<TOKEN>@⟪ADAPT: ваш git-хост⟫/⟪ADAPT: группа/репо⟫.git" <branch>
```
Токен, присланный в чат, использует **человек** и сразу **отзывает** (засветился).
**Агент:** токен в git-аутентификацию не вводит; пушит через `oauth2:$GIT_API_TOKEN` из `~/.zshenv` и ходит в API git-хоста тем же токеном — интерактивный keychain не нужен.
`oauth2:<token>` — надёжная форма PAT (username не важен); на интерактивном промпте Password = **токен**
(PAT git-хоста, не пароль аккаунта, при вводе не отображается). Стереть протухший кред:
`printf "host=⟪ADAPT: ваш git-хост⟫\nprotocol=https\n" | git credential-osxkeychain erase`.

**На macOS нет `timeout`** (и часто нет `gtimeout`) → `timeout 60 <cmd>` падает `command not found`. Для
тайм-аутов и долгих команд используй **встроенные средства Bash-инструмента** (`timeout`-параметр,
`run_in_background` + поллинг), а не shell-`timeout`.

**Если что-то зависло / не поднялось:** не молчать. Остановиться, показать человеку что
именно нужно сделать («что нажать / куда написать»), и продолжить после того как доступ получен.
