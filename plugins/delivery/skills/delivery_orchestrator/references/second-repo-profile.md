# Профиль ОПЦИОНАЛЬНОГО второго репо/стека

> 🔧 **ADAPT — второй репо/стек (шаблон-пример).** Этот файл — **необязательный пример второго профиля**
> на случай, если у команды ДВА фронт-репо с разными стеками, командами и i18n (в исходном проекте это был отдельный
> лендинг на Astro со своей i18n). По умолчанию флоу заточен под основной фронт-репо (`repo-conventions.md`).
> **Если второго репо нет — удалите этот файл и ссылки на него в orchestrator §3/§5.** Если есть — впишите
> сюда его специфику по образцу ниже (ниже показан заполненный пример из исходного проекта как ориентир формата).
>
> **Второй фронт-репо.** Для задач по этому репо действует **этот** профиль. Определи целевой репо на старте
> (см. orchestrator §3 / delivery_setup) и, если задача про него, читай этот файл вместо профиля основного репо.
>
> Репо: `⟪ADAPT: ваш git-хост⟫/⟪ADAPT: группа/путь второго репо⟫`.

---

## Что это

⟪ADAPT: тип и стек второго репо. Пример (лендинг исходного проекта): **Astro-лендинг** + React-острова
(`@astrojs/react`); **НЕ** Turborepo, **НЕ** тот же пакет, **НЕ** webpack; один пакет в корне репо;
контент — Astro content collections (`src/content/`: blog, academy, coins, reviews…) + страницы
`src/pages/[...locale]/*.astro`.⟫

## Node + bootstrap

Node ⟪ADAPT: версия Node второго репо, напр. 24 (CI-образ `node:24.5`)⟫. Установка зависимостей —
⟪ADAPT: команда bootstrap, напр. `npm run bootstrap` (= `npm ci`)⟫. ⟪ADAPT: нужны ли отдельный dev-bootstrap /
сертификаты — в примере лендинга исходного проекта **не нужны** (см. dev-сервер)⟫.

## Dev-сервер и проверка — ⟪ADAPT: есть ли стейдж-слоты по веткам⟫

> В примере лендинга исходного проекта **стейдж-слотов по веткам НЕТ** — проверка только локально. Впишите, как у вас.

- ⟪ADAPT: команда dev-сервера, напр. `npm run dev`⟫ → **⟪ADAPT: локальный URL, напр. `localhost:4321` (обычный HTTP)⟫**.
  ⟪ADAPT: в примере лендинга исходного проекта **не нужны** `/etc/hosts`, `mkcert`, VPN-к-localhost — проще, чем у основного
  app с HTTPS-хостом; VPN нужен только для git (GitLab) и, возможно, синка контента⟫.
- ⟪ADAPT: команды сборки/превью, напр. `npm run build` → `./dist/`; `npm run preview` — прод-сборка локально⟫.
- **Стейдж-слоты по веткам:** ⟪ADAPT: есть/нет⟫. Если слотов НЕТ — `delivery_qa_smoke` адаптируется: проверка
  не на слоте, а на **локальном dev-сервере** (`dev`) или прод-превью (`preview`). Прогон desktop + mobile,
  как обычно; «слот по ветке MR» тогда неприменим.

## Green-sequence (своя — из `package.json`)

> 🔧 **ADAPT — green-sequence второго репо.** Впишите точную последовательность из его `package.json`.
> Пример (лендинг исходного проекта на Astro):
> ```bash
> npm run ci:check     # astro check (диагностика Astro/типы в .astro)
> npm run ci:type      # tsc --noEmit
> npm run ci:eslint    # eslint src --quiet
> npm run ci:i18n      # node scripts/check-i18n.mjs — одинаковый набор ключей во всех локалях
> npm run ci:build     # astro build — финальная сборка
> ```
> В том примере **нет** `stylelint`, `ci:unused` (knip), `oxfmt` — их в лендинге нет, **не запускать**.
> Уберите/добавьте под свой стек. Всё — на требуемой версии Node.

## i18n — ⟪ADAPT: механизм i18n второго репо (в примере — свои JSON, НЕ i18n-платформа основного репо)⟫

> 🔧 **ADAPT — i18n второго репо.** Опишите его механизм локализации. Часто у второго репо **своя** i18n,
> отличная от i18n-платформы основного репо. Пример (лендинг исходного проекта):
> - Локали: **⟪ADAPT: путь и набор локалей, напр. `src/i18n/{en,ru,de,nl,pt}.json` — 5 языков⟫**, плоские
>   словари `ключ → текст`; `en` — reference; `defaultLocale` / `locales` — в `src/constants/i18n`.
> - **Правка текста = править ключ во ВСЕХ локалях.** `npm run ci:i18n` (`check-i18n.mjs`) падает, если в
>   какой-то локали ключ отсутствует или лишний (сверяет наборы ключей с reference-локалью).
> - **Нет** i18n-платформы (у основного репо она есть) / `generated` / `backups` / ⟪ADAPT: env-переменная токена вашей i18n-платформы⟫ / минификации — поэтому i18n-приёмы из
>   `repo-conventions.md` § i18n (i18n-платформа, ⟪ADAPT: команда вашей i18n-платформы⟫, «набор символов») **сюда не применимы**.
> - Контентные страницы (термсы, политики и т.п.) — `src/pages/[...locale]/*.astro`; текст берётся оттуда
>   и из локальных JSON.
> - Пример правки (термсы, замержен): страница `terms-of-use.astro` + ключи в JSON по всем локалям.
>
> Если у вашего второго репо i18n **такая же**, как у основного (та же i18n-платформа) — уберите этот пример и сошлитесь
> на `repo-conventions.md` § i18n.

## Конвенции кода

> 🔧 **ADAPT — конвенции кода второго репо.** Пример (лендинг исходного проекта):
> - Astro-компоненты (`.astro`) + React-острова (`.tsx`). **Нет `CLAUDE.md`** — правила выводятся из
>   `eslint.config.mjs` (perfectionist sort-imports, react-hooks, astro, import) + `.prettierrc`.
> - **НЕ** применять правила основного app-репо (`FC`+`memo`+named export, `Decimal.fromNumber`,
>   barrel/`integration`-импорты, `@jest/globals`) — они про главный пакет основного репо, не про лендинг.
> - Без поясняющих комментов в коде (общий принцип). Изображения/секвенции — `public/images/`, оптимизация —
>   `scripts/optimize-images.mjs`.
>
> Впишите конвенции своего второго репо; общий принцип «правила основного репо сюда не переносить
> автоматически, у второго стека они свои» сохраняется.

## MR / деплой — как везде

- Worktree от свежего `origin/main`; ветка по конвенции репо (⟪ADAPT: типы/формат веток второго репо⟫);
  коммиты — англ. императивы (`Add …` / `Update …`), без conventional-prefix.
- Draft-MR через URL (`/-/merge_requests/new?...`, переносы `%0A`) — **markdown-ссылкой с лейблом, не голым URL** (`repo-conventions.md` §MR); **ссылку давать после успешного `push`**;
  пуш — на выбор (агент / человек, спросить если не ясно). **Мерж в `main` и деплой — фронтендер/QA**, агент
  не мержит. Деплой — ⟪ADAPT: как деплоится второй репо, напр. GitLab CI (argocd/kaniko/cloudflare)⟫.

## Что из треков применимо

Треки **A / B / C**, git-worktree, лог заметок, ревью прототипа дизайнером, draft-MR, push-choice,
MR-link-after-push, хэндофф QA — **применимы как есть**. Отличия только в 4 точках: **green-sequence**
(команды выше), **i18n** (⟪ADAPT: механизм i18n второго репо, в примере — JSON по локалям, не i18n-платформа основного репо⟫),
**проверка** (dev-сервер ⟪ADAPT: локальный URL⟫ / `preview`, ⟪ADAPT: есть ли слоты⟫), **конвенции**
(⟪ADAPT: стек второго репо, в примере — Astro/eslint, не app-правила⟫). Всё, что специфично для основного
репо (i18n-платформа / стейдж-слоты / главный пакет / его HTTPS-хост), — игнорировать для этого репо, если у него иначе.
