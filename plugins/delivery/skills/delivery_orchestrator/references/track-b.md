# Трек B — фикс бага / правка существующего поведения (без дизайна)

Лёгкий флоу: баг → локализация → фикс → проверка + QA-смоук → draft-MR.
Дизайн и прототипирование **пропускаются** (кроме одного условного шага).

Сквозь весь трек: **лог заметок** (затыки, тупики — `notes.md` в папке задачи; раскладка — `workspace-layout.md`), общение на языке человека, guardrails (см. `repo-conventions.md`).

---

## Стадия 0 — Преднастройка (`delivery_setup`)

**Человек:** подтверждает, что доступы и окружение в порядке (или проходит настройку).

**Агент:**
- Вызывает скилл `delivery_setup` (один раз за сессию/машину).
- `delivery_setup` проверяет: Node ⟪ADAPT: требуемая версия, напр. ≥24⟫, git, доступ к фронт-репо (≥ Developer), дев-VPN (⟪ADAPT: ваш VPN-клиент⟫), нужные MCP (figma/notion при необходимости; **аналитический MCP по умолчанию не предлагать**).
- Сначала читает `CLAUDE.md` фронт-репо — это первоисточник правил кода на протяжении всего трека.
- При отсутствии доступа — **пауза + понятная плашка** «что сделать / куда писать» (⟪ADAPT: ваш чат-канал для запросов доступа⟫); не блокироваться молча. Детали — `access-gates.md`.

**Критерий готовности:** Node нужной версии ок, репо клонирован, `CLAUDE.md` прочитан, дев-VPN поднят (если нужен), нужные MCP отвечают.

## Стадия 1 — Захват бага

**Человек:** описывает дефект по структуре ниже; прикладывает скрины/видео и ссылку на карточку/репорт.

**Агент:** задаёт уточняющие вопросы, если описание неполное, и фиксирует в лог. Нужная информация:

- **Что не так** — краткая формулировка проблемы.
- **Где** — поверхность/экран/компонент (⟪ADAPT: перечислите основные поверхности/разделы вашего продукта — конкретная вкладка или поток⟫).
- **Шаги воспроизведения** — пронумерованный список действий от входа в приложение до момента ошибки.
- **Ожидаемое поведение** — что должно было произойти.
- **Фактическое поведение** — что происходит вместо этого.
- **Скриншоты / видео** — визуальные доказательства.
- **Ссылка на карточку / репорт** — Notion, GitLab issue или Slack-тред с исходным сообщением о баге.

> **Входные артефакты (видео/скрины/логи) часто недоступны агенту.** Файлы в `~/Downloads` блокирует
> macOS TCC (даже `cp`/`ffprobe`; `ls`/`stat` видят метаданные, содержимое — нет). Попросить человека
> **перенести файл в рабочую папку** проекта. Видео — разложить на кадры через `ffmpeg`
> (`ffmpeg -i <file> -vf fps=3 frames/f_%03d.jpg`) и смотреть выборку + **прицельно момент бага** —
> надёжнее, чем гадать по описанию.

## Стадия 2 — Воспроизведение и локализация причины

**Человек:** при необходимости помогает подтвердить гипотезу (воспроизвести вживую, уточнить среду/аккаунт).

**Агент:**
- Запускает `superpowers:systematic-debugging` для структурированного поиска корня проблемы.
- Читает код фронт-репо в нужной области (компонент, хук, утилита, стейт). Широкий поиск — через
  `Explore`-субагента, но **его гипотезу перепроверяет чтением реализации целиком** (не доверять пересказу).
- **«Когда-то работало» → ищет регрессию по git-истории** (`git log -S<маркер>` / история файла): какой
  коммит изменил поведение — часто прямо указывает на причину.
- Воспроизводит баг локально (dev-сервер) если возможно без sudo/интерактива. Если улика — **видео/скрин**,
  см. § «Входные артефакты» в стадии 1 (раскадровка). Чётко разделяет **«причина по коду»** (гипотеза из
  чтения) и **«подтверждение воспроизведением»** (увидел сам) — не выдавать одно за другое.
- Формулирует **гипотезу о корневой причине** и при неоднозначности **подтверждает с человеком**, прежде чем идти в фикс.
- **Если фикс зависит от продуктовой развилки** (как должно вести себя поведение — напр. «всегда вниз»
  vs «запоминать позицию») — это **решение человека, не выводится из кода**: согласовать ожидаемое поведение до фикса.

**Дизайн и прототипирование — пропускаем полностью.**

## Стадия 3 — Дизайнер (только при видимом изменении вёрстки/UX)

**Условие:** этот шаг активируется **только если фикс меняет визуальный облик** (перестановка элементов, исправление layout, изменение отступов/цветов, изменение UX-паттерна взаимодействия). Если изменение чисто логическое или технически не видно пользователю — **шаг пропускаем**.

**Человек:** смотрит before/after и подтверждает направление.

**Агент:**
- Готовит **короткий before/after** (скриншот или словесное описание изменения) и отправляет в Slack дизайнеру.
- Ждёт подтверждения или правок до перехода к фиксу.

## Стадия 4 — Фикс

**Человек:** если нужны sudo/VPN/логин/GitLab-креды (PAT со scope `api` + `write_repository`) — делает сам.

**Агент:**
- Создаёт **git-worktree** фронт-репо (скилл `superpowers:using-git-worktrees`): перед созданием —
  **`git fetch origin`**, ветку делать **от свежего `origin/main`** (`git worktree add -b bugfix-<slug> <path> origin/main`);
  ветка — **`bugfix-[TICKET-NNN-]<kebab>`** ⟪ADAPT: тип-префикс фикса + тикет-префикс по вашей конвенции (напр. `bugfix`/`fix`); формат — `repo-conventions.md` § Нейминг⟫. Существующую ветку — сверить с `origin/main` и влить
  main, если отстала (merge, не rebase; конфликты → пауза). Детали — `repo-conventions.md` § «Свежесть ветки».
- Реализует минимально необходимое исправление.
- **По возможности — регресс-тест:** пишет тест, который сначала падает (баг воспроизводится), затем зеленеет после фикса. Малые и тривиальные фиксы (1–2 строки, очевидная причина) — можно инлайн без полного subagent-цикла.
- Соблюдает правила репо (`repo-conventions.md`) — ⟪ADAPT: жёсткие стек-конвенции вашего фронта (в исходном проекте: `FC`+`memo`+named export, `Decimal.fromNumber`, `import type` отдельно, `FormattedMessage` для строк с хардкодом бренд-литералов); держите детали в `repo-conventions.md`⟫; без поясняющих комментов в коде.
- Коммит: простой англ. императив **`Fix …`** (без conventional-prefix, без сквоша).
- Прогоняет **green-sequence на Node нужной версии** (⟪ADAPT: точные команды из `repo-conventions.md`; в исходном проекте: из пакета `ci:type`/`ci:eslint`/`ci:stylelint`/`ci:test`/`i18n:extract`; из корня `ci:unused` + `oxfmt:check`⟫). Детали — `repo-conventions.md`.

## Стадия 5 — Проверка + QA-смоук

**Человек:** при необходимости предоставляет тестовый аккаунт/данные для stage; подтверждает, что баг устранён.

**Агент:**
- **Локальная проверка на dev-сервере — обязательный гейт ДО push, не опция:** ⟪ADAPT: команда старта dev-сервера, напр. `npm start`⟫ →
  ⟪ADAPT: локальный URL dev-сервера, напр. `https://local.<...>:8443`⟫, воспроизвести сценарий из стадии 1 (ожидаемое vs фактическое),
  скриншот **before/after**, green-sequence на нужной версии Node чисто. Выбор API (stage/prod) — предложить человеку.
  **Агент может прокликать сам** (Playwright + ⟪ADAPT: способ логина, напр. тестовый аккаунт/кошелёк⟫ + `ignoreHTTPSErrors`, обходит mkcert) — нужен
  лишь минимум данных (сценарий/состояние); предлагай это проактивно, а не сразу к push.
- **Стейдж-смоук (`delivery_qa_smoke`) — ПОСЛЕ push** (слот или временный деплой требует запушенной ветки —
  см. стадию 6): баг устранён (сценарий стадии 1 не воспроизводится), нет регресса рядом (соседние
  кнопки/состояния/экраны на той же поверхности), desktop + mobile, инкогнито.
- При обнаружении новых проблем — правки **в ту же ветку**, повторный смоук.
- Результат передаётся QA в Slack (хэндофф).

## Стадия 6 — MR (draft)

**Человек:** переходит по ссылке, убеждается, что описание заполнено корректно, и отправляет на ревью фронтендеру в GitLab; после апрува — **мерж и деплой делают QA или фронтендер** (не вы).

**Агент:**
- **Сначала прочитать `repo-conventions.md` § MR** — формат описания (draft, EN, чек-лист `- [ ]`,
  markdown-ссылка) учесть ДО написания, не подгонять после.
- **Перед пушем — синхронизация с main:** `git fetch origin`; если ветка отстала — `git merge origin/main`
  (конфликты → пауза) и пере-прогнать green-sequence.
- Пушит ветку `bugfix-*` в origin — **или отдаёт команды человеку, если он предпочитает пушить сам** (если из контекста не ясно — коротко спросить).
- Формирует **ссылку создания MR** с предзаполненным описанием (URL вида `/-/merge_requests/new?...`, переносы — `%0A`; push-опция `merge_request.description` переносы не принимает). **Ссылку отдаёт текстом в чат markdown-ссылкой с коротким лейблом** — не голым `%`-кодированным URL (он нечитаем): `[Создать draft-MR (...)](https://…)`.
- Заголовок MR: **`Fix …`** (кратко, что именно исправлено).
- Статус: **draft**.
- Описание MR на **английском** в стиле репо; содержит:
  - краткое описание проблемы и что изменилось;
  - **before/after скриншоты** (обязательно — ключевой элемент ревью);
  - ссылку на карточку/репорт из стадии 1;
  - **чек-лист — функциональный импакт-анализ, НЕ технические галочки.** Только пользовательские проверки:
    (1) **что правили** — что именно должно работать после фикса; (2) **что могло быть задето рядом** —
    смежные экраны/состояния на той же поверхности, общие компоненты. Каждый пункт — конкретный сценарий,
    который ревьюер/QA прокликает. **Не** включать `tested locally`, `green-sequence passed`, имена файлов
    и прочую техничку — это шум для ревью. Пример (обобщённый — фикс скролла в списке/чате):
    - `- [ ] Switching between items keeps scroll position stable (no jump)`
    - `- [ ] Opening the view lands on the latest content`
    - `- [ ] Scrolling history up still loads older items`
    - `- [ ] Adjacent surfaces on the same screen work as before`

**Стиль fix-MR:** ориентир — короткий точный замерженный фикс-MR из вашего репо (⟪ADAPT: ссылка на образцовый fix-MR, если есть⟫). Короткое точное название, before/after, минимальный diff, чек-лист.

**Перед отдачей ссылки — СТОП-самопроверка** (`repo-conventions.md` § MR): `[ ]` push сделан; `[ ]` заголовок `Fix …` на EN; `[ ]` описание на EN; `[ ]` before/after есть; `[ ]` чек-лист `- [ ]` конкретных проверок; `[ ]` ссылка markdown, не голый URL. Хоть один не выполнен — сначала исправить.

**Где найти MR после создания** (частый вопрос): GitLab → слева **Code → Merge requests** → вкладка **Open** (черновик с меткой `Draft`), твой MR сверху списка; прямая ссылка — `https://⟪ADAPT: ваш git-хост⟫/⟪ADAPT: группа/путь фронт-репо⟫/-/merge_requests`.

**Объясни человеку, что дальше** (простым языком): «создашь MR → пайплайн (тесты + сборка) → я подниму временный стейдж и пришлю ссылку → смоук (баг ушёл, регресса нет) → возвращаешь фронтендеру на ревью; мерж/деплой делают QA/фронтендер после апрува, **не ты**». **Предложи:** «фикс на стейдже могу проверить сам — скажи сценарий».

**После апрува фронтендера** → флоу завершён со стороны трека. Мерж/деплой — QA/фронтендер. Агент **никогда не мержит и не деплоит**.
