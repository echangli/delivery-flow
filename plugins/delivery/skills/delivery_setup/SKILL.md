# delivery_setup — шаг 0: окружение и доступы

Ты проверяешь машину человека **один раз в начале флоу доставки**. Цель — убедиться, что у него всё
нужное есть, и сразу объяснить, как разблокировать то, чего нет. После этого шага флоу может идти
дальше без сюрпризов.

---

## 1. Когда запускать / когда пропустить

**Запускать:** в самом начале Трека A, B или C — до brainstorming, до открытия кода, до любого
`git`-действия. **Проверку актуальности плагина (§2.7) делай при каждом старте флоу** — не только при
первом знакомстве человека с плагином, а в **каждой новой сессии** (плагин мог устареть с прошлого раза).

**Пропустить:**
- Если ты уже прогонял `delivery_setup` **в текущей сессии** — не перепроверяй снова (включая версию
  плагина), просто продолжай флоу. В **новой** сессии — проверь заново.
- Это **не онбординг с нуля**: Claude Code у человека уже установлен. Мы проверяем инструменты и
  доступы, а не устанавливаем рабочее окружение целиком.

---

## 2. Порядок проверок

Выполняй последовательно. Если что-то не так — сразу выдавай инструкцию (см. раздел 3), не копи
проблемы до конца.

### 2.0 Токены — проверка (заводятся одной командой в README §3 шаг 2)

Секреты флоу заводятся **один раз, единой командой ДО установки** — для нового человека это **README плагина, §3 шаг 2** (тот же шаг, что сохраняет логин рабочего git-хоста: одна команда кладёт PAT и в Keychain, и в `~/.zshenv`). Поэтому к началу флоу токен обычно уже на месте — здесь **только проверь**:
```bash
echo "GIT_API_TOKEN: ${GIT_API_TOKEN:+ok}"
printf "protocol=https\nhost=⟪ADAPT: ваш git-хост⟫\n" | git credential fill 2>/dev/null | grep -q '^password=' && echo "keychain: ok"
```
> 🔧 **ADAPT — проверка токена i18n-платформы (опционально).** Есть i18n-платформа/TMS со своим токеном — добавь в проверку строку `echo "i18n-token: ${⟪ADAPT: env-переменная токена вашей i18n-платформы⟫:+ok}"`. Нет мультиязычности — пропусти.

Всё `ok` → идём дальше, токен по флоу больше не трогаем. Чего-то нет (новая машина / ставили без README / токен протух) → дай человеку **ту же команду** (подставив токен; **в чат не слать**):
```bash
GITPAT="сюда-вставь-токен-git-хоста"      # ← PAT рабочего git-хоста (scope на push + API)
{ echo "export GIT_API_TOKEN=\"$GITPAT\"" ; } >> ~/.zshenv
printf 'protocol=https\nhost=⟪ADAPT: ваш git-хост⟫\nusername=oauth2\npassword=%s\n' "$GITPAT" | git credential approve
```
> 🔧 **ADAPT — токен i18n-платформы (опционально).** Если у вас есть i18n-платформа/TMS со своим CLI-токеном (роль `⟪ADAPT: роль/scope токена⟫`) — добавь его в ту же команду: `echo "export ⟪ADAPT: env-переменная токена вашей i18n-платформы⟫=\"...\"" >> ~/.zshenv` (+ хост, если нужен). Нет мультиязычности — пропусти.

- **PAT рабочего git-хоста** — scope, достаточный на push + API/авто-деплой (⟪ADAPT: конкретные scope-имена вашего git-хоста, напр. у GitLab `api`+`write_repository`⟫).
- Кладёт PAT и в **`~/.zshenv`** (`GIT_API_TOKEN` — агент: `curl`/API/push-oauth2/деплой), и в **osxkeychain** (`git credential approve` — git clone/fetch/autoUpdate/интерактивный push). **`~/.zshenv`, НЕ `~/.zshrc`** (агентский non-interactive Bash читает только zshenv). git сам env-токен не читает → keychain нужен отдельно, но заполняется той же командой.
- Протух (`401`/`403`) → перевыпустить токен, повторить команду. Дальше шаги 2.1–2.8 и весь флоу токены **не просят** — только эта проверка.

### 2.1 Определить целевой репо и прочитать его правила

> 🔧 **ADAPT — второй сайт/стек (опционально).** Частый случай — два фронта: основное приложение (репо приложения) и отдельный маркетинговый лендинг (напр. на другом фреймворке, со своими командами, i18n-JSON по нескольким локалям без i18n-платформы основного репо, своим dev-сервером без certs/hosts/mkcert и **без** стейдж-слотов). Если у вас тоже есть второй репо/стек с иным пайплайном — опиши его профиль в отдельном файле `delivery_orchestrator/references/second-repo-profile.md` и добавь сюда развилку «какой сайт правим». Если фронт один — убери эту развилку и оставь только основной репо ниже.

**Сначала пойми, какой репо правим** (если их несколько) — от этого зависят команды, i18n и проверка.
Для основного фронта `CLAUDE.md` фронт-репо это **единственный источник правды по правилам кода/git** (правила,
green-sequence, нейминг, i18n). **Читай его (и нужные `docs/*`) первым делом и сверяйся на реализации/перед MR —
не полагайся на память и не считай выжимку в `repo-conventions.md` заменой** (она лишь навигация + грабли,
которых в доках нет; правила во фронте **меняются**).

```bash
# Путь к фронт-репо: спроси человека или возьми из FRONTEND_REPO в .env рабочей папки.
FE="⟪ADAPT: путь к вашему фронт-репо⟫"
cat "$FE/CLAUDE.md" "$FE/⟪ADAPT: путь к главному под-CLAUDE.md, напр. apps/<app>/CLAUDE.md⟫"   # главный источник правил
# По задаче — релевантные доки: ⟪ADAPT: релевантные docs/* вашего репо, напр. docs/styling.md · docs/i18n.md · docs/boundaries.md ·
# docs/for-backend-developers.md (для не-фронтендеров, правящих через AI).⟫
```

**Авто-флаг дрейфа правил** (фронтендер мог подлить новые правила/хуки — лови это, а не узнавай по отклонённому пушу):
```bash
git -C "$FE" fetch origin -q
git -C "$FE" log --oneline -12 origin/main -- CLAUDE.md ⟪ADAPT: пути CLAUDE.md/docs/git-lint/hooks вашего репо, напр. apps/<app>/CLAUDE.md apps/<app>/docs ci/git-lint .githooks⟫
```
Свежие коммиты по этим файлам → прочитай изменённые секции в актуальных доках и следуй им. Если правка задевает
то, что зашито в плагин (git-конвенция, процесс) и **расходится** с ним — следуй фронту и **флагни владельцу плагина**,
что скилл стоит подправить.

Сохрани ключевые правила в своём контексте. Не перечитывай эти файлы повторно в ходе флоу —
береги кэш.

> **Для непостоянных фронтендеров (бэк/QA/дизайн), правящих фронт через AI** — если во фронт-репо есть
> док вроде `⟪ADAPT: путь к гайду для не-фронтендеров, если есть, напр. apps/<app>/docs/for-backend-developers.md⟫`
> (мост понятий, что ловит автоматика и какие **red flags в AI-дифе** читать глазами) — полезно прочитать
> в начале, это аудитория плагина.

### 2.2 Node ⟪ADAPT: минимальная версия Node вашего репо, напр. ≥ 24⟫

```bash
node -v
```

**Ожидаемый результат:** ⟪ADAPT: ожидаемая версия, напр. `v24.x.x`⟫ или выше.

**Почему это важно.** Фронт-репо требует ⟪ADAPT: версию Node — `.nvmrc` + `engines node`, напр. Node ≥24 (`.nvmrc` 24.5.0, `engines node>=24`)⟫. На более
старом Node `eslint`, `stylelint`, `knip` и `oxfmt` **виснут или падают** — у них нет
`darwin-arm64`-бинарей для старых версий (баг npm optional-deps). Запускать green-sequence на
неправильном Node бессмысленно.

**Если Node ниже требуемого или команда не найдена:**

Инструкция для человека:
```
brew install node@⟪ADAPT: мажор Node, напр. 24⟫
# После установки keg-only brew выведет путь; добавь в PATH (пример):
export PATH="/opt/homebrew/opt/node@⟪ADAPT: мажор Node, напр. 24⟫/bin:$PATH"
# Проверь:
node -v   # должно быть ⟪ADAPT: ожидаемая версия, напр. v24.x.x⟫
```

После переключения на нужный Node нужен **чистый bootstrap** — нативные бинари линкуются при
`npm ci`, поэтому старый `node_modules` не подойдёт:
```bash
cd "⟪ADAPT: путь к вашему фронт-репо⟫"
rm -rf node_modules ⟪ADAPT: доп. node_modules воркспейсов, напр. apps/<app>/node_modules⟫   # или npx lerna clean -y
npm run ⟪ADAPT: команда bootstrap dev, напр. bootstrap:dev⟫
```
Без этого green-sequence будет висеть даже на правильном Node.

> **Intel-мак (`darwin-x64`)?** Green-sequence на целевом Node пройдёт, но сборка i18n (только для
> i18n-задач) может упасть `Native … binding is unavailable` — если сборщик переводов использует нативный
> аддон только под `arm64` (напр. FormatJS). Ожидаемо; обход (pure-JS-регенерация `generated` / довериться
> CI) — в `delivery_orchestrator/references/repo-conventions.md § i18n`.

### 2.3 git + credential.helper

```bash
git --version
git config --global credential.helper
```
**Ожидаемый результат:** git установлен; `credential.helper` = `osxkeychain` (или аналог). git нет → `brew install git`; helper пустой → `git config --global credential.helper osxkeychain`.

**Токены уже заведены в §2.0** (PAT рабочего git-хоста в `~/.zshenv` + osxkeychain одной командой; scope на push + API). `credential.helper` нужен, чтобы **git по HTTPS** (clone/fetch/**autoUpdate**/интерактивный push человека) брал кред из keychain. **Агент** же пушит `oauth2`-URL из `$GIT_API_TOKEN` и ходит в API тем же токеном — helper ему не обязателен. Протух keychain-кред у человека → `printf "host=⟪ADAPT: ваш git-хост⟫\nprotocol=https\n" | git credential-osxkeychain erase`, затем повторить команду из §2.0.

**Автор коммита — корпоративная почта.** Серверный `pre-receive`-хук git-хоста часто отклоняет push с личной почты в авторе
(`Author '<...>@gmail.com' is not a member of team`). Сверь **до первого коммита**:
```bash
git -C "⟪ADAPT: путь к вашему фронт-репо⟫" config user.email
```
Не корп-домен (⟪ADAPT: ваш корп-домен почты⟫), а личная (gmail и т.п.) → поставить корпоративную **локально для репо**
(global не трогаем):
```bash
git -C "⟪ADAPT: путь к вашему фронт-репо⟫" config --local user.email "<логин>@⟪ADAPT: ваш корп-домен почты⟫"
```
Если коммит уже сделан с личной почтой — после смены `git commit --amend --reset-author`, затем push.

### 2.4 Доступ к фронт-репо

Проверь, что репо склонировано и человек может читать и пушить:

```bash
cd "⟪ADAPT: путь к вашему фронт-репо⟫"
git remote -v          # должен быть remote на ⟪ADAPT: ваш git-хост⟫/⟪ADAPT: группа/путь репо⟫
git fetch --dry-run    # подтверждает сетевой доступ
```

**Что нужно:** права **≥ Developer** (push ветки + открытие MR). Без этих прав `git push` упадёт.

Если репо недоступно или не склонировано — **пауза** (см. раздел 3, гейт «Доступ к фронт-репо»).

### 2.5 dev-VPN (VPN-клиент)

> 🔧 **ADAPT — VPN опционален.** Нужен, только если рабочий git-хост/стейдж команды за корпоративным VPN.
> Если репо и стейдж публично доступны — удали этот подраздел и пункты про VPN ниже.

VPN нужен для доступа к рабочему git-хосту и к stage-окружению. Без VPN `git fetch`/`git push` будут падать
с таймаутом.

Проверить косвенно:
```bash
# Если git fetch выше прошёл — VPN скорее всего поднят.
# Иначе попроси человека проверить статус VPN-клиента в трее.
```

VPN-клиент должен быть у всех (поставляется командой). Если нет или не запущен — **пауза**
(см. раздел 3, гейт «dev-VPN»).

**Локальный dev-сервер на stage-API без VPN = белый экран и не залогиниться** (`/api`→⟪ADAPT: stage API-хост⟫,
`/socket`→⟪ADAPT: stage WS-хост⟫, соц-логин через VPN) — это частый ложный «бека нет?». Перед тем как давать человеку
URL дев-сервера, быстрый health-check бэка (не таймаут = VPN поднят):
```bash
curl -sk -m8 -o /dev/null -w '%{http_code}\n' https://⟪ADAPT: stage API-хост для health-check⟫
```
Таймаут / `000` → VPN не поднят (или перейти на prod-API `npm run ⟪ADAPT: команда старта на prod-API, напр. start:api:prod⟫`). Детали — `repo-conventions.md` § Dev-сервер.

### 2.6 MCP: figma и notion

**Проверяет агент сам — через Bash, без интерактивной панели `/mcp`** (её агент открыть не может):
```bash
claude mcp list           # статус всех настроенных MCP-серверов
```
- `notion`/`figma` присутствуют и `✓ Connected` → ок.
- Отсутствуют или `✘ Failed to connect` → диагностируй и чини сам (ниже). Человека дёргать **только**
  если нужен его интерактивный OAuth-логин.

**Условный гейт — не блокируй флоу на MCP, который этой задаче не нужен:**
- `notion` нужен, только если есть карточка Notion для чтения (обычно Track A). Нет карточки или её
  содержимое уже дали в чат — notion не обязателен.
- `figma` нужен только при макетах в Figma.
- **Фолбэк всегда равноправен:** попросить человека вставить содержимое карточки/макета прямо в чат —
  и идти дальше. Не застревать на MCP.

**Самодиагностика и автофикс битого конфига** (частый кейс: сервер добавлен как `stdio` с URL вместо команды):
```bash
claude mcp get notion     # посмотри Type и Command
```
- Если `Type: stdio` и `Command:` — это URL (`https://…`) → конфиг сломан (клиент пытается **запустить
  URL как программу** → `ENOENT` / `Failed to connect`). Чинит агент:
  ```bash
  claude mcp remove notion -s user
  claude mcp add --transport http --scope user notion https://mcp.notion.com/mcp
  claude mcp get notion     # ожидаем Type: http, ✔ Connected
  ```
- **Для hosted-MCP всегда указывай `--transport http`** — без флага дефолт `stdio`, и URL уходит в команду.

**OAuth-логин (только это требует человека).** Если сервер сконфигурён верно, но не авторизован:
- ⚠️ **Из десктоп-приложения Claude Code OAuth часто не доходит до конца.** Подключать через
  **терминальный `claude`** (`/mcp → <server> → Authenticate`), либо через раздел **Connectors**
  (если у человека коннекторная модель — как claude.ai Google Drive).
- Токен хранится на уровне пользователя (`~/.claude`). Тулзы MCP могут подхватиться в текущую сессию
  не сразу — **возможно нужен рестарт** десктоп-сессии.

**Не бойся рестарта.** Артефакты флоу (лог заметок, спека, прототипы, worktree) живут **на диске** —
рестарт теряет только историю чата, не работу. Возобновление — заново `/deliver`, агент читает лог.
Поэтому MCP-чек делай **рано** (до тяжёлого диалога), чтобы рестарт ничего не стоил.

> **Аналитический MCP (⟪ADAPT: ваша аналитическая платформа⟫) — не предлагать по умолчанию.** Он нужен только для
> аналитических задач; в обычном флоу доставки он не используется.

### 2.7 Подключение и автообновление плагина delivery

Плагин ставится из git-marketplace `delivery-toolkit`; версия = git SHA (`plugin.json` без `version`). Для
приватного marketplace **автообновление по умолчанию выключено**.

- **При КАЖДОМ старте флоу** (не только при первом знакомстве; в каждой новой сессии) — быстро сверь,
  что marketplace на `main` (п.1) и версия не устарела (п.3). В рамках одной сессии повторно не проверяй.
- **One-time** (первый заход человека / автообновление не настроено) — помоги включить `autoUpdate`
  (п.2). Когда он включён, свежий `main` подтягивается при старте сам → ручная сверка версии не нужна,
  достаточно подтвердить, что `autoUpdate` на месте.

**1. Ветка marketplace** (частая поломка):
```bash
python3 -c "import json,os;print(json.load(open(os.path.expanduser('~/.claude/plugins/known_marketplaces.json'))).get('delivery-toolkit',{}).get('source',{}).get('ref','(none = default main)'))"
```
Должно быть `main` или `(none = default main)`. Если ref = feature-ветка (напр. `add-delivery-plugin`)
— её удаляют после мержа, и `/plugin marketplace update` падает с `Remote branch ... not found`.
**Recovery** (терминальный `claude`): `/plugin marketplace remove delivery-toolkit` →
`/plugin marketplace add delivery-toolkit` (возьмёт `main`) →
`/plugin install delivery@delivery-toolkit` (scope user) → `/reload-plugins`.

**2. Автообновление включено?** (one-time, «раз и навсегда»):
```bash
python3 -c "import json,os;print('autoUpdate:', json.load(open(os.path.expanduser('~/.claude/plugins/known_marketplaces.json'))).get('delivery-toolkit',{}).get('autoUpdate', False))"
```
`False` → предложи человеку включить **одним кликом** в UI (терминальный `claude`): `/plugin` → вкладка
**Marketplaces** → `delivery-toolkit` → **Enable auto-update**. Claude Code сам запишет `autoUpdate: true` в
`known_marketplaces.json` — править файлы не нужно. Тогда свежий `main` приходит при каждом старте; по
нотификации — `/reload-plugins`. Детали — README плагина, раздел «Автообновление».

**Нужен сохранённый логин к репо marketplace:** GitHub-репо marketplace приватный, при старте Claude Code
не может спросить креды — автоапдейт сработает, только если логин к этому репо уже в **osxkeychain**
(git-fetch при старте читает кред из keychain, env-переменную токена он **не** использует). Нет апдейтов
при старте → keychain пуст для хоста marketplace: сохранить кред (напр. разовый `git ls-remote`/`clone`
репо marketplace с GitHub-PAT, чтобы osxkeychain его запомнил).

> **Хост marketplace ≠ рабочий git-хост.** Кред для скачивания плагина — под GitHub-хост репо marketplace
> (VPN не нужен). Кред для рабочего фронт-репо (§2.0) — под ваш git-хост (может требовать VPN). Это две
> разные записи в osxkeychain.

**3. Свежесть версии** (пока autoUpdate не включён) — **сверяй В НАЧАЛЕ сессии, не откладывай к концу**
(иначе рискуешь весь флоу отработать на устаревшем плагине):
```bash
ls -d ~/.claude/plugins/cache/delivery-toolkit/delivery/*/ 2>/dev/null              # установленный SHA
git ls-remote delivery-toolkit refs/heads/main    # актуальный SHA на main
```
Локального клона marketplace нет → свежий SHA бери через `ls-remote` (а не `origin/main`). Installed
отстаёт → `/plugin marketplace update delivery-toolkit` → `/reload-plugins`. Если `autoUpdate: true`, а версия
всё равно отстала — это chicken-and-egg выше: на старте логин к репо marketplace ещё не был в keychain, апдейт — no-op.

**Не блокирующий шаг:** не до настройки сейчас — продолжай флоу; но автообновление стоит включить один
раз при первом заходе человека.

### 2.8 Дев-сервер на свежем git-worktree

Свежий worktree не имеет `.certs/` и не знает о занятости порта. Проверить заранее, чтобы не ловить
`EADDRINUSE` и `ERR_SSL_UNKNOWN_ERROR_CODE` в разгар сборки.

**Сертификаты mkcert — переиспользовать, не пересоздавать:**
```bash
# Проверить, есть ли уже сертификат в соседнем worktree (напр. основной клон фронта):
ls "<путь-к-основному-worktree>/.certs/⟪ADAPT: локальный dev-хост, напр. local.example.dev⟫*.pem" 2>/dev/null

# Если есть — скопировать в текущий worktree (не надо bootstrap certs):
mkdir -p "<путь-к-новому-worktree>/.certs"
cp "<путь-к-основному-worktree>/.certs/⟪ADAPT: локальный dev-хост, напр. local.example.dev⟫*.pem" "<путь-к-новому-worktree>/.certs/"
```
`/etc/hosts` глобальный (`127.0.0.1 ⟪ADAPT: локальный dev-хост, напр. local.example.dev⟫`) — правится один раз на машину, в новом worktree
перепроверять не нужно.

Если сертификатов нет ни в одном worktree — **человек** запускает `⟪ADAPT: команда bootstrap certs, напр. bootstrap:certs⟫` (интерактив, Keychain):
```
╔══ Нужен bootstrap certs ═══════════════════════════════
  Что:     сертификат mkcert для ⟪ADAPT: локальный dev-хост, напр. local.example.dev⟫
  Зачем:   дев-сервер на https://⟪ADAPT: локальный dev-хост:порт, напр. local.example.dev:8443⟫
  Что сделать: из <путь-к-worktree>/⟪ADAPT: рабочая директория, напр. apps/<app>⟫:
               npm run ⟪ADAPT: команда bootstrap certs, напр. bootstrap:certs⟫
               (установит корневой CA в Keychain; нужен интерактив)
  После:   скажите «готово»
══════════════════════════════════════════════════════════╝
```

**Проверить занятость порта ⟪ADAPT: порт dev-сервера, напр. 8443⟫ перед стартом:**
```bash
lsof -nP -iTCP:⟪ADAPT: порт dev-сервера, напр. 8443⟫ -sTCP:LISTEN
```
Если порт занят (например, висит дев-сервер другого worktree) — `EADDRINUSE` при запуске. Убивать
**только с согласия человека** (чужой сервер мог быть нужен):
```
⚠️ Порт ⟪ADAPT: порт dev-сервера, напр. 8443⟫ занят
Процесс: <вывод lsof>
Убить его? Это остановит чужой дев-сервер.
Скажите «да» — убью и запущу новый.
```

**Запускать дев-сервер с логом, не перенаправляя в `/dev/null`:**
```bash
# Правильно (лог виден; нужен для until-loop-вотчера «сервер готов»):
npm start > /tmp/dev.log 2>&1 &

# Затем ждать первой успешной компиляции:
until grep -q "⟪ADAPT: маркер успешной сборки в логе, напр. webpack compiled⟫" /tmp/dev.log; do sleep 3; done
echo "Сервер готов → https://⟪ADAPT: локальный dev-хост:порт, напр. local.example.dev:8443⟫"
```
Без лога нельзя понять, когда сборка завершилась, и until-loop-вотчер не сработает.

### 2.9 Рабочая папка флоу и раскладка задач

Проверь, **где человек ведёт артефакты флоу** (спеки, планы, прототипы, скрины, заметки). Дефолт —
**отдельная папка под флоу**, внутри — папка на задачу `tasks/<тикет>-<слаг>/` с короткими именами
(`design.md`/`plan.md`/`prototype.html`/`notes.md`/`retro.md`/`screenshots/`). Канон и процедура —
`delivery_orchestrator/references/workspace-layout.md`.

- Рабочая папка не заведена / задачи ведутся «где попало» (Desktop, Downloads, корень home) →
  предложить завести **отдельную папку под флоу** (напр. `~/<команда>-delivery/`) и вести задачи там.
- **Рабочая папка — свалка** (в корне вперемешку `*.html`-прототипы, `delivery-notes-*.md`,
  `prototypes/`, скрины без группировки по задачам) → **проактивно предложить навести порядок**:
  собрать по `tasks/<слаг>/`. Не молча — показать, что нашли, и спросить. Выполнить можно здесь же
  или командой **`/delivery_tidy`** (та же процедура — `workspace-layout.md` § 4).
- Workspace-инструкции (`CLAUDE.md`) со своей раскладкой — приоритетнее; приводить к ней.

Разово и необязательно — **порядок не блокирует** старт флоу; это гигиена, а не гейт.

## 3. Access-гейты: как разблокировать

Если что-то не работает — **не блокируйся молча**. Выдай чёткую плашку: что именно нужно сделать
и куда писать. После того как человек всё разблокирует — продолжай флоу с того места, где остановился.

Подробные инструкции по каждому гейту живут в
`delivery_orchestrator/references/access-gates.md` — обращайся туда за деталями.

**Шаблон плашки:**

```
╔══ Нужен доступ ══════════════════════════════════════════
  Что:   <название доступа/инструмента>
  Почему: <одна строка — зачем нужно>
  Что сделать: <конкретный шаг>
  Куда писать: ⟪ADAPT: ваш чат-канал для запросов доступа⟫
══════════════════════════════════════════════════════════╝
```

**Основные гейты:**

| Гейт | Признак отсутствия | Что просить/делать |
|---|---|---|
| **dev-VPN** | `git fetch` таймаут; нет связи с рабочим git-хостом | Запустить VPN-клиент; написать в ⟪ADAPT: ваш чат-канал для запросов доступа⟫ если нет клиента |
| **Доступ к фронт-репо** | `git push` ошибка 403/permission denied | Попросить роль **Developer** на `⟪ADAPT: группа/путь репо⟫` у ⟪ADAPT: кто в команде даёт доступ к репо⟫ (личка / общий командный чат), запасной канал — ⟪ADAPT: ваш чат-канал для запросов доступа⟫. Developer — безопасный минимум (push ветки + MR, без мержа в protected `main`). Не дают — альт. процесс пуша с разработчиком |
| **Аккаунт на git-хосте** | Нет логина на рабочем git-хосте | Завести аккаунт; написать в ⟪ADAPT: ваш чат-канал для запросов доступа⟫ |
| **MCP figma/notion** | `claude mcp list` — нет в списке / `✘ Failed to connect` | Агент чинит сам: `claude mcp get` → если `stdio`+URL, `remove` + `add --transport http` (см. 2.6). Если нужен OAuth — человек подключает через **терминальный `claude`** (десктоп OAuth не доводит), возможен рестарт. **Фолбэк — вставить содержимое вручную.** Не помогает → ⟪ADAPT: ваш чат-канал для запросов доступа⟫ |

## 4. Человек vs агент: кто что делает

Некоторые шаги требуют sudo, Keychain или интерактивного ввода — агент в песочнице это не может.
**Всегда давай точные команды и объясняй, что именно вводить.**

| Действие | Кто делает | Почему |
|---|---|---|
| `brew install node@⟪ADAPT: мажор Node, напр. 24⟫`, `export PATH=...` | **Человек** | Изменяет системный PATH; требует sudo или профайл шелла |
| `rm -rf node_modules && npm run ⟪ADAPT: команда bootstrap dev, напр. bootstrap:dev⟫` | Агент (с разрешения) или человек | Долгая операция; агент может запустить если есть доступ к терминалу |
| Настройка `/etc/hosts` (`127.0.0.1 ⟪ADAPT: локальный dev-хост, напр. local.example.dev⟫`) | **Человек** | Требует `sudo` |
| `mkcert` / `⟪ADAPT: команда bootstrap certs, напр. bootstrap:certs⟫` | **Человек** | Устанавливает корневой сертификат в Keychain (интерактив) |
| Запуск VPN-клиента / подключение к VPN | **Человек** | GUI-приложение, интерактив |
| **PAT рабочего git-хоста** (+ опц. токен i18n-платформы) — §2.0, одной командой | **Человек** | `GIT_API_TOKEN` в `~/.zshenv` + тот же PAT в osxkeychain (`git credential approve`). Секреты не в чат. ⟪ADAPT: если есть i18n-платформа/TMS — её токен туда же; если он короткоживущий: `401` при pull = протух → перевыпустить в UI платформы и повторить команду §2.0.⟫ |
| Проверка/починка конфига MCP (`claude mcp list` / `get` / `remove` / `add --transport http`) | **Агент** | Не интерактив — из Bash |
| Аутентификация MCP (`/mcp` → Authenticate) — **через терминальный `claude`**, не десктоп | **Человек** | OAuth в браузере; из десктопа часто не доходит, возможен рестарт сессии |

Когда человеку нужно что-то сделать — дай **точную команду или шаг**, не просто «настройте».

## 5. Критерий готовности (чек-лист)

После прохождения всех проверок подтверди, что все пункты выполнены:

```
Готовность к флоу доставки:

  [ ] CLAUDE.md фронт-репо прочитан (правила кода загружены в контекст)
  [ ] Требуемый Node активен (node -v = ⟪ADAPT: ожидаемая версия, напр. v24.x.x⟫)
  [ ] bootstrap выполнен на требуемом Node (нативные бинари корректны)
  [ ] git установлен, credential.helper настроен
  [ ] Фронт-репо склонировано, git fetch проходит (dev-VPN поднят)
  [ ] Права ≥ Developer на фронт-репо подтверждены
  [ ] MCP, нужные задаче (notion — при карточке Notion; figma — при макете), Connected (через `claude mcp list`)
  [ ] Ненужные задаче MCP не блокируют флоу; фолбэк «вставить содержимое вручную» допустим
  [ ] Плагин delivery: marketplace на `main` (не feature-ветке), автообновление включено (autoUpdate в user settings) — или версия вручную свежая
  [ ] Если автообновление не настроено — человеку предложено включить тумблер `/plugin` → Marketplaces → `delivery-toolkit` → Enable auto-update
  [ ] Рабочая папка флоу определена; артефакты ведутся по задачам (`tasks/<слаг>/`). Свалка? — предложен `/delivery_tidy`
```

Если хотя бы один пункт не выполнен — не переходи к следующему шагу флоу: сначала разблокируй
через инструкцию из раздела 3.

Когда все пункты выполнены — сообщи человеку: **«Всё готово, можем начинать»** и передай
управление `delivery_orchestrator`.
