# delivery_qa_smoke — лёгкий QA-смоук на слоте стейджа

Лёгкий встроенный смоук-прогон изменений ветки MR на стейдже.
Задача — до того как QA берёт полноценное ревью, убедиться: основное работает, явных поломок
рядом нет, а для багфикса — сам баг устранён.

---

## 1. Когда запускать

**После того как черновой MR создан** — держим его в статусе **Draft** до прохождения смоука.

| Трек | Объём смоука |
|------|-------------|
| **Трек A** (новая фича/идея) | Полный смоук: вся затронутая поверхность + smoke-чеклист |
| **Трек B** (багфикс / правка существующего) | Всегда лёгкий: баг устранён + нет регресса рядом |

Смоук **не заменяет** полноценное QA — по его результатам QA получает готовый чеклист и отчёт.

## 2. Предусловия

### dev-VPN до стейджа
Включи **VPN-клиент** (`⟪ADAPT: ваш VPN-клиент⟫`). Без VPN стейдж недоступен — смоук остановится на
проверке доступности слота; выдай отчёт с пометкой `[не проверено: нет VPN]`.

### Playwright MCP
Playwright должен быть подключён как MCP-инструмент (браузерная автоматизация).

**Проверка:**
```bash
claude mcp list
```
Ожидаемый вывод содержит строку с `playwright` и статусом `✓ Connected`.

**Если не установлен — добавить:**
```bash
claude mcp add --scope user --transport stdio playwright -- npx -y @playwright/mcp@latest --isolated
```
После — снова проверить `claude mcp list`.

> `--isolated` обеспечивает инкогнито-режим (чистый профиль на каждый прогон).
> Это **разовая настройка**. Если команда не идёт или ты не работаешь с терминалом — попроси помочь
> в `⟪ADAPT: ваш чат-канал для запросов доступа⟫` (или передай смоук QA).

### Где брать стейдж для смоука
**Основной путь — агент сам деплоит временный стейдж и забирает ссылку** через GitLab API (ниже, «Стейдж
через GitLab»). Человеку не надо идти в GitLab. Подходит всем, у кого нет своего слота.

**Если у тебя есть свой слот** — используй его: URL и креды храни **исключительно в локальном `.env`** рабочей
папки (**не в плагине, не в git**):
```
SLOT_URL=https://<url-твоего-слота-стейджа>
⟪ADAPT: секреты авторизации на стейдже, если нужны — см. references/auth-scenario.md; в исходном проекте было TEMP_WALLET_PASSWORD=<пароль-temp-кошелька>⟫
```
`.env` — в `.gitignore`; `SLOT_URL` и пароли не коммить. Нет ни своего слота, ни `.env` → идёшь по основному пути (ниже).

### Стейдж через GitLab — авто-деплой (основной путь)

Временный стейдж = окружение `⟪ADAPT: шаблон имени временного окружения, напр. <окружение>-<MR_iid>⟫` (на скрине MR это кнопка
**Deploy Development**, после неё — **View app**). **Агент достаёт ссылку и при необходимости деплоит сам** —
человеку не надо идти в GitLab.

Нужно: **dev-VPN**, **токен со scope `api`** (`write_repository` на API даёт `403`, см. `delivery_setup §2.3`),
`curl`. Проект `⟪ADAPT: группа/путь репо⟫`. **MR-JSON парсить с `strict=False`** — в title/description бывают
control-символы (иначе `json.load` падает `Invalid control character`).

**Шаг 1 — забрать ссылку (часто слот УЖЕ поднят → деплой не нужен).** Идти от **окружения**, а не от имени джобы:
```bash
PROJ=⟪ADAPT: URL-encoded группа/путь репо⟫; API=https://⟪ADAPT: ваш git-хост⟫/api/v4
H="PRIVATE-TOKEN: $GIT_API_TOKEN"          # scope api; не в чат, не в git (настроен в delivery_setup §2.0)
MR=<iid>
curl -s -H "$H" "$API/projects/$PROJ/environments?states=available&search=⟪ADAPT: префикс имени окружения⟫-$MR" \
  | python3 -c "import sys,json;[print(e['id'],e['name'],'->',e.get('external_url')) for e in json.loads(sys.stdin.read(),strict=False)]"
```
Есть `external_url` при `state: available` → URL получен. **Но `available` в API ≠ рабочая страница**
(видели вживую: окружение `available`, а по URL `404`) → перед отдачей ссылки **обязателен HTTP-healthcheck
(Шаг 3)**. `200`+контент — отдавай (`SLOT_URL`=этот URL); `404` — нужен деплой (Шаг 2).

**Шаг 2 — задеплоить фронт на слот (если URL не отвечает `200` или нужен свежий деплой).** Целевая deploy-джоба
stage `devenv` — **`Deploy Development`** (она кладёт фронт на URL; `Run Development` сама по URL фронт НЕ отдаёт).
**Не полагайся на `environment.last_deployment.deployable.name`.** Найди в пайплайне (`GET /merge_requests/$MR` →
`head_pipeline.id` → `GET /pipelines/<id>/jobs`) джобы stage `devenv` и действуй **по статусу `Deploy Development` —
два проверенных паттерна:**
- **`Deploy Development` = `manual`** → играть напрямую: `POST /jobs/<deploy_id>/play` (или `/retry`, если уже игралась).
- **`Deploy Development` = `skipped`** (зависит от `Run Development`; на нём `play` → `400 Unplayable`, `retry` → `403 not retryable`)
  → сначала сыграть **`Run Development`** (`manual`): `POST /jobs/<run_id>/play` → поллить до `success` (~45с); после этого
  **`Deploy Development` запускается САМ** (`running`) — вручную не трогать, поллить его до `success` (~15с).
```bash
# id джоб: name=='Run Development' / name=='Deploy Development', stage=='devenv'
curl -s -X POST -H "$H" "$API/projects/$PROJ/jobs/<job_id>/play"   # Run (если Deploy skipped), иначе Deploy (если manual)
# poll GET /jobs/<job_id> до success (на macOS без shell `timeout` — фоновый поллер Bash-инструмента)
```

> 🔴 **zsh-гоча (Bash-инструмент = zsh).** В циклах play/poll по нескольким джобам (Desktop/Tablet/Phone e2e и т.п.)
> НЕ полагайся на word-splitting `for J in $JOBS`: zsh, в отличие от bash, **не разбивает** неквотированную
> переменную на слова — цикл прокрутится **один раз** со всей строкой-id, `play`/poll ударят по несуществующему
> id и **молча ничего не сыграют** (джобы останутся `manual`, а поллер будет ждать «пустой» статус). Перечисляй
> id **явными литералами** (`for J in 1938622 1938623 1938624`) или массивом (`jobs=(a b c); for J in $jobs`).
> Реальная грабля из практики: e2e «висели», хотя всё было готово.
`failure_reason: runner_system_failure` (`allow_failure: true`; частый кейс — `jfrog … 403` при pull образа
раннера) — инфра, не код: `POST /jobs/<id>/retry`.

**Шаг 3 — ОБЯЗАТЕЛЬНЫЙ HTTP-healthcheck URL. `success` в GitLab ≠ рабочая страница.** Фронт встаёт с лагом
(~1–2 мин ПОСЛЕ `success`). Поллить `external_url`, пока не **`200` И контент приложения** (⟪ADAPT: маркер контента приложения в HTML, напр. `<title>…`⟫),
а не `404`/nginx-default/maintenance:
```bash
for i in $(seq 1 15); do
  code=$(curl -sL -o /dev/null -w '%{http_code}' -m 12 "$URL")
  [ "$code" = 200 ] && curl -sL -m 12 "$URL" | grep -q '⟪ADAPT: маркер контента приложения, напр. <title>…⟫' && { echo UP; break; }
  sleep 20
done
```
**Ссылку человеку отдавать ТОЛЬКО после `200`+контент.** Не поднялось за ~5 мин → не отдавать, эскалировать
(возможно сыграна не та джоба или деплой неполный).

**Шаг 4 — предложи проверить фичу сам.** Когда URL отвечает `200`, **проактивно предложи: «могу сам
прокликать фичу на стейдже — скажи только, что проверить (в каком состоянии / на каких данных)».** Дальше
гони смоук Playwright-ом (§3 ниже) на этом URL: вход по процедуре из `references/auth-scenario.md`, нужные действия (логин/подготовка данных),
desktop + mobile. Человеку остаётся подтвердить сценарий, а не кликать самому.

> ✅ **Обкатано в бою:** API-механика (`play`/`retry`/poll) + забор
> `external_url` подтверждены. **Рабочая deploy-джоба — `Deploy Development`** (`devenv`); URL отдаёт `200`
> (⟪ADAPT: маркер контента приложения, напр. `<title>…`⟫) ~1–2 мин ПОСЛЕ `success` → **Шаг 3 (healthcheck) обязателен** (`success` ≠ рабочая страница).
> **Два паттерна джоб (зависит от конфигурации пайплайна):** на одном MR `Deploy Development` была `manual` (играть
> напрямую); на другом — `skipped` (зависит от `Run Development`): сперва `Run Development` (play→success ~45с),
> затем `Deploy Development` стартует САМ (а `play`/`retry` на нём → `400`/`403`). URL вида
> `⟪ADAPT: шаблон URL временного слота, напр. https://frontend-<branch>.<dev/stage-хост>⟫`. `runner_system_failure` (jfrog 403) лечится `retry`.
> **Фолбэк:** человек жмёт **Deploy Development** → **View app** сам.

## 3. Где и как проводить смоук

### Готовые e2e-job'ы CI против слота — быстрый регресс (предпочтительно перед своим Playwright)

После создания MR в его пайплайне уже есть **manual** e2e-job'ы, гоняющие полную e2e-сьюту фронта против
слота по вьюпортам: `⟪ADAPT: имена e2e-job'ов по вьюпортам, напр. End To End Desktop / Tablet / Phone Stage Slot⟫` (stage `e2e-main`; слот к этому
моменту поднят job'ой `⟪ADAPT: имя job'ы деплоя слота на MR, напр. Deploy Stage Slot On MR⟫`). Сыграть их через GitLab API:

- найти: `GET /projects/<id>/pipelines/<mr_pipeline_id>/jobs` → имена выше, `status=manual`;
- сыграть каждый вьюпорт: `POST /projects/<id>/jobs/<job_id>/play`;
- опросить `GET /projects/<id>/jobs/<job_id>` до `success`/`failed` (на macOS без shell `timeout` — фоновый
  поллер Bash-инструмента); упало → трейс `GET /jobs/<job_id>/trace`.

**Предпочитай этот путь своему Playwright-spec'у:** многие компоненты идут
**без `data-test`**, а навигация к конкретной сущности бывает по **slug/deep-link** (напр. `?dl_id=<сущность>&slug=…`, не по
id), поэтому устойчивый кастомный прогон дорог. Готовые e2e закрывают desktop+tablet+phone разом и ловят
регресс в покрытых флоу. Свой смоук (ниже) добавляй только для **фиче-специфичной** визуалки, которую
e2e-сьюта не покрывает (напр. новый контрол на мобайле), либо когда e2e-job'ов в пайплайне нет.

**Слот** — стейдж-слот, развёрнутый по ветке MR (не `main`, не прод).

> **Почему слот, а не локалка, для state-dependent UI.** Состояния, завязанные на бэк (locked/empty/role,
> сущность в нужном статусе) трудно поднять локально; вдобавок агент **не кликает сквозь mkcert
> cert-интерстишл** на `https://⟪ADAPT: локальный dev-хост⟫` (Playwright/preview упирается в предупреждение
> сертификата). На слоте — настоящий сертификат, реальные состояния и рабочий OAuth. Локалка годится лишь
> подтвердить, что собирается/рендерится.

> 🔧 **ADAPT — второй репо без стейдж-слотов (опционально).** В исходном проекте был второй фронт (лендинг),
> который **стейдж-слотов не имел** — смоук шёл на локальном dev-сервере (`npm run dev` → `localhost:<порт>`) или
> `npm run preview` (прод-сборка), desktop + mobile. Если у вас есть такой второй репо — опиши, где для него
> гонять смоук (всё про `SLOT_URL`/слот в этом файле — для основного фронта), и сошлись на его профиль в
> `delivery_orchestrator/references/second-repo.md`. Один фронт со слотами — убери эту заметку.

**Браузерные режимы:**

| Режим | Параметры |
|-------|-----------|
| Desktop | Вьюпорт `1440×900`, тёмная тема (`colorScheme: 'dark'`), инкогнито |
| Mobile | Полная эмуляция **iPhone 16 Pro** `402×760`, CDP-override (`mobile: true`, touch), тёмная тема, инкогнито |

> Тёмную тему форсируй до навигации: `page.emulateMedia({ colorScheme: 'dark' })`.
> Светлую тему проверяй только если MR её прямо затрагивает.

**Правило выполнения (важно):** прокликивай сценарий целиком, без пауз на болтовню — клик →
проверка → следующий клик подряд. **Отвечай в чат по завершении прогона**, а не после каждого клика.
Промежуточный статус — только когда реально заблокирован: нет VPN / нет баланса / упал слот / найден
баг, требующий решения.

**Инкогнито** — Playwright MCP запущен с `--isolated`, каждый прогон независим (чистый профиль).

**Авторизация** — по процедуре из `references/auth-scenario.md` (в исходном проекте — вход через временный тестовый
кошелёк в UI). Общий каркас: открыть `SLOT_URL` → дождаться сигнала готовности приложения
(⟪ADAPT: атрибут готовности, напр. `body[e2e_ready=true]`⟫) → выполнить шаги логина из auth-scenario →
убедиться в признаке авторизованного состояния (исчезла Join-кнопка, появились элементы профиля).

**Подготовка данных / баланс** — если флоу требует предзаготовленного состояния (баланс, созданная
сущность и т.п.) — выполнить подготовку по `references/test-data.md` (в исходном проекте — тестовое пополнение
«Refill»). В отчёте пометить, что использовалась тестовая подготовка данных. Если подготовить нельзя —
остановиться «до действия» и пометить `[не проверено: нет данных/баланса]`.

**Видимость ≠ DOM.** Наличие узла в DOM не означает, что пользователь его видит. Проверяй позицию
через `getBoundingClientRect` (в пределах `innerHeight`/`innerWidth`, не `display:none`/нулевой
размер); что под фолдом — **отмечай в отчёте**, проверяй прокруткой или `fullPage`-скриншотом.

**Закрывай браузер** в конце каждого прогона (`browser_close`).

## 4. Что выдать

### 4.1 Анализ изменений
Перед прогоном — разобрать diff ветки MR:
- Какие файлы/компоненты изменились.
- Какие поверхности продукта затронуты (перечень поверхностей — `references/platform-map.md`; ⟪ADAPT: перечисли основные экраны своего продукта⟫).
- **Куда смотреть в первую очередь** (самое рискованное).
- **Корнер-кейсы и edge cases** (явно, с точки зрения тест-мышления):
  - граничные значения;
  - пустые/нулевые состояния;
  - **«ещё не онбордился»**: сущность не создана/не добавлена (нет аккаунта/сессии, нулевая доходность) —
    частый источник «кнопка молча не работает»; стейдж-тест-сущности обычно живут именно в этом состоянии;
  - состояния ошибок / потери сети;
  - поведение без авторизации / с нулевым балансом;
  - desktop vs. mobile-расхождения.

- **i18n-чек (если в diff есть новые/изменённые `<FormattedMessage>`):**
  - новые ключи → `⟪ADAPT: путь к сгенерированным переводам, напр. apps/web-client/src/i18n/generated/**⟫` **перегенерён и закоммичен** в ветку MR
    (иначе на стейдже **«набор символов»** — минифицированные id). Проверять сгенерированные, **не** бэкапы (`⟪ADAPT: путь к бэкапам i18n, напр. backups/i18n⟫`).
  - на стейдже строки реально локализованы: не сырой id, не ⟪ADAPT: fallback-локаль, напр. английский⟫ на ⟪ADAPT: интерфейсная локаль, напр. русском⟫ интерфейсе.
  - сверять текст **по `id`** (через вашу i18n-платформу), а не по тексту со скрина — grep текста ловит артефакты
    разных веток. Механика и диагностика — orchestrator `references/repo-conventions.md` § i18n.

> Нет доступа к GitLab → взять описание MR / скриншоты от автора и идти к живому смоуку.
> Пометить `[diff пропущен: нет фронт-репо]`.

### 4.2 Smoke-чеклист
Сформировать и прогнать чеклист с чекбоксами `- [ ]` / `- [x]`. Структура:

```markdown
#### Desktop-прогон
- [ ] Слот открывается, `body[e2e_ready=true]`
- [ ] Авторизация через временный кошелёк — успешно
- [ ] <Основной флоу изменения: конкретный шаг 1>
- [ ] <Основной флоу изменения: конкретный шаг 2>
- [ ] Смежные поверхности не сломаны (указать какие именно)
- [ ] Нет ошибок в console/network для затронутых флоу

#### Mobile-прогон (iPhone 16 Pro 402×760)
- [ ] Нав-бар виден и работает
- [ ] <Основной флоу изменения: мобильный вариант>
- [ ] Нет расхождений desktop↔mobile, не описанных в дизайне
```

Конкретные пункты формируются на основе анализа diff (п. 4.1).

> **Важно: чеклист = только кликабельные ручные проверки, не процессные шаги.**
> Каждый пункт — конкретное действие с конкретным ожидаемым результатом:
> что нажать, что должно произойти. Примеры уместных пунктов:
> - `- [ ] Фича открывается только у агентов из allowlist — у других кнопки нет`
> - `- [ ] Ввод суммы выше максимума → значение клампится до max, не ломает UI`
> - `- [ ] Слайдер и поле ввода синхронизированы: изменение одного обновляет другое`
> - `- [ ] Кнопка «Назад» из калькулятора → возврат в паспорт, не закрывает оверлей`
> - `- [ ] Mobile: элементы не перекрывают друг друга, кнопка в зоне досягаемости`
>
> **Не класть в чекбоксы:** `i18n:build закоммичен`, `дизайнер одобрил`, `фронтендер посмотрел`.
> Это процессные шаги — они идут отдельным параграфом (например «Before removing Draft»), не чекбоксами.

### 4.3 Для багфикса (Трек B) — before/after подтверждение

```markdown
#### Before (воспроизведение)
- Описание поведения до фикса: <…>
- Шаги воспроизведения: <…>
- Ожидаемый результат: <…>
- Фактический результат до фикса: <…>

#### After (после фикса)
- [ ] Баг не воспроизводится: <конкретный шаг проверки>
- [ ] Граничные сценарии рядом не сломаны: <перечислить>
```

### 4.4 Правила отчётности
- Не выдумывать результаты прогона. Шаг не выполнен (нет VPN / нет баланса / упал слот) → `[не проверено: причина]`.
- Скриншоты — в `artifacts/<mr-слаг>/` (имена `NN-краткий-шаг.png`; мобильные — с префиксом `phone-`).
- Зафиксировать слот и ветку MR, против которых гонялось.
- Усиливать отчёт явными AC (acceptance criteria) и edge cases — это конкурентное преимущество
  команды с QA-бэкграундом.

## 5. Цикл: найденное → правки → повторный смоук

```
смоук → баги/замечания
            ↓
        правки — в ТУ ЖЕ ветку MR
            ↓
        повторный смоук (только затронутые флоу)
            ↓
        все пункты чеклиста зелёные → хэндофф QA
```

- Правки идут только в ту же ветку (MR остаётся Draft).
- Не создавать новые ветки для фиксов смоука.
- После правок перепроверить только затронутые пункты, а не весь чеклист заново.

---

## 6. Хэндофф QA

После того как все пункты smoke-чеклиста пройдены:

1. MR остаётся **Draft** — мерж/деплой делает QA или фронтендер после полного ревью.
2. Передать QA в **Slack** (`⟪ADAPT: ваш чат-канал для запросов доступа⟫`):
   - ссылку на MR;
   - заполненный smoke-чеклист;
   - скриншоты (или ссылку на `artifacts/<mr-слаг>/`);
   - before/after для багфикса (если Трек B);
   - `[не проверено: причина]`-пункты — явно.

Пример хэндоффа:
```
🔍 QA Smoke — MR !<номер> «<заголовок>»
Слот: <SLOT_URL>  Ветка: <branch>

Smoke-чеклист:
✅ Авторизация — ок
✅ <флоу 1> — ок
❌ <флоу 2> — <краткое описание бага> (скрин: artifacts/<mr-слаг>/03-bug.png)
⚠️  <флоу 3> — [не проверено: нет баланса]

Передаю на полноценное QA.
```

## 7. Связь с другими QA-скиллами плагина

Этот скилл — **лёгкий встроенный смоук** в рамках delivery-флоу. Рядом в этом же плагине:

- **`qa_mr`** — полный QA-разбор любого (обычно чужого) MR: diff с анализом компонентов, маппинг
  `data-test`, живой прогон, UX/UI-советы, отчёт. Зови его, когда MR отдают «на QA», а не для
  смоука собственной доставки.
- **`autotest` / `selectors_sync` / `autotest_run`** — durable-автотесты в `⟪ADAPT: путь к e2e-тестам, напр. e2e/web⟫`:
  написать тест по итогам смоука/разбора, синхронизировать `data-test`-селекторы, прогнать и
  довести до зелёного.

Справочники `references/` этого скилла (auth-scenario, platform-map, test-data) — общий источник
правды для всех QA-скиллов плагина; правки фиксируй здесь, не плоди копий.
