# Тест-данные и ограничения стейджа

## Подготовка данных / баланс (каркас — generic)
- **Граница по умолчанию (без подготовки данных):** проверяем «до действия» — экран открывается, форма
  открывается и **валидируется** (мин/макс, пустые поля, недостаток прав/средств), переключатели работают;
  **без фактических необратимых действий**.
- **С подготовкой данных (когда флоу требует состояния):** выполнить минимальную подготовку → пройти
  сценарий. **Помечай в отчёте, что использовалась тестовая подготовка данных.**
- Шаги, которые не удалось завершить (не нашли кнопку / не хватило данных или прав), помечай
  `[не проверено: нет данных/баланса]`.

## Процедура подготовки тестовых данных

> 🔧 **ADAPT — тестовые данные вашего продукта.** Опиши, как на стейдже привести систему в нужное для
> смоука состояние (создать сущность, пополнить баланс, выдать роль/флаг и т.п.), с реальными
> `data-test` и, если есть, **API-ручками для подготовки напрямую (минуя UI, для ускорения смоука)**.
> Для справки, как это выглядело в исходном проекте (тестовое пополнение «Refill»):
>
> 1. Авторизоваться (см. `auth-scenario.md`).
> 2. Открыть **Deposit** (кнопка в шапке или онбординг-поп-ап → Add funds); страница депозита
>    `layouts:web:desktop-body-content-deposit`.
> 3. Вкладка **Crypto** → блок **Refill**: поле **Amount** + чипы **10 / 100 / 1000** (`data-test="Chip"`)
>    + кнопка **Refill** (`data-test="Button"`, disabled пока сумма пустая).
> 4. Выбрать чип → **Refill** → `POST /api/balance/top-up/v1 → 200`, баланс аккаунта обновляется.
>
> **API-ручки (пример из исходного проекта):** `POST /api/balance/top-up/v1` (пополнение),
> `POST /api/balance/account/create/v1` (создание аккаунта), `POST /api/balance/operation/up/list/v1`
> (история). **Грабля-урок (generic):** элементы формы часто с generic `data-test` (`Button`/`Chip`/`Input`)
> — опирайся на контейнеры + текст/role, фронту заведи задачу на уникальные id.

## Тест-данные под ключевые фичи

> 🔧 **ADAPT — тест-данные под фичи вашего продукта.** Для каждой значимой фичи — какие сущности/данные на
> стейдже её покрывают и в каком состоянии. В исходном проекте основным объектом доставки был **центральный объект**
> (см. `context/dca.md`), и здесь лежали живые тест-данные под его функции — для примера структуры:
>
> - **Гейт по id-allowlist:** фича включалась только для перечисленных id (напр. отдельный расчётный экран —
>   флаг `isCalculatorEntity`); в таблице хранили пары «окружение → id → есть ли фича» (напр. прод `<id-A>` — с
>   расширенным контролом, стейдж `<id-B>` — без) плюс каким id проверять каждый вариант.
> - **Проверка события/настроек:** какой оверлей открыть, что изменить, какое событие с какими свойствами
>   ждать (в исходном проекте — событие сохранения настроек со свойствами вида `<свойство-1>`+`<свойство-2>` в
>   соответствующем оверлее настроек).
> - **Share/производные фичи:** откуда доступна и на какой сущности со стейджа проверять (напр. сущность в
>   активном состоянии).
>
> Замени на свои фичи и id; сохрани связку «фича → тест-сущность → как проверить».
