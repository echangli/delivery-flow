# Техники тест-дизайна — общий арсенал (канон)

Домен-агностичный набор техник: проектируй кейсы **по технике, а не «на глаз»**. Годится обеим
границам — **UI** (фронт: формы, состояния экрана, роуты) и **API** (бэк: ручки, события, контракты).
Канон; реиспользуется скиллами `qa_mr` (UI-разбор MR) и `qa_backend` (бэкенд/API-QA) — не дублируй,
ссылайся сюда.

## Дисциплина оракула (читать до проектирования кейсов)

- **На каждый кейс — источник ожидания (оракул) ДО прогона:** спека/тикет, код, заметки разработчика
  или паритет с эталоном. Нет источника ожидания — кейс не годен («факт есть, правильность неизвестна»).
  Ожидание пиши **до** прогона, не подгоняй под факт.
- **Факт ≠ гипотеза.** Наблюдаемое (статус, значение, лог-строка) держи отдельно от вывода. Аномалия
  ≠ баг, пока не исключил дрейф стенда/чужой трафик/фикстуру.
- **NOT-RUN не молчать.** Незапущенный кейс — явный статус `⏳NOT-RUN`, не пропуск и не FAIL.
  «done/PASS» — только при наблюдаемом доказательстве.

## Как применять

Прогоняй **каждый вход** и позитивы, и негативы через одни и те же линзы. Не считай покрытым, пока не
прошёл по списку и не отметил каждую линзу: **покрыто / N/A с причиной**. Веди это в матрице/тест-плане
(колонки «Оракул» и «Техника/линза» — см. `test-plan.md`).

---

## 1. Эквивалентные классы (EP) — и для позитива тоже

Для каждого входного параметра выпиши валидные и невалидные классы, бери по 1 представителю. Позитив
≠ один «идеальный» вход: прогони по представителю **каждого** валидного класса.
- **UI:** каждый допустимый тип актива/агента/роли; оба пути входа (десктоп/мобайл, разные точки входа).
- **API:** разные источники/каналы, каждая допустимая роль/бренд/проект, оба транспорта.

## 2. Граничные значения (BVA)

Для полей с длиной/диапазоном/размером/временем/лимитом: `min-1, min, min+1 … max-1, max, max+1` +
пусто/0/один-символ/переполнение. Границы прогоняй и на «годном» (позитив на min/max), и за границей
(негатив). Примеры: длина строки/кода; id (`0, -1, max int, нечисловой, ведущие нули`); TTL/срок (ровно
на границе, истёкший на секунду); лимиты (`N-1, N, N+1`); суммы/пагинация/оффсеты.

## 3. Таблица решений

Где исход зависит от **комбинации** условий (флаг + состояние + роль + гео/тип): построй таблицу
условие→исход и покрой значимые комбинации, не только «всё true». Пример: гейт фичи по id-агента +
состоянию сессии + балансу.

## 4. Переходы состояний

Для сущностей с ЖЦ (created→confirmed; active→expired→consumed; valid→revoked): каждый валидный переход
**И** запрещённые:
- повтор из конечного состояния = **идемпотентность**;
- действие из неверного состояния;
- использование потреблённого/удалённого/отозванного.

## 5. Негативы + authz-линза  (обязательно для авторизации/платежей/подтверждений)

- чужой / протухший / не-того-скоупа токен;
- **replay** уже использованного артефакта/колбэка/события;
- **cross-user / cross-tenant** (валидный артефакт одного субъекта + id другого — чужой НЕ должен
  пройти, и его сущность не должна измениться);
- эскалация прав; пропуск/подмена шага флоу; отозванная сессия.

Негативы прогоняй **дифференциально**, если есть baseline (эталон/прод/main → ветка) — см. дифф-прогон
в `qa_backend`. Красный флаг: понятная ошибка на эталоне → 5xx/пустой ответ/таймаут/новый неописанный
код на ветке.

## 6. Rate-limit / брутфорс

Обязательная полоса для авторизации/кодов/попыток: гони `N+1` неверных подряд, **зафиксируй границу
срабатывания** (какая попытка флипает в throttled/санкцию — сверь с лимитом из конфига) и проверь **что
НЕ считается в лимит** (отбраковка по формату ДО счётчика). Убедись, что неверные попытки НЕ портят
валидную сущность и не блокируют легитимный успех после. Часто лимит **per-id** (перебор по разным id
не ограничен — отметь).

## 7. Инъекции / спецсимволы / кодировка

В текстовые входы: кавычки/`;`/`--`/юникод/URL-энкодинг/`../`/пустая строка/`%00`. Ожидание —
**безопасная ошибка** (НЕ 5xx, НЕ инъекция, НЕ traversal). Проверь **кейс-чувствительность
идентификаторов**: коллация БД `_ci` делает матч регистронезависимым → может ослабить коды/токены;
прогони валидный идентификатор в другом регистре и сверь с ожиданием.

## 8. Конкурентность / идемпотентность

double-submit (реально **параллельно**), повторный колбэк/событие, гонка на общем ресурсе. Проверяй не
только финальный статус, но и **отсутствие двойных сайд-эффектов** (двойное событие/списание/письмо).

## 9. API-контракт и HTTP/gRPC-семантика  (для любой ручки/метода)

- **Коды/семантика метода:** GET без сайд-эффектов; PUT/DELETE идемпотентны; `201/204/404/409/422` по
  смыслу, не «200 на всё»; gRPC-статусы вместо «OK с ошибкой в теле».
- **Сериализация:** content-type; `null` vs отсутствующее поле vs пустая строка; точность чисел/денег,
  формат дат/таймзон; неизвестные/лишние поля — игнор или ошибка строго по спеке.
- **Пагинация/сортировка/фильтры:** курсор/оффсет, стабильность порядка, пустая и последняя страница,
  невалидный/протухший курсор.
- **Заголовки и версия:** обязательные (auth, correlation-id, idempotency-key), поведение при их
  отсутствии; версионирование ручки.
- **Транспорт:** для браузерных API — CORS/preflight; таймауты/ретраи и **частичный отказ даунстрима**
  (клиент видит понятную ошибку, а не зависание/5xx).

## 10. Регрессия вокруг изменённого

По кросс-импакту: соседнее существующее поведение; базовый набор, не только новый код. Каждый пункт —
обоснован (какая зависимость).

## Опциональные линзы (включай, если применимо к задаче)

- **Производительность / нагрузка** — менялся горячий путь/запросы к БД/размер выборки/таймауты/пулы:
  зафиксируй SLO-оракул (latency/throughput из требований), проверь под реалистичной нагрузкой; следи за
  N+1, ростом задержек, исчерпанием пула. Нет требований — согласуй ожидание с владельцем.
- **Корректность data/schema-миграций** — не только «применилась»: корректность backfill, поведение
  отката (down), совместимость старого кода во время раскатки (rename/добавление NOT NULL),
  блокировки/длительность на больших таблицах, **mixed-state** старых данных (старое+новый код).
- **Async / cron / consumer** — корректность результата, идемпотентность при повторе, поведение при
  рестарте посреди работы, «ядовитое» сообщение, отсутствие потери/дублей.
- **Backward-compat контрактов** — старый клиент/консюмер продолжает работать (доп. поле не ломает;
  удаление/сужение — breaking, планируй порядок деплоя / mixed-version-окно).

---

## Трекинг покрытия

Каждая линза выше — в матрице как **покрыто** / **N/A с причиной**. «Что не проверено» всегда под рукой
(живая матрица). Заранее недостижимое на окружении (geo/IP, отсутствующие интеграции, реальные внешние
провайдеры) — помечай ⛔ с причиной, циклы не трать.
