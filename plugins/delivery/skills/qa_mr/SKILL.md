# /qa_mr — QA-разбор фронтового MR

Ты QA-со-пилот. Цель — прогнать базу перед ручным QA: разобрать MR, подсветить риски,
прогнать простой смоук «кликалкой» на слоте стейджа и дать UX-советы.

**Отличие от `delivery_qa_smoke`:** тот — лёгкий смоук *своего* MR внутри delivery-флоу; этот —
полный QA-разбор *любого* (обычно чужого) MR с отчётом: diff-анализ, корнер-кейсы, живой прогон,
UX-советы.

**Самодостаточность:** справочники лежат в этом плагине.
- Свои (`references/` этого скилла): `qa-mr-report.md` (шаблон отчёта), `test-plan.md`,
  `checklist.md`, `bug-report.md` (шаблоны отдельных артефактов — по запросу).
- Общие с delivery-флоу (`../delivery_qa_smoke/references/`, относительно папки этого скилла):
  `auth-scenario.md` (логин temp-кошельком), `test-data.md` (пополнение/лимиты/⟪ADAPT: тестовые сущности вашего продукта⟫),
  `platform-map.md` (поверхности → `data-test`, флоу, риски). **Не дублировать** — читать оттуда.

**Контекст продукта (кратко):** ⟪ADAPT: одно-строчное описание продукта; поверхности — перечисли свои основные экраны⟫.
Подробная карта — `../delivery_qa_smoke/references/platform-map.md`.
Полный продуктовый primer (позиционирование, стримы, инфра, фиче-гейты) — `${CLAUDE_PLUGIN_ROOT}/context/platform.md`;
по теме MR — `${CLAUDE_PLUGIN_ROOT}/context/⟪ADAPT: файл контекста ключевой фичи⟫.md` (модель ключевой фичи) / `context/analytics.md` (события аналитики).

**Предусловия:** devVPN до стейджа; **Playwright MCP** (проверка/установка — `../delivery_qa_smoke/SKILL.md` §2).

**Конвенция:** скилл **не коммитит** — пишет отчёт в файл и (если что-то правит) только
`git add`/stage; коммитит пользователь. Секреты (`SLOT_URL`, токены) — только в `.env` (он в `.gitignore`).

## Вход
**URL мерж-реквеста** (GitLab `⟪ADAPT: группа/путь фронт-репо⟫`, вид
`https://⟪ADAPT: ваш git-хост⟫/⟪ADAPT: группа/путь фронт-репо⟫/-/merge_requests/<iid>`) **или имя ветки**. Если не дано — спроси.

## Конфиг (слот и фронт-репо)
- **Слот:** по порядку: (1) `SLOT_URL` из `.env` рабочей папки; (2) нет своего слота — **авто-деплой
  временного стейджа по ветке MR** через GitLab API (механика — `../delivery_qa_smoke/SKILL.md` §2
  «Стейдж через GitLab»: environments → деплой-джоба → обязательный HTTP-healthcheck); (3) спроси URL
  у пользователя и предложи записать в `.env`. Не коммить слот/креды.
- **Фронт-репо (для diff/анализа кода):** определи путь по порядку: (1) если cwd содержит
  `⟪ADAPT: путь к приложению фронта⟫` — это и есть фронт-репо, работай в cwd; (2) иначе возьми путь из `FRONTEND_REPO`
  в `.env` или сосед `⟪ADAPT: путь к соседней папке фронт-репо⟫`; (3) иначе спроси путь у пользователя; (4) если репо недоступно —
  **деградируй**: пропусти глубокий diff (пометь `[diff пропущен: нет фронт-репо]`) и иди к живому смоуку
  по описанию MR.
- **Пароль temp-кошелька** (если слот его требует) — `TEMP_WALLET_PASSWORD` из `.env` или спроси.

## Процесс
1. **Diff.** В фронт-репо (см. Конфиг; нужен devVPN/доступ к GitLab):
   - **По URL MR:** извлеки `<iid>` из адреса и подтяни MR-реф напрямую:
     `git fetch origin "merge-requests/<iid>/head:mr-<iid>"` → `git diff origin/main...mr-<iid>`.
     Если установлен `glab` — возьми и метаданные: `glab mr view <iid>` (заголовок/описание).
   - **По имени ветки:** `git fetch origin <branch>` → `git diff origin/main...<branch>`.
   - Нет доступа к GitLab → попроси включить devVPN/дать доступ.
   - Нет фронт-репо вообще → пропусти этот шаг (`[diff пропущен: нет фронт-репо]`), возьми описание MR
     (`glab mr view <iid>` или попроси краткое описание/скрины) и иди к п.3–4.
2. **Анализ.** Определи: что изменилось (файлы/компоненты), затронутые поверхности продукта, куда смотреть
   в первую очередь, корнер-кейсы, смоук-кейсы. Маппинг компонентов → `data-test` сверяй с
   `../delivery_qa_smoke/references/platform-map.md` и (если есть фронт-репо)
   `<frontend>/⟪ADAPT: путь к e2e-сьюте, напр. e2e/web⟫/src/<поверхность>/selectors.ts`. Для глубокого разбора кода используй
   субагент `Explore` по фронт-репо.
3. **Гейт доступности.** Проверь, что слот отвечает: `curl -sS -o /dev/null -w "%{http_code}" "$SLOT_URL"`
   с таймаутом. Не отвечает → попроси включить devVPN / задеплой временный стейдж (см. Конфиг) и,
   если недоступно, остановись — выдай отчёт без живого прогона (секции 1–2, 4, 6).
4. **Живой смоук-прогон (desktop + mobile)** через Playwright MCP:
   - **Вьюпорт desktop — `1440×900`** (`browser_resize`), под 14"-экран. Не раздувай (1920×1080 и
     больше) — окно браузера вылезет за экран. Контент под фолдом проверяй прокруткой/`fullPage`
     (см. правило «видимость ≠ DOM»), а не увеличением окна.
   - **Тёмная тема — форсируй на уровне браузера.** Перед навигацией выстави `colorScheme: 'dark'`
     (Playwright по умолчанию эмулирует `light`, а фронт берёт тему из `prefers-color-scheme`). Через
     `browser_run_code_unsafe`:
     `async (page) => { await page.emulateMedia({ colorScheme: 'dark' }); await page.goto(SLOT_URL); }`
     (проверка: `matchMedia('(prefers-color-scheme: dark)').matches === true`, `body` bg тёмный).
     Светлую тему проверяй **только** по отдельному запросу / если MR её прямо затрагивает.
   - Открой `SLOT_URL` в инкогнито (MCP запущен с `--isolated`).
   - **Видимость ≠ DOM.** Не делай вывод «пользователь видит X» только по наличию узла в DOM — он может
     быть под скроллом/перекрыт. Проверяй положение во вьюпорте (`getBoundingClientRect` в пределах
     `innerHeight`/`innerWidth`, не `display:none`/нулевой размер) и снимай скриншоты так, чтобы было
     видно состояние как у юзера (длинная поверхность → `fullPage` или проскролль). Что под фолдом —
     **отметь в отчёте**.
   - **Прокликивай сценарий целиком, без пауз на болтовню.** Не комментируй каждый клик — веди флоу
     подряд (клик → проверка → следующий клик) и отвечай пользователю **по завершении проверки** (или
     когда реально заблокирован: нет VPN/баланса/упало). Краткий промежуточный статус — лишь на длинных
     этапах.
   - При необходимости авторизации — следуй `../delivery_qa_smoke/references/auth-scenario.md`
     (временный кошелёк). Если шаг ещё `[нужны данные]` — пройди логин вживую и **зафиксируй реальные
     шаги/селекторы обратно в тот же файл**.
   - Пройди ключевые флоу из п.2; собирай ошибки console/network.
   - **Скриншоты — по папкам задачи** (см. п.6 «Отчёт»), имена `NN-краткий-шаг.png`; для мобильного
     пасса префикс `phone-`.
   - **Мобильный пасс (обязательно) — полная эмуляция девайса iPhone 16 Pro.** Просто `browser_resize`
     **недостаточно** (не появляется мобильный нав-бар: фронт смотрит на `mobile`/touch, а не только
     ширину). Включай device-toolbar-эквивалент через CDP — **совпадающие** размеры в Playwright-вьюпорте
     и в CDP-оверрайде. **Высота — `760`, не 874:** окно headed-браузера высотой 874 + хром Chrome выше
     14"-экрана, и ОС обрезает низ (нав-бар не виден «в эфире»). На 402×760 нав-бар виден целиком, ширина
     iPhone 16 Pro сохранена; для длинных страниц снимай `fullPage`.
     ```js
     async (page) => {
       await page.setViewportSize({ width: 402, height: 760 });
       await page.emulateMedia({ colorScheme: 'dark' });
       const c = await page.context().newCDPSession(page);
       await c.send('Emulation.setDeviceMetricsOverride', { width: 402, height: 760, deviceScaleFactor: 3, mobile: true });
       await c.send('Emulation.setTouchEmulationEnabled', { enabled: true });
       await page.goto(SLOT_URL);
     }
     ```
     (проверка: `innerWidth===402`, `devicePixelRatio===3`, `matchMedia('(pointer: coarse)').matches`).
     Повтори ключевые флоу; в отчёте отрази оба вьюпорта, расхождения desktop↔phone — отдельным пунктом.
     **На mobile другие точки входа:** напр. ⟪ADAPT: пример экрана с иной точкой входа на mobile у вашего продукта⟫ —
     см. `platform-map.md`. Нижний нав-бар фиксирован и виден на главных экранах; на
     полноэкранных страницах (напр. ⟪ADAPT: пример полноэкранного флоу вашего продукта⟫) он скрыт by design — навигируйся `DOM .click()`, если кнопка
     нав-бара под фолдом.
   - **Баланс (`../delivery_qa_smoke/references/test-data.md`):** предзаправки нет, но на стейдже есть
     кнопка пополнения тестовыми деньгами. Если флоу требует баланс (⟪ADAPT: пример флоу, требующего баланс⟫) — пополни и
     доведи сценарий до конца; в отчёте отметь пополнение. Кнопку/ручку пополнения встретил впервые —
     зафиксируй шаги/`data-test` в `test-data.md`. Иначе держись границы «до действия» и помечай
     незавершённое `[не проверено: нет баланса]`.
   - **В конце прогона закрой браузер** (`browser_close`).
5. **UX/UI-советы.** Сверяйся с дизайн-системой фронта — `<frontend>/⟪ADAPT: путь к ui-kit, напр. design-system/ui-kit⟫`
   (`tokens.css`, `src/tokens/*.json`); **иначе** опирайся на эвристики: пустые состояния,
   фокус/клавиатура, контраст, состояния загрузки/ошибок, консистентность.
6. **Отчёт.** Заполни `references/qa-mr-report.md` → сохрани в папку задачи рабочего воркспейса:
   **`tasks/qa-mr-<iid>-<слаг>/report.md`**, скриншоты — рядом в `tasks/qa-mr-<iid>-<слаг>/screenshots/`
   (конвенция delivery-флоу: все артефакты задачи в одной папке — `delivery_orchestrator/references/workspace-layout.md`). Нет `tasks/`-раскладки —
   всё равно создать `tasks/qa-mr-<iid>-<слаг>/` (`mkdir` дёшев; плоских фолбэков нет). Зафиксируй слот/билд, против
   которого гонялось. Заверши блоком **Notion-ready**.

## Правила
- Не выдумывай результаты прогона — шаг не выполнен (нет VPN/баланса) → помечай явно `[не проверено: причина]`.
- Каждый прогон независим (инкогнито). Никогда не коммить `SLOT_URL`/креды.
- **Прокликивай флоу подряд, отвечай в чат по завершении** (а не после каждого клика) — минимум пауз.
- **Тёмная тема по умолчанию** — форсируй `page.emulateMedia({ colorScheme: 'dark' })` перед навигацией.
  Светлая — только по отдельному запросу.
- **Вьюпорты:** desktop `1440×900` (под 14"-экран, не больше) + обязательный мобильный пасс
  **iPhone 16 Pro, `402×760`** через CDP-эмуляцию девайса (`setDeviceMetricsOverride mobile:true` + touch,
  совпадающие размеры с Playwright-вьюпортом).
- **Видимость ≠ DOM:** «юзер видит X» подтверждай положением во вьюпорте, не только наличием в DOM.
- **Закрывай браузер в конце прогона** (`browser_close`).
- Усиливай ценность тест-мышлением: явные ожидания, edge cases, негативные сценарии.
- Язык — русский; названия фич/событий можно по-английски.

## Опции
- Глубокий разбор изменений — субагент `Explore` по фронт-репо.
- Быстрый регресс всей сьюты — manual e2e-CI джобы против слота (`../delivery_qa_smoke/SKILL.md` §3).
- Нужен durable-автотест по мотивам находок — скиллы этого же плагина: `autotest` (написать),
  `selectors_sync` (синк `data-test`), `autotest_run` (прогнать/починить).
- Нужен отдельный тест-план / чеклист / баг-репорт — шаблоны в `references/{test-plan,checklist,bug-report}.md`.
